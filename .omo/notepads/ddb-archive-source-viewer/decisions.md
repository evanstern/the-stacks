### Current backend data flow

- Upload intake is `routes_uploads.create_upload()`: accepts `.html`/`.htm`, stores bytes, creates an `Upload` row and queued `IngestionJob`. It currently accepts only multipart file content; no separate archive URL/capture metadata request fields exist.
- Parser dispatch is `ingestion.parse_document()`: DDB saved HTML is detected by `ddb_import.is_ddb_saved_html()` and parsed by `ingestion._parse_ddb_html()`, while generic HTML uses `_parse_html()`.
- DDB parser contract is already archive-like: `ddb_import.py` extracts `source_type`, `book_title`, `document_title`, `source_url`, `raw_sha256`, `raw_byte_size`, `raw_html_path`, `rendered_html_path`, `jsonl_path`, `content_chunk_id(s)`, `heading_level`, `heading_id`, `section_path`, sanitized `html`, `citation_label`, and `citation_anchor`. `write_ddb_artifacts()` writes `raw.html`, `rendered.html`, `chunks.jsonl`, and `manifest.json`.
- Metadata persistence path is `ingestion.chunk_document()` plus `process_claimed_job()`: document metadata and section metadata are merged into every `DocumentChunk.metadata_json`; source/job/document/section metadata also persist through `Source.metadata_json`, `IngestionJob.metadata_json`, `Document.metadata_json`, and `Section.metadata_json`.
- Qdrant payload creation is `ingestion._qdrant_points()`: current payload is deliberately narrow (`source_id` currently equals `chunk.upload_id`, `chunk_id`, `filename`, `section`, `embedding_model`, `embedding_dimensions`, `chunk_index`, `ingestion_job_id`). Archive fields are not yet copied into Qdrant payload.
- Retrieval/citation flow is `chat_rag._context_from_hit()` -> `_record_hits()`/citation creation -> `routes_sessions._read_citation()`. `_context_from_hit()` reloads `DocumentChunk` by Qdrant `chunk_id`, reads full `chunk.metadata_json`, then overlays Qdrant payload, so archived citation metadata can already reach `RetrievalHit.metadata_json` and `Citation.metadata_json` even before Qdrant payload expansion. `_read_citation()` injects `cited_text` if absent.
- Records API currently exposes chunks with arbitrary `metadata` via `ChunkRead`, but `SourceRead` omits `Source.metadata_json`; archive source list/detail UX would need either source metadata exposure or chunk-level lookup.

### Implementation map for archived webpage source ingestion/citation metadata

- `apps/api/app/ddb_import.py`: keep/extend the DDB artifact and metadata extraction contract; if archived webpage ingestion expands beyond DDB saved HTML, add a parallel archive parser rather than weakening DDB-specific detection.
- `apps/api/app/ingestion.py`: keep `chunk_id` in Qdrant payload for DB metadata reload; add selected archive/citation payload keys only if Qdrant filtering/source preview needs them. Candidate Qdrant payload keys: `source_type`, `source_url`, `raw_sha256`, `raw_html_path`, `rendered_html_path`, `jsonl_path`, `book_title`, `document_title`, `section_path`, `heading_level`, `heading_id`, `content_chunk_id`, `content_chunk_ids`, `citation_label`, `citation_anchor`. Also consider correcting or renaming Qdrant payload `source_id` because it currently uses `chunk.upload_id`, not `chunk.source_id`.
- `apps/api/app/chat_rag.py`: `_citation_metadata()` is the citation metadata choke point; it already returns the whole context metadata plus `cited_text`. Future implementation can leave it as passthrough or explicitly whitelist/normalize archive fields for citation payload stability.
- `apps/api/app/routes_sessions.py`: `_read_citation()` is the route-level fallback point for citation display metadata. If citation rows predate archive metadata, this can merge missing archive keys from `DocumentChunk.metadata_json`.
- `apps/api/app/schemas.py`: `CitationRead.metadata` and `ChunkRead.metadata` already allow metadata-only delivery. `SourceRead` needs a `metadata` field only if the viewer lists archive fields at source level rather than through chunk/citation metadata.
- `apps/api/app/routes_records.py`: `list_chunks()` already returns chunk metadata; `list_sources()` would need to parse/expose `Source.metadata_json` for source-level archive browsing.
- `apps/api/app/routes_uploads.py`: no route change is required for file-based saved HTML ingestion. A new request schema/route is only needed if the product must ingest URL + archived snapshot metadata directly instead of deriving metadata from uploaded HTML.
- Test targets: extend `test_ddb_import.py` for extraction/artifact contract, `test_worker_jobs.py` for metadata persistence, `test_qdrant_indexing.py` for payload propagation, `test_citations.py`/`test_chat_rag.py` for citation metadata response shape, and `test_uploads.py` if upload intake changes.

### Migration decision

- Decision: **metadata-only storage is sufficient for the first archived webpage source viewer/ingestion implementation; no Alembic migration is required unless requirements demand indexed/queryable archive fields or a dedicated archive manifest table.**
- Evidence: current models already include `metadata_json` on `Source`, `Document`, `Section`, `DocumentChunk`, `IngestionJob`, `RetrievalHit`, and `Citation`; migrations `20260531_0003`, `20260531_0005`, and `20260601_0006` create/repair those canonical metadata-bearing tables; `ingestion.chunk_document()` already propagates DDB document/section metadata into chunk rows; chat/citation code reloads DB chunk metadata from Qdrant `chunk_id`.
- Migration would become required only for explicit query/index needs such as source-level `source_url`, `archive_url`, `capture_timestamp`, `provider`, `canonical_url`, or citation fields that must be filtered/sorted without JSON parsing. Candidate targets would be `sources` first, then `citations` or `chunks` only if typed citation/source fields must be independently indexed.
