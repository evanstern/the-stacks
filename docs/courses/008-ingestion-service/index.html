The Pipeline & The Plugin Seam

A D&D Beyond saved page, a Markdown file, and a plain-text note walk into a pipeline. By the time they reach chunking, the pipeline can’t tell them apart — and that’s the entire design.

One plug shape, many sockets

Every country’s wall socket is different — a universal power adapter doesn’t care, because its whole job is making all of them fit one plug shape on the other side. That plug shape, here, is NormalizedDocument. Plugins are the adapters.

Past the transform stage, the pipeline sees exactly ONE shape, forever. Everything downstream — chunking, embedding, indexing — was written once, against that shape, and never needs to change when a new source format shows up.

The contract every plugin signs

The real TypeScript interface lives in packages/ingestion-contract/src/plugin.ts, but its documented shape — the contract plugins are actually held to — is short enough to read in one breath. Every plugin is a pure transform: declare identity and version, detect with a confidence score, transform to ONE shape, nothing else.

CODE — specs/008-ingestion-service/contracts/plugin-contract.md 

interface IngestionPlugin {
  /** Stable unique name, e.g. "ddb-saved-html", "markdown", "generic-html". */
  readonly name: string;
  /** Plugin semver — stamped on every source it produces (FR-016). */
  readonly version: string;
  /** Media types this plugin will even look at (pre-filter for detect). */
  readonly accepts: readonly string[];   // e.g. ["text/html"]
  /** Optional soft guidance to the pipeline-owned chunker (FR-019). */
  readonly chunkingHints?: ChunkingHints;
  /** Cheap, side-effect-free recognition. MUST NOT throw on garbage input. */
  detect(input: DetectInput): DetectResult;
  /** Full extraction + transformation to the normalized document. */
  transform(input: TransformInput): Promise<NormalizedDocument>;
}
PLAIN ENGLISH

The whole contract. Every plugin — shipped or future third-party — is held to exactly these six members and nothing more (FR-013).

The docstring pins the naming register: short, stable, kebab-case — and those three example names are the shipped plugins you’ll meet on the next screen.

Identity. The pipeline stamps this name on every source it ingests, so “which plugin produced this document?” is always answerable.

Why the version matters: it’s recorded per-source at produce time (FR-016)…

…so a plugin semver bump can enumerate exactly which documents are stale and need re-ingestion — no guessing, no full re-crawl.

A cheap pre-filter: if the sniffed media type isn’t in accepts, this plugin’s detect() is never even called.

Declared as data, not code — so the registry can filter candidates without running anything.

Soft guidance only (FR-019): the chunker is pipeline-owned and free to ignore hints…

…which is why the field is optional — a plugin with no chunking opinion simply omits it.

The two MUSTs of detection: cheap (it runs for every registered plugin on every upload) and never-throws (garbage input is normal input here).

Synchronous and side-effect-free. It returns a confidence score, not a decision — Module 2 traces how the pipeline picks the winner.

One call, two observed pipeline stages — this is the extract/transform badge from the hero diagram.

The only place real work happens: raw bytes in, a Promise of NormalizedDocument out — the one plug shape, and nothing else.

That’s the entire surface. No save(), no fetch(), no db handle — “nothing else” is the contract, not a convention.

interface NormalizedDocument {

  title: string;                    // best-effort document title, never empty
  language?: string;                // BCP-47 hint when the plugin knows it
  sections: Section[];              // ordered; MAY be empty (=> honest "empty" outcome)

  warnings: string[];               // non-fatal extraction notes, operator-readable

PLAIN ENGLISH

The one plug shape (FR-017/018). Six fields — everything downstream of transform was written against exactly this, once.

Stamped with the shape’s own version at produce time — so a future breaking change to the contract can enumerate exactly which sources need re-ingestion, the same way a plugin bump can.

A best-effort title, never empty — even a nameless plain-text file must produce something here.

An optional BCP-47 language hint, present only when the plugin actually knows it — the contract bakes in no guessing.

The ordered content itself — allowed to be empty on purpose: an honest “nothing ingestible” outcome, not a silent zero. Each section carries an anchor back into the artifact it came from.

Sanitized HTML fragments for the archive viewer — the viewer renders these, never raw bytes (Principle III’s one safe shape).

Non-fatal extraction notes, written for an operator to read — problems that shouldn’t fail the ingest but shouldn’t vanish either.

Six fields, closed. Adding one is a contract-version event (FR-013), not a quiet edit.

Every plugin gets a vote. The registry — not the plugin — picks the winner, and writes down why.

Detection is not "the first plugin that says yes." Every accepting plugin is asked, every answer is recorded, and the registry — not the plugin — picks the winner. A fallback claiming everything at a confidence floor of 0.1 can never beat a real detector scoring 0.85 or higher.

🎯
Why should I care?

This is the exact mechanism that lets Module 6's demo-format plugin get added later with zero risk of accidentally hijacking traffic meant for ddb-saved-html — the floor is structural, not a promise.

The dispatch loop, line by line

The canonical chain: accepts filter, then detect() per candidate, then highest confidence wins, ties break by registration order, then the winning decision is recorded.

CODE — packages/ingestion/src/registry.ts 

    detect(input) {
      const candidates: Record<string, number> = {};
      let winner: IngestionPlugin | null = null;
      let winning = 0;
      for (const plugin of plugins) {
        if (!plugin.accepts.includes(input.mediaType)) continue;
        let confidence = 0;
        try {
          confidence = plugin.detect(input).confidence;
        } catch {
          confidence = 0; // plugin bug — scored as "not mine", surfaced by conformance
        }
        candidates[plugin.name] = confidence;
        // Strictly-greater keeps the first-registered winner on ties.
        if (confidence > winning) {
          winner = plugin;
          winning = confidence;
        }
      }
      if (!winner || winning <= 0) return null;
      return { plugin: winner, confidence: winning, candidates };
    },

One Function, Every Stage

Once a job is claimed, the worker doesn't hand-roll detect/extract/transform/chunk/embed/index separately. It runs every one of them through a single stage driver.

Picture a relay race where every runner hands off through the exact same timing gate — the gate doesn't care which leg of the race it is, it just starts the clock, waits, and records what happened.

CODE — packages/ingestion/src/ingest-source.ts:97-131 
  /** Runs one stage: started/completed events, timing, and the failure
   * ritual (failed event + scrubbed last_error + status flip) in one place. */
  async function stage<T>(
    name: IngestionStage,
    body: () => Promise<{ result: T; detail?: Record<string, unknown> }>,
    options: { skipStarted?: boolean } = {},
  ): Promise<T> {
    if (!options.skipStarted) await event(name, "started");
    const startedAt = Date.now();
    try {
      const { result, detail } = await body();
      await event(name, "completed", detail, Date.now() - startedAt);
      return result;
    } catch (cause) {
      const error =
        cause instanceof DomainError
          ? cause
          : new DomainError({
              class: "internal_fault",
              seam: name,
              message: `Unexpected failure in ${name} stage.`,
              cause,
            });
      await event(name, "failed", { class: error.class, message: error.message }, Date.now() - startedAt);
      await db
        .update(sources)
        .set({
          status: "failed",
          lastError: { class: error.class, stage: name, message: error.message },
          updatedAt: new Date(),
        })
        .where(sql`${sources.id} = ${sourceId}`);
      throw error;
    }
  }

Generation-Flip

Why re-ingesting a source can never leave a reader looking at a half-old, half-new corpus.

One Column, Two Meanings

A plugin bug fix ships. Every source that plugin ever touched needs to be re-derived. How do you redo work on a live, queryable corpus without a reader ever seeing a half-old, half-new source?

Every section and chunk row carries a generation — one integer that decides which rows are live. A retry replays a job at the SAME generation. A re-ingest starts a brand-new job at generation + 1. One column separates two situations that would otherwise be easy to confuse.

1

Still the rows every reader sees. Untouched, still fully queryable, about to become stale.

2

Already fully constructed — chunked, embedded, indexed — but invisible. No reader has seen one row of it yet.

One UPDATE moves current_generation from 1 to 2 — the instant it commits, generation 1 is swept in the same transaction. Never a moment where a reader could see both.

Think blue-green deployment, shrunk to one integer column: build the new version fully aside, flip one pointer, then clean up the old version. There is never a moment where half the traffic sees old and half sees new.

The Flip Is the Commit

commitGeneration is where generation N+1 stops being build-aside work and becomes the version every reader sees. Read it as two moves, one transaction: flip, then sweep.

CODE — packages/ingestion/src/index-chunks.ts:71-107 

export async function commitGeneration(
  db: Database,
  input: { sourceId: string; generation: number; contractVersion: string },
): Promise<CommitResult> {
  return db.transaction(async (tx) => {
    // The flip IS the commit: one UPDATE makes generation N the one readers
    // see. Everything before this line was invisible build-aside work.
    await tx
      .update(sources)
      .set({
        currentGeneration: input.generation,
        status: "ingested",
        contractVersion: input.contractVersion,
        lastError: null,
        updatedAt: new Date(),
      })
      .where(sql`${sources.id} = ${input.sourceId}`);
    // Sweep is AFTER the flip, same transaction: a reader either saw the old
    // generation before the transaction or the new one after — never neither.
    const sweptSections = await tx
      .delete(documentSections)
      .where(
        and(
          sql`${documentSections.sourceId} = ${input.sourceId}`,
          lt(documentSections.generation, input.generation),
        ),
      )
      .returning({ id: documentSections.id });
    // … sweptChunks: the identical delete against the chunks table, elided …
    return { sweptSections: sweptSections.length, sweptChunks: sweptChunks.length };
  });
}

export async function reingestSource(db: Database, input: ReingestSourceInput): Promise<Job> {
  const [source] = await db.select().from(sources).where(eq(sources.id, input.sourceId));

}

CODE — packages/ingestion/src/sniff.ts 
export function sniffMediaType(filename: string, bytes: Uint8Array): SniffResult | null {
  const extension = extensionOf(filename);
  if (hasMagic(bytes, ZIP_MAGIC)) {
    // A ZIP is a ZIP whatever it is named — but only the .zip extension is an
    // intended batch; a renamed one is a mismatch the caller refuses.
    return extension === "zip" ? { mediaType: "application/zip" } : null;
  }
  const binary = looksBinary(bytes);
  if (binary) return null; // renamed binary or genuinely unsupported (e.g. PDF)
  switch (extension) {
    case "html":
    case "htm":
      return { mediaType: "text/html" };
    case "md":
    case "markdown":
      return { mediaType: "text/markdown" };
    case "txt":
      return { mediaType: "text/plain" };
    case "zip":
      return null; // .zip extension without ZIP magic: a renamed non-zip
    default:
      return null;
  }
}

#3 — a key that shouldn't have been there

The markdown plugin built an artifact object with a spread. When title was undefined, the spread kept the key anyway — present, but undefined-valued. That's not the same thing as the key being absent once it crosses a JSON round-trip purity check.

CODE — packages/ingestion-plugins/src/markdown/index.ts 

  // `title` must be OMITTED, never `undefined` — JSON.stringify drops absent
  // keys but keeps `undefined`-valued ones out of the wire shape differently
  // from how they compare in-memory, which trips the JSON-round-trip purity
  // check (NormalizedDocument invariant 7).
  return title === undefined
    ? { id, kind: "html" as const, content: html }
    : { id, kind: "html" as const, content: html, title };
PLAIN ENGLISH

The comment is the scar tissue, and it opens with the rule: omit the key entirely — never assign undefined to it.

Why: a key holding undefined exists in memory but vanishes when serialized — two "empty" states that aren't the same thing...

...and that gap is exactly what the round-trip purity check compares: the object before JSON.stringify versus after.

The scar ends by naming the law it obeys — NormalizedDocument invariant 7 — so the next author can look it up instead of relearning it.

Branch explicitly on whether title is undefined. No spread, so no way to smuggle in an undefined-valued key.

If it is undefined, build the object with NO title key at all — absent, not empty.

Only include the key when there's a real value to put in it. Presence of the key now tracks presence of a value.

#4 — an anchor that only knew about one line

The demo plugin's detect() matched its own marker with a regex anchored to the start and end of the string, not each line. On real multi-line input, it never found the marker — and returned confidence 0 for every fixture that should have positively matched.

CODE — packages/ingestion-plugins/src/demo/index.ts 

const MARKER = /^@@\s+(.+)$/m;
PLAIN ENGLISH

The pattern was never wrong — the fix is one character, the trailing m flag. Without it, ^ and $ anchor to the whole string's start and end; with it, to each line's — which is what "recognize my format from the first few lines" actually needs.

#1 — the same mistake, twice, weeks apart

Cheerio re-exports the AnyNode type inconsistently across its own subpath exports — enough to compile locally, not enough to survive CI's stricter tsc --noEmit. It hit ddb-saved-html during US1, then hit generic-html independently during US4.

PLAIN ENGLISH

The types cheerio genuinely owns keep coming from cheerio — nothing wrong with this line.

#5 — the container that didn't listen where the compose file assumed

The API service binds to whatever V3_API_PORT says at runtime — unlike the web service, which hardcodes a fixed internal port. But the compose file's port mapping, healthcheck, and the web service's internal URL to reach it all hardcoded the literal default, 4401, assuming the container-side port never moved. It didn't matter in the default worktree. It was fatal in a parallel worktree that overrode the port — found only by actually running docker compose up --wait there.

CODE — docker-compose.yml 

      # BOTH sides must track V3_API_PORT: main.ts binds to
      # process.env.V3_API_PORT (passed through above), so unlike web's fixed
      # internal PORT=4400, the container-side port here is NOT always 4401 —
      # a worktree overriding V3_API_PORT (the spec-cycle port-block
      # convention) must move both the host mapping and the container's own
      # listen port together, or the healthcheck below probes the wrong port.
      - "127.0.0.1:${V3_API_PORT:-4401}:${V3_API_PORT:-4401}"
PLAIN ENGLISH

The scar comment opens with the coupling itself: BOTH sides of this mapping must track V3_API_PORT...

...because the API's own process reads that env var at startup and binds whatever it says...

...which makes the API the one service whose container-side port moves — web's is a fixed PORT=4400, so 4401 here is only a default, never a fact.

And worktrees DO move it: the spec-cycle port-block convention gives every parallel worktree its own port range...

...so the host mapping and the container's own listen port have to change as one...

...or the healthcheck a few lines down probes a port nothing is listening on — exactly how this bug surfaced.

The fix: same variable on both sides of the colon. Host mapping and container listen port now move together, whatever a worktree sets.

Three scenarios built from what you just saw. Not "what's the syntax" — "what's the first thing you'd check."
