04

Generation-Flip

Why re-ingesting a source can never leave a reader looking at a half-old, half-new corpus.

One Column, Two Meanings

A plugin bug fix ships. Every source that plugin ever touched needs to be re-derived. How do you redo work on a live, queryable corpus without a reader ever seeing a half-old, half-new source?

Every section and chunk row carries a generation — one integer that decides which rows are live. A retry replays a job at the SAME generation. A re-ingest starts a brand-new job at generation + 1. One column separates two situations that would otherwise be easy to confuse.

1

Generation 1 — live

Still the rows every reader sees. Untouched, still fully queryable, about to become stale.

→
flip
2

Generation 2 — build aside

Already fully constructed — chunked, embedded, indexed — but invisible. No reader has seen one row of it yet.

One UPDATE moves current_generation from 1 to 2 — the instant it commits, generation 1 is swept in the same transaction. Never a moment where a reader could see both.

Think blue-green deployment, shrunk to one integer column: build the new version fully aside, flip one pointer, then clean up the old version. There is never a moment where half the traffic sees old and half sees new.

↻

Deterministic ids make replay a no-op: the same job re-run targets the same generation and writes nothing new.

A deliberate redo: a brand-new job targets generation + 1, so old and new rows can never collide.

The Flip Is the Commit

commitGeneration is where generation N+1 stops being build-aside work and becomes the version every reader sees. Read it as two moves, one transaction: flip, then sweep.

CODE — packages/ingestion/src/index-chunks.ts:71-107 

export async function commitGeneration(
  db: Database,
  input: { sourceId: string; generation: number; contractVersion: string },
): Promise<CommitResult> {
  return db.transaction(async (tx) => {
    // The flip IS the commit: one UPDATE makes generation N the one readers
    // see. Everything before this line was invisible build-aside work.
    await tx
      .update(sources)
      .set({
        currentGeneration: input.generation,
        status: "ingested",
        contractVersion: input.contractVersion,
        lastError: null,
        updatedAt: new Date(),
      })
      .where(sql`${sources.id} = ${input.sourceId}`);
    // Sweep is AFTER the flip, same transaction: a reader either saw the old
    // generation before the transaction or the new one after — never neither.
    const sweptSections = await tx
      .delete(documentSections)
      .where(
        and(
          sql`${documentSections.sourceId} = ${input.sourceId}`,
          lt(documentSections.generation, input.generation),
        ),
      )
      .returning({ id: documentSections.id });
    // … sweptChunks: the identical delete against the chunks table, elided …
    return { sweptSections: sweptSections.length, sweptChunks: sweptChunks.length };
  });
}

Re-Ingest: Just Another Job

Something has to decide "generation + 1" and put a job on the queue. That something is reingestSource — and it is deliberately boring.

CODE — packages/ingestion/src/reingest.ts:47-69 

export async function reingestSource(db: Database, input: ReingestSourceInput): Promise<Job> {
  const [source] = await db.select().from(sources).where(eq(sources.id, input.sourceId));
  if (!source) {
    throw new DomainError({
      class: "unknown_thing",
      message: `No such source: ${input.sourceId}.`,
    });
  }
  if (source.currentGeneration < 1) {
    // Nothing has ever been committed for this source — there is no prior
    // generation to build past; the FIRST ingestion is the plain intake
    // path's job (ingest_source at generation 1), not re-ingestion's.
    throw new DomainError({
      class: "unsupported_type",
      message: `Source ${input.sourceId} has never completed an ingestion (currentGeneration=0); re-ingest applies only to a source with an existing generation.`,
    });
  }
  return enqueue(db, {
    kind: "ingest_source",
    payload: { sourceId: source.id, targetGeneration: source.currentGeneration + 1 },
  });
}
PLAIN ENGLISH

A domain function that returns the enqueued Job — deliberately NOT an HTTP endpoint. Mutation verbs like this wait for the corpus-lifecycle spec's own dry-run/confirm guardrails (Principle IV); 008 ships the tested primitive, not the exposed trigger.

The entire trigger is one sourceId — look the source up, whole row, currentGeneration included.

First guard: the id must name a real source.

Refuse with a typed DomainError — domain code never speaks HTTP; the status-code mapping lives in exactly one place, apps/api's app.ts.

unknown_thing: the caller named something that doesn't exist.

The message names the missing id, verbatim.

End of the error object.

End of the not-found guard.

Second guard: below 1 means nothing has ever committed for this source.

The comment explains why that is a refusal: there is no prior generation to build past…

…the FIRST ingestion belongs to the plain intake path…

…an ingest_source job at generation 1. Re-ingestion only ever redoes something that already exists.

Same typed-error discipline:

unsupported_type — re-ingest does not apply to this source yet.

And the message spells out exactly why, id and all.

End of the error object.

End of the guard. Past here we hold a real, committed source.

The happy path is ONE enqueue — no pipeline runs inline. Slow work is accept-then-async off the Postgres jobs table (D12).

kind is ingest_source — the exact same job kind every first-time ingestion uses. "Redo" never got its own code path.

Same sourceId, targetGeneration = currentGeneration + 1. That +1 is the entire difference from a retry, which replays the SAME generation and no-ops on deterministic ids. And note what is absent: the archive — it is never touched (FR-023), the stage driver just re-reads the original bytes.

The enqueued Job is the function's whole output.

Decide +1, enqueue, done. Module 3's stage driver does everything else on its next claim — it has no idea this is a redo.
