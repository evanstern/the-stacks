The Pipeline & The Plugin Seam

A D&D Beyond saved page, a Markdown file, and a plain-text note walk into a pipeline. By the time they reach chunking, the pipeline can’t tell them apart — and that’s the entire design.

One plug shape, many sockets

Every country’s wall socket is different — a universal power adapter doesn’t care, because its whole job is making all of them fit one plug shape on the other side. That plug shape, here, is NormalizedDocument. Plugins are the adapters.

Past the transform stage, the pipeline sees exactly ONE shape, forever. Everything downstream — chunking, embedding, indexing — was written once, against that shape, and never needs to change when a new source format shows up.

The contract every plugin signs

The real TypeScript interface lives in packages/ingestion-contract/src/plugin.ts, but its documented shape — the contract plugins are actually held to — is short enough to read in one breath. Every plugin is a pure transform: declare identity and version, detect with a confidence score, transform to ONE shape, nothing else.

CODE — specs/008-ingestion-service/contracts/plugin-contract.md 

interface IngestionPlugin {
  /** Stable unique name, e.g. "ddb-saved-html", "markdown", "generic-html". */
  readonly name: string;
  /** Plugin semver — stamped on every source it produces (FR-016). */
  readonly version: string;
  /** Media types this plugin will even look at (pre-filter for detect). */
  readonly accepts: readonly string[];   // e.g. ["text/html"]
  /** Optional soft guidance to the pipeline-owned chunker (FR-019). */
  readonly chunkingHints?: ChunkingHints;
  /** Cheap, side-effect-free recognition. MUST NOT throw on garbage input. */
  detect(input: DetectInput): DetectResult;
  /** Full extraction + transformation to the normalized document. */
  transform(input: TransformInput): Promise<NormalizedDocument>;
}
PLAIN ENGLISH

The whole contract. Every plugin — shipped or future third-party — is held to exactly these six members and nothing more (FR-013).

The docstring pins the naming register: short, stable, kebab-case — and those three example names are the shipped plugins you’ll meet on the next screen.

Identity. The pipeline stamps this name on every source it ingests, so “which plugin produced this document?” is always answerable.

Why the version matters: it’s recorded per-source at produce time (FR-016)…

…so a plugin semver bump can enumerate exactly which documents are stale and need re-ingestion — no guessing, no full re-crawl.

A cheap pre-filter: if the sniffed media type isn’t in accepts, this plugin’s detect() is never even called.

Declared as data, not code — so the registry can filter candidates without running anything.

Soft guidance only (FR-019): the chunker is pipeline-owned and free to ignore hints…

…which is why the field is optional — a plugin with no chunking opinion simply omits it.

The two MUSTs of detection: cheap (it runs for every registered plugin on every upload) and never-throws (garbage input is normal input here).

Synchronous and side-effect-free. It returns a confidence score, not a decision — Module 2 traces how the pipeline picks the winner.

One call, two observed pipeline stages — this is the extract/transform badge from the hero diagram.

The only place real work happens: raw bytes in, a Promise of NormalizedDocument out — the one plug shape, and nothing else.

That’s the entire surface. No save(), no fetch(), no db handle — “nothing else” is the contract, not a convention.

💡
Nothing else

Plugins declare identity + version, detect with confidence, transform to ONE shape — nothing else. FR-013.

That output shape is the more teachable artifact — the pivotal contract of the whole design, from contracts/normalized-document.md:

CODE — specs/008-ingestion-service/contracts/normalized-document.md 

interface NormalizedDocument {
  contractVersion: string;          // = NORMALIZED_DOCUMENT_VERSION at produce time
  title: string;                    // best-effort document title, never empty
  language?: string;                // BCP-47 hint when the plugin knows it
  sections: Section[];              // ordered; MAY be empty (=> honest "empty" outcome)
  artifacts: DisplayArtifact[];     // sanitized fragments for the archive viewer
  warnings: string[];               // non-fatal extraction notes, operator-readable
}
PLAIN ENGLISH

The one plug shape (FR-017/018). Six fields — everything downstream of transform was written against exactly this, once.

Stamped with the shape’s own version at produce time — so a future breaking change to the contract can enumerate exactly which sources need re-ingestion, the same way a plugin bump can.

A best-effort title, never empty — even a nameless plain-text file must produce something here.

An optional BCP-47 language hint, present only when the plugin actually knows it — the contract bakes in no guessing.

The ordered content itself — allowed to be empty on purpose: an honest “nothing ingestible” outcome, not a silent zero. Each section carries an anchor back into the artifact it came from.

Sanitized HTML fragments for the archive viewer — the viewer renders these, never raw bytes (Principle III’s one safe shape).

Non-fatal extraction notes, written for an operator to read — problems that shouldn’t fail the ingest but shouldn’t vanish either.

Six fields, closed. Adding one is a contract-version event (FR-013), not a quiet edit.

The wall plugins can’t see over

Plugins aren’t just asked nicely to stay in their lane — they’re structurally DB-blind. The doctrine is written right at the top of the package, from packages/ingestion-plugins/src/index.ts:

CODE — packages/ingestion-plugins/src/index.ts 

/**
 * @stacks/ingestion-plugins — the shipped ingesters for spec 008 (FR-028):
 * ddb-saved-html (flagship), markdown, generic-html (fallbacks), plus the
 * test-only demo-format plugin that proves the extensibility promise (SC-007).
 *
 * DOCTRINE (FR-014, the seam that makes "write a new ingester" a small task):
 * everything in this package is a PURE TRANSFORM — bytes in, NormalizedDocument
 * out. Nothing here may import @stacks/db, @stacks/core, @stacks/ingestion, or
 * any HTTP/model client; the only internal dependency is the contract package.
 * scripts/check-boundaries.mjs rule 4 fails the build otherwise. Parsing libs
 * (cheerio, sanitize-html) live here and ONLY here (rule 5).
 *
 * Fixtures under fixtures/ are synthetic look-alikes exercising DDB-shaped
 * structure without any proprietary text (constitution Principle I, FR-024).
 */
PLAIN ENGLISH

A package-level doc comment — the doctrine sits at the top of the entry file, where no contributor opening this package can miss it.

The roster: this package ships the real ingesters delivered by spec 008 (FR-028).

ddb-saved-html is the flagship; markdown and generic-html are the fallback floor…

…plus a test-only demo-format plugin whose entire job is proving that adding a new format is a small, isolated task (SC-007 — Module 6 walks that proof).

A breath — roster above, doctrine below.

The doctrine, named and cited: FR-014 is the seam that makes “write a new ingester” a plugin task instead of a risky pipeline-core change.

The phrase to remember: PURE TRANSFORM — bytes in, NormalizedDocument out, no side effects.

The import blacklist is explicit: no @stacks/db, no @stacks/core, no @stacks/ingestion…

…and no HTTP or model client either — the contract package is the ONLY internal dependency a plugin may reach for.

Not a promise, a linter: scripts/check-boundaries.mjs rule 4 fails the build if any of those imports appear.

The confinement cuts both ways — parsing libraries (cheerio, sanitize-html) may live here and ONLY here (rule 5), so no other package quietly grows a DOM parser.

Second breath — doctrine above, provenance below.

Test fixtures are synthetic look-alikes: they exercise DDB-shaped structure…

…without a single line of proprietary text (constitution Principle I, FR-024) — the repo never ships real game content.

End of the header. Everything below it is just export lines.

🔒
A linter, not a promise

“Structurally unable to import @stacks/db” isn’t a promise — it’s scripts/check-boundaries.mjs failing your build. FR-014 is enforced by a linter, not a code reviewer’s memory.

Three plugins, one shape

Three shipped plugins prove the seam holds under real variety: wildly different internals, identical output contract.

📖

ddbSavedHtmlPlugin

The flagship. Cheerio selectors hunt DDB-specific structural markers — canonical links, saved-from comments, heading stacks — ported from v2 as data-driven rule tables.

📝

markdownPlugin

A fallback floor. No markdown-rendering dependency at all — just an ATX heading (#..######) regex walk to build a heading trail.

🌐

genericHtmlPlugin

The other fallback. A generic DOM heading/section walk with zero DDB signal-hunting — it never even tries to outrank ddb-saved-html.

Cheerio selectors, a regex over heading lines, a generic DOM walk — three completely different implementations. What makes any of it work downstream is that all three end at the same door: every one of them returns a NormalizedDocument. Chunking, embedding, and indexing never know or care which plugin produced it.
