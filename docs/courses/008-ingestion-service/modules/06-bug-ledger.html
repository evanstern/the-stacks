The Bug Ledger

Five real bugs that passed tsc --noEmit. Two passed their own unit tests, wrong. All five were caught the same way — by running the thing.

Every one of these shipped code that looked correct

Every bug on this ledger passed type-checking. Two of them passed their own unit tests on the first try — wrong. What caught all five was the discipline this course keeps insisting on: running the real system, not just reading it.

They cluster into two classes. Type-system blind spots — correct-looking code that a stricter check or a real run exposes as wrong. And environment coupling — code that's only correct under an assumption the codebase itself eventually violated. Neither class shows up in a diff review.

Symptom: compiled fine locally, failed tsc --noEmit in CI — for two different plugins, days apart, same mistake.

Lesson: when a fix is a genuine scar, note why at the import site, or the next author pays the same tax.

#2 — The ??/|| that wouldn't parse

Symptom: the build itself refused to run — mixing ?? and || without parentheses is a syntax error, not a type error.

Lesson: ?? and || are never implicitly composable. The language won't guess which you meant.

Green until you look closer

A test suite that agrees with broken behavior is more dangerous than one that's simply missing. Bugs #3 and #4 are that shape — and #4 is the single scariest bug class in software: the test agreed with the bug.

#3 — a key that shouldn't have been there

The markdown plugin built an artifact object with a spread. When title was undefined, the spread kept the key anyway — present, but undefined-valued. That's not the same thing as the key being absent once it crosses a JSON round-trip purity check.

CODE 

  // `title` must be OMITTED, never `undefined` — JSON.stringify drops absent
  // keys but keeps `undefined`-valued ones out of the wire shape differently
  // from how they compare in-memory, which trips the JSON-round-trip purity
  // check (NormalizedDocument invariant 7).
  return title === undefined
    ? { id, kind: "html" as const, content: html }
    : { id, kind: "html" as const, content: html, title };
PLAIN ENGLISH

The comment is the scar tissue: it names the exact invariant this line protects, so the next person doesn't relearn it the hard way.

Branch explicitly on whether title is undefined...

...and if it is, build the object with NO title key at all...

...only include the key when there's a real value to put in it. No spread, no accidental undefined-valued key.

#4 — an anchor that only knew about one line

The demo plugin's detect() matched its own marker with a regex anchored to the start and end of the string, not each line. On real multi-line input, it never found the marker — and returned confidence 0 for every fixture that should have positively matched.

CODE 

const MARKER = /^@@\s+(.+)$/m;
PLAIN ENGLISH

The marker pattern itself was never wrong — it was missing one character: the trailing m flag.

Without m, ^ and $ anchor to the whole string's start and end. With it, they anchor to each line's — which is what "recognize my format from the first few lines" actually needs.

Worked yesterday, broke somewhere else

#1 and #5 aren't logic bugs at all — they're assumptions that held everywhere the code had been run before, and broke the moment something outside that history changed.

#1 — the same mistake, twice, weeks apart

Cheerio re-exports the AnyNode type inconsistently across its own subpath exports — enough to compile locally, not enough to survive CI's stricter tsc --noEmit. It hit ddb-saved-html during US1, then hit generic-html independently during US4.

CODE 

import type { Cheerio, CheerioAPI } from "cheerio";
import type { AnyNode } from "domhandler";
PLAIN ENGLISH

Keep the types cheerio genuinely owns coming from cheerio — and get AnyNode straight from domhandler, the package that actually defines it.

#5 — the container that didn't listen where the compose file assumed

The API service binds to whatever V3_API_PORT says at runtime — unlike the web service, which hardcodes a fixed internal port. But the compose file's port mapping, healthcheck, and the web service's internal URL to reach it all hardcoded the literal default, 4401, assuming the container-side port never moved. It didn't matter in the default worktree. It was fatal in a parallel worktree that overrode the port — found only by actually running docker compose up --wait there.

CODE 

      # BOTH sides must track V3_API_PORT: main.ts binds to
      # process.env.V3_API_PORT (passed through above), so unlike web's fixed
      # internal PORT=4400, the container-side port here is NOT always 4401 —
      # a worktree overriding V3_API_PORT (the spec-cycle port-block
      # convention) must move both the host mapping and the container's own
      # listen port together, or the healthcheck below probes the wrong port.
      - "127.0.0.1:${V3_API_PORT:-4401}:${V3_API_PORT:-4401}"
PLAIN ENGLISH

Same variable, both sides of the colon: host mapping and container listen port move together, whatever a worktree sets V3_API_PORT to.

File: docker-compose.yml, lines 61–67.

Spot the bug

Three scenarios built from what you just saw. Not "what's the syntax" — "what's the first thing you'd check."

You see this line committed as-is: const x = a ?? b || c; What happens when you try to build it?

A plugin's detect() returns confidence 0 for every fixture that should positively match its own format — but the conformance suite's negative-fixture assertions all pass. What's the most likely bug class, and what's the first thing to check?

A service in a Docker Compose file has one hardcoded port in three places — its own port mapping, its healthcheck, and a sibling service's internal URL to reach it — and also reads a port env var into its own process. What question do you ask before touching any ONE of those three places?

The skeleton walked. Now it can read.
