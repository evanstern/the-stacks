One path, every request. There is no separate "duplicate detector" bolted on the side — a duplicate is just what an optimistic insert looks like when it loses a race it didn't know it was running.

Magic bytes beat declared type, every time

sniffMediaType never asks the client what it thinks the file is. It reads the bytes, then decides — and it never mentions HTTP at all; that translation into a DomainError class (here, unsupported_type → 415) happens one layer up, at the caller.

CODE — packages/ingestion/src/sniff.ts 
export function sniffMediaType(filename: string, bytes: Uint8Array): SniffResult | null {
  const extension = extensionOf(filename);
  if (hasMagic(bytes, ZIP_MAGIC)) {
    // A ZIP is a ZIP whatever it is named — but only the .zip extension is an
    // intended batch; a renamed one is a mismatch the caller refuses.
    return extension === "zip" ? { mediaType: "application/zip" } : null;
  }
  const binary = looksBinary(bytes);
  if (binary) return null; // renamed binary or genuinely unsupported (e.g. PDF)
  switch (extension) {
    case "html":
    case "htm":
      return { mediaType: "text/html" };
    case "md":
    case "markdown":
      return { mediaType: "text/markdown" };
    case "txt":
      return { mediaType: "text/plain" };
    case "zip":
      return null; // .zip extension without ZIP magic: a renamed non-zip
    default:
      return null;
  }
}
PLAIN ENGLISH

Takes the filename AND the raw bytes. The extension is only ever a hint — nothing in this function reads the client's declared Content-Type, ever.

Pull the extension once, lowercased — held for later. The bytes get the first word.

First check is pure bytes: PK\x03\x04 at offset zero means this IS a ZIP, whatever it's named.

The comment carries the doctrine: what the bytes ARE and what the name CLAIMS are separate questions…

…and when they disagree, the answer is an honest refusal (US3), never a best-effort guess.

Only bytes-say-ZIP plus name-says-.zip earns application/zip — the intended batch format. A ZIP renamed .html gets null.

End of the ZIP fast path — anything reaching past this brace has non-ZIP leading bytes.

Second gate: looksBinary scans known magic prefixes — PDF, PNG, JPEG, GIF — plus NUL bytes in the first 512.

Anything binary — a PDF's bytes, a renamed image — is refused HERE, before the extension is even looked at. This is where the report.html-that's-really-a-PNG trap springs.

Only bytes that pass as plausibly-text reach this switch. NOW the extension gets to pick WHICH text format.

.html…

…and .htm share a case…

…both map to text/html.

.md…

…and .markdown…

…are text/markdown.

.txt…

…is text/plain. Three text types plus ZIP is the ENTIRE supported set — PDF is deliberately absent in v3.

A .zip name that already failed the magic check up top lands here…

…and gets its own named refusal — a renamed non-zip is an explicit case, not a fallthrough accident.

Everything else —

null. Unrecognized means refused, full stop.

End of the extension switch — the only place extensions matter, and only after the bytes vouched for them.

null here never mentions HTTP: the CALLER phrases the refusal as an unsupported_type DomainError class, and only the API boundary turns that into a 415.

"Does this exist?" and "create it" are one statement

Every admitted file is identity-checked by its content fingerprint, never its filename. There is no lookup query anywhere in this function that asks "does this already exist?" before writing — the insert itself IS the check.

CODE — packages/ingestion/src/admit.ts 
export async function admitSource(db: Database, input: AdmitSourceInput): Promise<AdmitSourceResult> {
  const fingerprint = deriveArchiveFingerprint(input.bytes);
  return db.transaction(async (tx) => {
    // Content-addressed: same bytes, same row — re-insert is a no-op (R1).
    await tx
      .insert(sourceArchives)
      .values({
        fingerprint,
        bytes: input.bytes,
        byteSize: input.bytes.length,
        mediaType: input.mediaType,
      })
      .onConflictDoNothing();
    const inserted = await tx
      .insert(sources)
      .values({
        corpusId: input.corpusId,
        batchId: input.batchId,
        fingerprint,
        originalFilename: input.filename,
      })
      .onConflictDoNothing({ target: [sources.corpusId, sources.fingerprint] })
      .returning();
    const source = inserted[0];
    if (!source) {
      // Conflict = this corpus already has this CONTENT (filename is never
      // identity). Answer with the existing source's ticket (FR-003).
      const [existing] = await tx
        .select()
        .from(sources)
        .where(and(sql`${sources.corpusId} = ${input.corpusId}`, sql`${sources.fingerprint} = ${fingerprint}`));
      return { source: existing!, duplicate: true };
    }
    // …
    return { source, duplicate: false };
  });
}
PLAIN ENGLISH

The ONE admission path — API intake and the worker's ZIP expansion both call this, so dedupe and archiving can never disagree between the two doors.

Fingerprint the bytes FIRST — sha256 of content, never of whatever the client named the file. A renamed re-upload hashes identical (FR-003).

Everything below runs in ONE transaction: archive row, source row, and the eventual job either all land, or none do — zero residue by construction, not by cleanup.

The comment IS the design: same bytes, same row, so re-inserting is a defined no-op rather than an error to handle.

First write: archive the raw bytes…

…into source_archives, whose PRIMARY KEY is the fingerprint — content-addressed bytea storage, so dedupe is a primary-key lookup, no extra index, no filesystem (FR-003).

Four columns describe the archived content:

the fingerprint — identity derived from the bytes, unspoofable by renaming,

the payload itself, as Postgres bytea — one datastore, no blob store on the side,

the byte count — the in-stream size cap already fired upstream, so this is always under the limit,

and the SNIFFED media type — what the bytes proved, never what the client declared.

That closes the archive row.

If this exact content was ever archived before, onConflictDoNothing makes the re-insert a silent no-op — a collision here is agreement, not an error.

Second write — the load-bearing one. Whatever this insert returns decides new-vs-duplicate.

Insert into sources OPTIMISTICALLY — there is no "does this exist?" query anywhere before this line.

A source is content placed in a corpus:

which corpus — dedupe is scoped per corpus, not global,

the parent batch, when this file arrived inside a ZIP,

the same content fingerprint — one half of the identity pair,

and the filename — kept for DISPLAY only; it participates in no uniqueness check.

End of the row.

THIS is the dedupe check: the unique (corpus_id, fingerprint) index. "Does it exist?" and "create it" are the same statement — the database arbitrates the race.

.returning() hands back the inserted row — or nothing at all, if the constraint fired.

Unpack the verdict: a row means genuinely new content…

…empty means the constraint fired — this corpus already holds these exact bytes, whatever they were called.

The comment names the invariant: filename is never identity…

…and a duplicate is ANSWERED, not rejected — the caller gets the existing ticket (FR-003).

Fetch whichever row actually won the race —

a plain select,

from the same table,

keyed by the exact (corpus_id, fingerprint) pair the constraint enforces — guaranteed to find exactly one row.

Hand back the EXISTING source's ticket, flagged duplicate: true — an honest 200, not a 409: "nothing new happened, here's what already exists."

End of the duplicate branch — the transaction commits having changed nothing.

Elided: the new-content path enqueues the ingest_source job and records the intake event — inside this SAME transaction, so accepting the upload and scheduling its slow work are atomic (accept-then-async, D12).

New content's answer: the fresh source's ticket, duplicate: false — handed back well inside the 2-second claim-a-ticket budget (SC-002), while parsing happens later on the worker.

Commit — only now do archive, source, and job become visible, and they become visible together.

One path, one transaction, one honest answer either way.

What the operator sees — and what stays behind the glass

Every refusal in this system is one of four typed classes. The mapping to HTTP happens exactly once, at the API edge — never inside the code that throws.

unknown_thing 404-shaped — the caller referenced something that doesn't exist (e.g. an unknown corpus)
unsupported_type 415 in this API — recognized the request, won't handle this variant (every refusal in this module)
dependency_down 503-shaped — an external dependency (Postgres, the ML sidecar) is unreachable
internal_fault 500-shaped — our own bug or invariant violation

Here is the REAL scrubbed failure from this cycle's live validation (Scenario 2, ticket 5cffc3a1-29ef-4d0e-9e0f-aacb8493dc4a, uploading truncated.html):

Real output — live validation run 
{
  "class": "unsupported_type",
  "stage": "detect",
  "message": "No registered ingester recognizes \"truncated.html\" (text/html)."
}

That's the entire scrubbed copy: class, stage, and a human-readable message — enough to know exactly why, never enough to leak internals. Everything else — every retry, every attempt's raw detail — stays in the append-only event trail, operator-side only.
