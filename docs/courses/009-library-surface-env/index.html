The Invisible Library

008 shipped a working ingestion pipeline — upload page, ticket page, an eight-stage worker, a pgvector index. Then the operator asked: “is there more to this than the home page?” There was. Nobody could see it.

A library with no card catalog

Picture a library where the books are real, shelved, and indexed — but there is no catalog, no signage, no front desk. The only way to find a book is the receipt you were handed at the door. Lose the receipt, and the book might as well not exist. That was The Stacks after 008: every submission got a claim ticket, and the ticket URL was the ONLY door back in.

The gap, precisely

Nothing was broken. Every capability worked — the pipeline ingested, the ticket page polled, the events recorded. The gap was that working and visible turned out to be different properties, and 008’s spec had only gated on the first one.

The amendment: visibility becomes law

The operator’s 2026-07-09 request didn’t become a bug ticket — it became a constitutional amendment. Version 2.2.0 expands Principle V (Operator Control and Observability) with a mandate: every delivered capability must have a visibility avenue. That’s the right altitude for this fix: a nav link would have patched 008’s pages, but the failure mode — shipping a capability whose spec never asks “how does the operator see this?” — would recur in every future spec. So the rule moved into the document every spec is gated on. Here is the new text, verbatim:

CODE — .specify/memory/constitution.md (Principle V, v2.2.0) 

Every delivered capability MUST have a visibility avenue — being able to see inside is
a property of each feature, not only of the system:

- Operator-facing capabilities MUST be visible in the web UI: Records-style,
  URL-addressable, and reachable through the product's navigation. A page reachable
  only by typing its URL does not satisfy this.
- Where a web surface is not warranted or not yet feasible — developer-facing tooling,
  protocols, background machinery — the capability MUST instead be visible through
  other output: CLI output, log files, or a documented inspection path.
- Each feature's spec or plan MUST record which visibility avenue every delivered
  capability satisfies and why that avenue is the right one.
- A capability with no visibility avenue at all is incomplete, the same way a spec
  cycle without its Principle VIII learning artifact is incomplete.
PLAIN ENGLISH

The headline reframe: every delivered capability must have a visibility avenue…

…because visibility is a property of each FEATURE, not of the system. “The system has an observability surface somewhere” no longer counts — each capability must name its own.

…then three bullets pin down what that means.

If it’s operator-facing, the avenue is the web UI: Records-style pages, addressable by URL…

…and reachable through the product’s navigation — not merely existing at some address.

A page you can only reach by typing its URL is explicitly called out as NOT satisfying this. That sentence is 008’s upload page, described exactly.

Not everything warrants a web page. Developer tooling, protocols, background machinery…

…still owe visibility — they just satisfy it through a different avenue…

…CLI output, log files, or a documented inspection path. The mandate is an avenue, not a UI.

The avenue is recorded in the spec or plan, capability by capability…

…along with WHY that avenue is the right one — a reasoned decision, not a checkbox.

And a capability with no avenue at all is incomplete, by the same standard…

…that makes a cycle without its learning artifact incomplete. Incomplete means the gate fails.

📜
The four-line rule

Web UI when operator-facing. CLI/logs/docs otherwise. The spec records which avenue and why. No avenue = incomplete. That’s the whole amendment — and 009 (FR-018) is the first spec gated on it.

009 itself had to file under this rule. Its converge evidence records four avenues: the library listing and upload/ticket pages (web UI, reachable via nav on every protected page), GET /api/uploads (consumed by the web UI, contract documented), the worktree env protocol (CLI output plus contracts/environment.md), and the Docker lifecycle rules (documentation). Every capability, an avenue, a reason.

02

The Submissions Timeline

A checkout ledger, not two filing cabinets: one merged page, five queries, constant cost — no matter what landed in the library.

Five Queries. Always Five.

Design question: a 200-entry ZIP lands in your library. Is that one row in your timeline, or 200?

One. Rows are submissions — what the operator uploaded — and the database stores those submissions in TWO tables (sources and batches). The listing's job is to answer as the ledger the operator remembers, not as the cabinets the schema uses. And it must do it without the classic listing failure mode: an N+1 that gets slower with every row on the page.

Here is the whole read model in motion. The canonical five, in order: page query (UNION ALL) → total → sections aggregate → chunks aggregate → member-status aggregate. The numbers are the live 009 stack — 57 real submissions, last page.

Why should I care?

This is the repo's canonical collection read model. The retrieval and corpus-lifecycle specs will copy this exact shape — one merged page query, grouped aggregates over the page's ids, an honest total. Learn it once here.

One honest footnote from the evidence: research R3 budgeted three constant queries; the shipped count is five (1 page + 1 total + 3 grouped aggregates). The third aggregate exists because the expand report cannot know post-admission member outcomes — Screen 4's whole subject. The requirement that held is "constant regardless of page size," and it did.

Submissions, Not Members

So: the 200-entry ZIP. Its members live in sources like any other document — but they were never submitted, so the timeline must never show them as rows (research R2). And the merge of the two tables into one newest-first ledger has to page correctly, which means it cannot happen in TypeScript: only the database can sort before it cuts the page.

Both rules are one query. This is UNION ALL doing structural work, verbatim from the route:

CODE — apps/api/src/ingestion/list.ts:74-95 
      // One UNION ALL page query: sorting sources and batches into a single
      // timeline is the database's job, not a TS merge — LIMIT/OFFSET stay
      // correct across both kinds by construction. Kind-specific columns ride
      // as NULLs on the other kind. id DESC tiebreaks same-millisecond rows
      // so paging is deterministic.
      const page = await db.execute(sql`
        SELECT kind, id, original_filename, status, created_at, updated_at,
               plugin_name, plugin_version, detect_confidence, current_generation,
               last_error, entry_report
        FROM (
          SELECT 'source' AS kind, id, original_filename, status, created_at, updated_at,
                 plugin_name, plugin_version, detect_confidence, current_generation,
                 last_error, NULL::jsonb AS entry_report
            FROM sources WHERE batch_id IS NULL
          UNION ALL
          SELECT 'batch' AS kind, id, original_filename, status, created_at, updated_at,
                 NULL, NULL, NULL, NULL, NULL::jsonb, entry_report
            FROM batches
        ) submissions
        ORDER BY created_at DESC, id DESC
        LIMIT ${limit} OFFSET ${offset}
      `);
PLAIN ENGLISH

The comment is the doctrine: merging two tables into one ledger is the database's job, never a TypeScript merge.

…because LIMIT/OFFSET are only correct if the sort runs before the page is cut, and only the database can do both in one pass.

…and the trick that makes one query legal: kind-specific columns ride along…

…as NULLs on the kind that lacks them, while id DESC breaks same-millisecond ties…

…so two identical timestamps page in the same order every time.

One raw SQL statement, executed once — everything the timeline needs comes back in this single round trip.

The outer SELECT names the one uniform row shape both kinds must fit: identity, status, timestamps…

…plus the source-only evidence columns (plugin, confidence, generation)…

…and the two payloads: last_error for failures, entry_report for batches.

Open the subquery that builds the merged ledger.

Arm one: standalone sources, tagged 'source' so the mapper later knows which kind each row is.

…sources genuinely own these columns, so they ride through as real values…

…except entry_report, which is a batch thing — sources carry an explicit NULL passenger there.

WHERE batch_id IS NULL is the R2 exclusion — a ZIP member has a batch_id, so this query cannot see it. Not filtered later by discipline; invisible by shape.

UNION ALL, not UNION: stack the two result sets with zero dedupe cost. Safe because there is nothing to dedupe — see the callout below.

Arm two: batches ride the identical column list, tagged 'batch'.

…with NULLs where source-only evidence goes (plugin, generation, last_error) and a real entry_report where sources carry NULL. NULL passengers keep the UNION legal.

…every batch qualifies — batches are always submissions, so no WHERE clause is needed.

Close the subquery and name the merged set what it is: submissions.

Newest first, with id DESC as tiebreak — two rows born in the same millisecond page in the same order every time. Deterministic paging is what makes "Older" a link you can trust.

Cut the page after the merge. The database pages the ledger, not each cabinet separately.

…and the statement ends: one trip, one merged, sorted, correctly-cut page.

💡
Why UNION ALL is free here

008's intake dedupes by fingerprint: re-upload goblin-page.html and you get the SAME source row back (duplicate=true), never a second one. One record per stored upload means there are no duplicate rows for a plain UNION to remove — so the query skips the dedupe pass entirely. No duplicate timeline rows, ever, and no cost paid to guarantee it.

Counting Only What Readers May See

Each source row shows sections and chunks counts — live in the 009 stack as ddb-saved-html@1.0.0 · gen 1 · sections · passages. But 008's re-ingestion writes a source's next generation aside while the old one stays live. If the listing counted naively, a half-written re-ingest would inflate the numbers mid-flight.

It cannot, because the aggregates apply the reader predicate to observability itself — the counts JOIN through s.current_generation:

CODE — apps/api/src/ingestion/list.ts:104-115 (sections aggregate) 
      // Grouped aggregates for exactly the page's sources, joined through
      // current_generation — the reader predicate applied to observability.
      const [sectionCounts, chunkCounts] = await Promise.all([
        sourceIds.length === 0
          ? { rows: [] as Array<Record<string, unknown>> }
          : db.execute(sql`
              SELECT ds.source_id, count(*) AS n
              FROM document_sections ds
              JOIN sources s ON s.id = ds.source_id
              WHERE ds.source_id IN (${idList(sourceIds)}) AND ds.generation = s.current_generation
              GROUP BY ds.source_id
            `),
        // … the chunks aggregate — the exact twin over the chunks table — elided …
      ]);
PLAIN ENGLISH

The why-comment names the move: 008 R8's reader rule, reused for counting.

…anything a reader may not retrieve, the dashboard may not count.

Both aggregates — sections and chunks — fire in parallel; two queries, one wait.

If the page holds zero source rows, the aggregate does not run at all — the constant five is a ceiling, not a floor.

…the empty case hands back a bare empty result, shaped like the real one, without touching the database.

Otherwise, one raw SQL trip does all the counting.

Ask for one count per source id — n sections apiece.

…counting rows in document_sections…

…joined back to each row's source, solely to reach one number: current_generation.

IN (${idList(sourceIds)}) pins the work to exactly the page's ids — the aggregate never scans sources you are not looking at. And the load-bearing clause: ds.generation = s.current_generation. Rows from a generation still being written aside fail the predicate and simply do not exist to this query.

One grouped count for the whole page — GROUP BY ds.source_id is the anti-N+1: fifty sources, still one query.

…close the statement.

The second Promise.all entry (elided): the chunks aggregate, character-for-character the same shape over the chunks table.

…and both aggregates resolve together — the destructure up top receives them in order.

Aha!

The gen-2 rows are sitting in the same table, same indexes, same source id — and the count is still 2 and 3, not 3 and 5. One JOIN clause is the entire isolation mechanism. When the re-ingest's one-UPDATE flip lands, the same query starts answering with generation 2's rows, atomically.

The Question the Entry Report Cannot Answer

Batch rows summarize their members without being opened. The obvious source for that summary is the entry report — but the report is a photograph taken at expand time. It knows admitted-vs-skipped. It cannot know what an admitted member did afterwards: whether its own ingest pipeline finished or failed lives only on the member source row, written later, by a different job.

That is why query 5 exists — and why the summary is assembled from both truths:

CODE — apps/api/src/ingestion/list.ts:129-131, 183-193 
      // Batch summaries need member STATUSES, not just the expand report: the
      // report says admitted-vs-skipped at expand time; whether an admitted
      // member then ingested or failed lives on the member source row.

          const report = (row.entry_report as EntryReportEntry[] | null) ?? [];
          const members = byBatch.get(base.id) ?? new Map<string, number>();
          return {
            ...base,
            entrySummary: {
              ingested: members.get("ingested") ?? 0,
              skipped: report.filter((entry) => entry.outcome === "skipped").length,
              failed: members.get("failed") ?? 0,
              total: report.length,
            },
          };
PLAIN ENGLISH

The why-comment states the split: the report is frozen at expand completion…

…it knows admitted-vs-skipped and nothing more…

…while member fate keeps evolving afterwards, on the member source rows.

(A gap in the file — we jump from the why-comment down into the per-row assembly.)

report is the photograph — the frozen per-entry outcome list off the batch row (query 1 already fetched it; no extra trip).

members is the live feed — this batch's status counts from query 5, the member-status aggregate.

Build the timeline item the page will render…

…starting from the fields every submission shares…

…and adding the one summary line a batch row carries.

ingested comes from the live feed: only member source rows know what happened after admission.

skipped comes from the photograph: skipped entries never became source rows at all — the report is the only place they exist.

failed is the live feed again — a member that was admitted, then died in its own ingest.

total is the report's length: every entry the ZIP contained, whatever became of it. Two data sources, one honest line per batch.

…close the summary…

…and the item is done: one row, both truths.

The Navigation Shell

The fix for “no link to /library/upload” was NOT adding a link to /library/upload — it was fixing the navigation model, once, in the one place every page inherits from.

Fix the Lobby, Not the Rooms

A library can have perfect shelves, but if the lobby has no signage and no librarian, visitors wander back out. That was the-stacks before 009: the upload page worked, the ticket page worked — and both were reachable only by typing the URL.

The obvious patch is to add a link somewhere. The actual fix is structural: every authenticated route already nests under the protected layout. Put a nav header there, and every protected page — including pages that don't exist yet — renders it. The layout is the lobby. Fix the signage in the lobby and every room inherits it.

The Evidence Column

Reachable is half the job. The other half is that the listing page (/library) tells you something worth reaching: each row's Evidence cell renders a different story per kind of item — a discriminated union, one branch per story.

/** The per-row evidence cell — one component per kind's story (US3). */
function Evidence({ item }: { item: LibraryListItem }) {
  if (item.kind === "batch") {
    const { ingested, skipped, failed, total } = item.entrySummary;
    return (
      <span>
        {total} entries: {ingested} ingested · {skipped} skipped ·{" "}

      </span>

    return (

      </span>

  }
  if (!item.plugin) {

    <span>
      {item.plugin.name}@{item.plugin.version} · gen {item.generation} · {item.counts.sections}{" "}

Whose Fault Is It?

Triage at the Door

Every verdict travels in the same error envelope, and honest blame is the whole design.

The Hazard: A 400 That Fell Into the 500 Pit

Here's the trap. 009's new GET /api/uploads declares a query schema, so Fastify validates limit and offset before the handler ever runs. When validation fails, Fastify throws an error carrying FST_ERR_VALIDATION and statusCode 400 — the framework already knew the right answer. But the API's error handler only recognized two things: a DomainError and Fastify's content-type errors. A validation error is neither, so it fell through to the scrubbed 500 catch-all — which threw away the 400 and confessed to an internal fault instead.

/**
 * Worktree-environment derivation — the pure math behind mint-worktree-env.mjs
 * (009 US2; contracts/environment.md §2–§4). Kept CLI-free so the protocol's

const BLOCK_STRIDE = 10;

/** @param {string} dirname worktree directory name (basename, not a path) */
export function deriveProfile(dirname) {
  if (dirname === "main") {
    // main/ predates the protocol and its compose identity is load-bearing
    // (container/volume names depend on it) — it stays the-stacks-v3 forever.
    return { dirname, projectName: "the-stacks-v3", offset: 0, ports: { ...DEFAULT_PORTS } };
  }
  const match = /^(\d+)-/.exec(dirname);
  if (!match) {
    throw new Error(
      `Cannot derive an environment for "${dirname}": no leading feature number ` +
        `(expected NNN-slug, e.g. 009-library-surface-env). Deterministic derivation ` +
        `is the whole protocol — name the worktree after its feature branch.`,
    );
  }
  const featureNumber = Number.parseInt(match[1], 10);
  const offset = BLOCK_STRIDE * featureNumber;
  return {
    dirname,
    projectName: `the-stacks-${dirname}`,
    offset,
    ports: {
      web: DEFAULT_PORTS.web + offset,
      api: DEFAULT_PORTS.api + offset,
      ml: DEFAULT_PORTS.ml + offset,
      postgres: DEFAULT_PORTS.postgres + offset,
    },
  };
}

Bug #1 — ANY(array) and drizzle's parameter expansion

Symptom: the listing endpoint returned 500 on every request that had at least one row to aggregate. The page query itself ran fine — it was the per-source counts that blew up. Here's the sections aggregate as it first ran. One line is wrong.

🐛 Spot the bug — click the broken line

1SELECT ds.source_id, count(*) AS n
2FROM document_sections ds
3JOIN sources s ON s.id = ds.source_id
4WHERE ds.source_id = ANY(${sourceIds}) -- ← the line as first written
5 AND ds.generation = s.current_generation
6GROUP BY ds.source_id

Mechanism: parameter expansion. When drizzle's sql template meets a JS array, it doesn't bind one array-typed parameter — it splices the array into $1, $2, $3. Inside IN (...) that's exactly what you want. Inside ANY(...) — which takes a single array argument — it's a syntax error at execution time. Same template, same value, opposite outcomes, decided entirely by the SQL construct around it.

Fix: stop fighting the expansion and lean into it — build explicit IN lists. The helper, verbatim from the source, why-comment and all:

CODE 

      // drizzle's sql template expands a JS array into comma-joined params —
      // fine for IN (...), wrong inside ANY(...). Build IN lists explicitly.
      const idList = (ids: string[]) => sql.join(ids.map((id) => sql`${id}`), sql`, `);
PLAIN ENGLISH

The why-comment is the scar: it names the hazard at the exact place the next author will meet it.

Each id becomes its own one-parameter fragment (sql`${id}`)...

...and sql.join stitches them with literal commas — a hand-built $1, $2, $3 that goes inside IN (...), where comma-joined placeholders are exactly the right shape.

The call sites read WHERE ds.source_id IN (${idList(sourceIds)}) — expansion semantics and SQL construct finally agreeing. File: apps/api/src/ingestion/list.ts, lines 100–102.

Bug #2 — the 400 that was a 500

Symptom: GET /api/uploads?limit=nope answered 500 internal_fault — "An internal error occurred." Nothing internal had erred. The caller sent a string where the schema demanded an integer; the server took the blame anyway.

Here's the error handler as 009 inherited it from 008 — the single seam where domain vocabulary becomes HTTP vocabulary, checks running most-specific first. Fastify's schema validator throws FST_ERR_VALIDATION for ?limit=nope. Click the branch that catches it.

🐛 Spot the bug — where does FST_ERR_VALIDATION land?

1if (error instanceof DomainError) { /* class → status, typed envelope */ }
2if (error.code.startsWith("FST_ERR_CTP")) { /* 415 unsupported_type */ }
3if (error.statusCode === 415) { /* 415 unsupported_type */ }
4request.log.error(...); reply.code(500).send("An internal error occurred.");

Mechanism: the catch-all is supposed to scrub — it exists so genuine server faults never leak stack traces to the wire. But scrubbing a caller's shape error hides the one thing the caller needs: that the fix is on their side. Module 4 called this the honest-status rule; this is the bug that motivated it.

Fix: a branch above the catch-all, most-specific-first order preserved — verbatim from apps/api/src/app.ts, lines 50–60 (Module 4 walked this snippet in full; here it is as the reveal):

CODE 

    // Fastify schema-validation failures (malformed query/body per a route's
    // declared schema) are the caller's shape error — an honest 400, never the
    // scrubbed 500 the catch-all would turn them into (009, contracts/api.md).
    // error.message here is ajv's generated text (e.g. "querystring/limit must
    // be integer") — mechanical, parameter-scoped, safe for the wire.
    if (typeof error.code === "string" && error.code === "FST_ERR_VALIDATION") {
      reply
        .code(statusForErrorClass("invalid_input"))
        .send(errorEnvelope("invalid_input", error.message));
      return;
    }
PLAIN ENGLISH

The comment argues the two decisions that matter: why this is a 400 (caller's shape error), and why relaying ajv's message is safe (mechanical, parameter-scoped — no internals to leak).

…a request that fails a route's declared schema is the caller's mistake, so it earns an honest 400…

…never the scrubbed 500 the catch-all below would otherwise turn it into.

The second decision: ajv generates these messages by machine, straight from the schema…

…so they name the offending parameter and nothing else — safe to send over the wire.

Match Fastify's validation errors by their stable code...

Then build the reply in two chained steps…

…set the status by asking the shared error-class table what invalid_input maps to (400)…

...and answer with the typed envelope: invalid_input, status 400, ajv's own diagnosis as the message.

Return early — this error is fully handled, so the catch-all never sees it…

…and close the branch.

Bug #3 — the $$ that wasn't

Symptom: during the live worktree pivot, a freshly minted .env carried OPERATOR_PASSWORD_HASH=b. A 60-character bcrypt hash, reduced to one letter. Two mechanisms stacked to produce it:

Mechanism (a): a bcrypt hash looks like $2b$10$... — and sourcing a file containing HASH=$2b$10$... hands it to the shell, which happily expands $2 as a positional parameter. Empty. What survives is the literal debris between the expansions: b. The secret wasn't corrupted in storage; it was eaten in transit, by a shell doing exactly what shells do.

Mechanism (b): the first repair attempt, in JS. Compose interpolation needs every $ in the stored value escaped as $$, so the obvious move was a one-liner. Click the broken line.

🐛 Spot the bug — the repair that repaired nothing

1const hash = readOperatorHash(); // "$2b$10$..."
2const escaped = hash.replaceAll('$', '$$'); // escape for compose
3lines.push(`OPERATOR_PASSWORD_HASH=${escaped}`);

Fix: two layers. Tactically, use a form that never interprets its replacement — the function form replaceAll('$', () => '$$'), or split('$').join('$$'), where join treats its argument as bytes, full stop. Structurally — and this is the fix that actually shipped — the mint tool copies secrets verbatim from an existing .env, already-escaped bytes in, identical bytes out, so no human (or shell) ever re-derives the escaping again. That doctrine is written into parseEnv itself, verbatim from scripts/worktree-env-lib.mjs, lines 57–59:

CODE 

/** Minimal .env parser — KEY=value lines, comments/blanks skipped. Values are
 * kept verbatim (no unquoting/expansion: compose does its own interpolation,
 * and round-tripping bytes untouched is what keeps escaped bcrypt hashes safe). */
PLAIN ENGLISH

A parser that does almost nothing, on purpose: split KEY=value, skip comments and blank lines, and stop there.

Every clause of this comment is a scar from the bug above: no expansion (mechanism a), no re-escaping (mechanism b), bytes in = bytes out.

The parser refuses to be clever precisely because two layers of cleverness — the shell's and JS's — each destroyed the value in a different way.

📌
Pinned by

The protocol suite's mint test (scripts/worktree-env-lib.test.mjs): it feeds OPERATOR_PASSWORD_HASH: "$$2b$$10$$x" through mintEnv → parseEnv and asserts the identical string comes back — "Secrets are copied verbatim, never invented." Any future "helpful" escaping or expansion turns that assertion red.

What the ledger teaches

Three bugs, one shape: semantics that depend on value shape. An array means one thing to IN and another to ANY. A $ means one thing in a pattern and another in a replacement string — and a third thing to a shell. An error object means "caller's fault" or "server's fault" depending on which branch reads it first. None of these are exotic; all of them look correct in a diff. The defense was never cleverness — it was a test, written first, that pinned what the code was supposed to mean.

💡
Aha!

Notice what the three fixes have in common: none of them made the code smarter. The idList helper hand-builds the dumbest possible SQL fragment. The invalid_input branch just relays ajv's own words. parseEnv refuses to interpret anything at all. When semantics depend on value shape, the winning move is to remove a layer of interpretation, not add one — and then pin the boring behavior with a test so nobody helpfully re-adds the cleverness later.
