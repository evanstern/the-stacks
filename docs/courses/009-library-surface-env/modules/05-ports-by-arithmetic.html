Ports by Arithmetic

Street addresses, not parking spots — a worktree's whole environment derives from its directory name, and the mint tool only verifies the math.

Where was 4500 written down? Nowhere.

The operator's own instance historically ran web on port 4500. Where is that recorded? Before 009: nowhere. It was folklore — the kind that works until the day a second stack needs a port and nobody remembers who owns what.

A parking lot solves this with an attendant, a registry, and arguments about who took spot 14. Street addresses solve it with arithmetic: house NNN is at base + 10×NNN, forever, no attendant. 009 chose street addresses. Every worktree derives its entire environment — its port block and its compose project — from its DIRECTORY NAME:

The 4500 cautionary tale

Feature 010's derived web port is 4400 + 10×10 = 4500 — exactly where the folklore instance lived. That near-miss is why the contract sends manual overrides ≥ 10000: above every block the arithmetic will ever produce.

Uniqueness is inherited, not managed: spec-kit assigns each feature a unique number, so derived blocks can't collide. No registry exists because none is needed.

The tool doesn't allocate. It verifies.

deriveProfile is pure math, kept CLI-free so the protocol's guarantees are unit-tested inside pnpm verify. Read the header comment first — it's the whole doctrine in one place:

CODE — scripts/worktree-env-lib.mjs:1-55 (PORT_KEYS table elided) 

/**
 * Worktree-environment derivation — the pure math behind mint-worktree-env.mjs
 * (009 US2; contracts/environment.md §2–§4). Kept CLI-free so the protocol's
 * guarantees are unit-testable (worktree-env-lib.test.mjs, wired into
 * `pnpm verify`): determinism here is what lets the repo skip a port registry.
 *
 * The rule in one line: a worktree's identity and ports derive from its
 * DIRNAME alone — `NNN-slug` gets default + 10×NNN and the-stacks-NNN-slug;
 * `main` is the fixed point (offset 0, the-stacks-v3). Feature numbers are
 * unique by spec-kit construction, so derived blocks can't collide;
 * findPortCollisions() merely VERIFIES that inheritance against reality
 * (manual overrides are where mistakes live — see §3's web-on-4500 tale).
 */

/** Default published ports — mirrors .env.example, the variable contract. */
export const DEFAULT_PORTS = Object.freeze({ web: 4400, api: 4401, ml: 4402, postgres: 5442 });

const BLOCK_STRIDE = 10;

/** @param {string} dirname worktree directory name (basename, not a path) */
export function deriveProfile(dirname) {
  if (dirname === "main") {
    // main/ predates the protocol and its compose identity is load-bearing
    // (container/volume names depend on it) — it stays the-stacks-v3 forever.
    return { dirname, projectName: "the-stacks-v3", offset: 0, ports: { ...DEFAULT_PORTS } };
  }
  const match = /^(\d+)-/.exec(dirname);
  if (!match) {
    throw new Error(
      `Cannot derive an environment for "${dirname}": no leading feature number ` +
        `(expected NNN-slug, e.g. 009-library-surface-env). Deterministic derivation ` +
        `is the whole protocol — name the worktree after its feature branch.`,
    );
  }
  const featureNumber = Number.parseInt(match[1], 10);
  const offset = BLOCK_STRIDE * featureNumber;
  return {
    dirname,
    projectName: `the-stacks-${dirname}`,
    offset,
    ports: {
      web: DEFAULT_PORTS.web + offset,
      api: DEFAULT_PORTS.api + offset,
      ml: DEFAULT_PORTS.ml + offset,
      postgres: DEFAULT_PORTS.postgres + offset,
    },
  };
}

The values that must move together

Moving four ports is arithmetic. The trap is the port-coupled value: the api container binds V3_API_PORT INSIDE the container (its process reads the env var directly), so the in-network URL API_INTERNAL_URL must move with it. Meanwhile EMBEDDING_ENDPOINT and DATABASE_URL never move — ml and postgres keep fixed container-side ports; only their published host ports shift.

This warning used to live as a compose-file comment — advice you could scroll past. Now it's derivation plus a --check finding:

scripts/worktree-env-lib.mjs:80-95 
export function mintEnv(template, dirname, secrets = {}) {
  const profile = deriveProfile(dirname);
  const replacements = {
    COMPOSE_PROJECT_NAME: profile.projectName,
    V3_WEB_PORT: String(profile.ports.web),
    V3_API_PORT: String(profile.ports.api),
    V3_ML_PORT: String(profile.ports.ml),
    V3_POSTGRES_PORT: String(profile.ports.postgres),
    // The port-coupled derivation (§4): the api container binds V3_API_PORT
    // INSIDE the container, so the in-network URL must move with it.
    API_INTERNAL_URL: `http://api:${profile.ports.api}`,
    ...(secrets.OPERATOR_PASSWORD_HASH !== undefined
      ? { OPERATOR_PASSWORD_HASH: secrets.OPERATOR_PASSWORD_HASH }
      : {}),
    ...(secrets.SESSION_SECRET !== undefined ? { SESSION_SECRET: secrets.SESSION_SECRET } : {}),
  };

Note the last two entries: secrets are copied or left blank, never generated — inventing an operator password silently would hide the one thing the operator must own.

--check is the drift mode. Its three finding kinds, all named, none auto-repaired:

missing template keys absent from the .env — .env.example grew and your file didn't
unknown .env keys the template no longer knows — retired or typo'd variables
couplings port-coupled values that stopped moving together — the classic ECONNREFUSED-from-every-loader footgun, now named

Lifecycle rules — memorize the middle one

down stops containers but KEEPS volumes — your database survives a restart. down --volumes is teardown: data gone, deliberately. And teardown always precedes git worktree remove — deleting the directory first orphans the compose project's containers and volumes with nobody left to name them.
