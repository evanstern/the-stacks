The Submissions Timeline

A checkout ledger, not two filing cabinets: one merged page, five queries, constant cost — no matter what landed in the library.

Five Queries. Always Five.

Design question: a 200-entry ZIP lands in your library. Is that one row in your timeline, or 200?

One. Rows are submissions — what the operator uploaded — and the database stores those submissions in TWO tables (sources and batches). The listing's job is to answer as the ledger the operator remembers, not as the cabinets the schema uses. And it must do it without the classic listing failure mode: an N+1 that gets slower with every row on the page.

Here is the whole read model in motion. The canonical five, in order: page query (UNION ALL) → total → sections aggregate → chunks aggregate → member-status aggregate. The numbers are the live 009 stack — 57 real submissions, last page.

Why should I care?

This is the repo's canonical collection read model. The retrieval and corpus-lifecycle specs will copy this exact shape — one merged page query, grouped aggregates over the page's ids, an honest total. Learn it once here.

One honest footnote from the evidence: research R3 budgeted three constant queries; the shipped count is five (1 page + 1 total + 3 grouped aggregates). The third aggregate exists because the expand report cannot know post-admission member outcomes — Screen 4's whole subject. The requirement that held is "constant regardless of page size," and it did.

Submissions, Not Members

So: the 200-entry ZIP. Its members live in sources like any other document — but they were never submitted, so the timeline must never show them as rows (research R2). And the merge of the two tables into one newest-first ledger has to page correctly, which means it cannot happen in TypeScript: only the database can sort before it cuts the page.

Both rules are one query. This is UNION ALL doing structural work, verbatim from the route:

CODE — apps/api/src/ingestion/list.ts:74-95 
      // One UNION ALL page query: sorting sources and batches into a single
      // timeline is the database's job, not a TS merge — LIMIT/OFFSET stay
      // correct across both kinds by construction. Kind-specific columns ride
      // as NULLs on the other kind. id DESC tiebreaks same-millisecond rows
      // so paging is deterministic.
      const page = await db.execute(sql`
        SELECT kind, id, original_filename, status, created_at, updated_at,
               plugin_name, plugin_version, detect_confidence, current_generation,
               last_error, entry_report
        FROM (
          SELECT 'source' AS kind, id, original_filename, status, created_at, updated_at,
                 plugin_name, plugin_version, detect_confidence, current_generation,
                 last_error, NULL::jsonb AS entry_report
            FROM sources WHERE batch_id IS NULL
          UNION ALL
          SELECT 'batch' AS kind, id, original_filename, status, created_at, updated_at,
                 NULL, NULL, NULL, NULL, NULL::jsonb, entry_report
            FROM batches
        ) submissions
        ORDER BY created_at DESC, id DESC
        LIMIT ${limit} OFFSET ${offset}
      `);
PLAIN ENGLISH

The comment is the doctrine: merging two tables into one ledger is the database's job, never a TypeScript merge.

…because LIMIT/OFFSET are only correct if the sort runs before the page is cut, and only the database can do both in one pass.

…and the trick that makes one query legal: kind-specific columns ride along…

…as NULLs on the kind that lacks them, while id DESC breaks same-millisecond ties…

…so two identical timestamps page in the same order every time.

One raw SQL statement, executed once — everything the timeline needs comes back in this single round trip.

The outer SELECT names the one uniform row shape both kinds must fit: identity, status, timestamps…

…plus the source-only evidence columns (plugin, confidence, generation)…

…and the two payloads: last_error for failures, entry_report for batches.

Open the subquery that builds the merged ledger.

Arm one: standalone sources, tagged 'source' so the mapper later knows which kind each row is.

…sources genuinely own these columns, so they ride through as real values…

…except entry_report, which is a batch thing — sources carry an explicit NULL passenger there.

WHERE batch_id IS NULL is the R2 exclusion — a ZIP member has a batch_id, so this query cannot see it. Not filtered later by discipline; invisible by shape.

UNION ALL, not UNION: stack the two result sets with zero dedupe cost. Safe because there is nothing to dedupe — see the callout below.

Arm two: batches ride the identical column list, tagged 'batch'.

…with NULLs where source-only evidence goes (plugin, generation, last_error) and a real entry_report where sources carry NULL. NULL passengers keep the UNION legal.

…every batch qualifies — batches are always submissions, so no WHERE clause is needed.

Close the subquery and name the merged set what it is: submissions.

Newest first, with id DESC as tiebreak — two rows born in the same millisecond page in the same order every time. Deterministic paging is what makes "Older" a link you can trust.

Cut the page after the merge. The database pages the ledger, not each cabinet separately.

…and the statement ends: one trip, one merged, sorted, correctly-cut page.

💡
Why UNION ALL is free here

008's intake dedupes by fingerprint: re-upload goblin-page.html and you get the SAME source row back (duplicate=true), never a second one. One record per stored upload means there are no duplicate rows for a plain UNION to remove — so the query skips the dedupe pass entirely. No duplicate timeline rows, ever, and no cost paid to guarantee it.

Counting Only What Readers May See

Each source row shows sections and chunks counts — live in the 009 stack as ddb-saved-html@1.0.0 · gen 1 · sections · passages. But 008's re-ingestion writes a source's next generation aside while the old one stays live. If the listing counted naively, a half-written re-ingest would inflate the numbers mid-flight.

It cannot, because the aggregates apply the reader predicate to observability itself — the counts JOIN through s.current_generation:

CODE — apps/api/src/ingestion/list.ts:104-115 (sections aggregate) 
      // Grouped aggregates for exactly the page's sources, joined through
      // current_generation — the reader predicate applied to observability.
      const [sectionCounts, chunkCounts] = await Promise.all([
        sourceIds.length === 0
          ? { rows: [] as Array<Record<string, unknown>> }
          : db.execute(sql`
              SELECT ds.source_id, count(*) AS n
              FROM document_sections ds
              JOIN sources s ON s.id = ds.source_id
              WHERE ds.source_id IN (${idList(sourceIds)}) AND ds.generation = s.current_generation
              GROUP BY ds.source_id
            `),
PLAIN ENGLISH

The why-comment names the move: 008 R8's reader rule, reused for counting.

…anything a reader may not retrieve, the dashboard may not count.

Both aggregates — sections and chunks — fire in parallel; two queries, one wait.

If the page holds zero source rows, the aggregate does not run at all — the constant five is a ceiling, not a floor.

…the empty case hands back a bare empty result, shaped like the real one, without touching the database.

Otherwise, one raw SQL trip does all the counting.

Ask for one count per source id — n sections apiece.

…counting rows in document_sections…

…joined back to each row's source, solely to reach one number: current_generation.

IN (${idList(sourceIds)}) pins the work to exactly the page's ids — the aggregate never scans sources you are not looking at. And the load-bearing clause: ds.generation = s.current_generation. Rows from a generation still being written aside fail the predicate and simply do not exist to this query.

One grouped count for the whole page — GROUP BY ds.source_id is the anti-N+1: fifty sources, still one query.

…close the statement; the chunks aggregate (not shown) is its exact twin over document_chunks.

Aha!

The gen-2 rows are sitting in the same table, same indexes, same source id — and the count is still 2 and 3, not 3 and 5. One JOIN clause is the entire isolation mechanism. When the re-ingest's one-UPDATE flip lands, the same query starts answering with generation 2's rows, atomically.

The Question the Entry Report Cannot Answer

Batch rows summarize their members without being opened. The obvious source for that summary is the entry report — but the report is a photograph taken at expand time. It knows admitted-vs-skipped. It cannot know what an admitted member did afterwards: whether its own ingest pipeline finished or failed lives only on the member source row, written later, by a different job.

That is why query 5 exists — and why the summary is assembled from both truths:

CODE — apps/api/src/ingestion/list.ts:129-131, 183-193 
      // Batch summaries need member STATUSES, not just the expand report: the
      // report says admitted-vs-skipped at expand time; whether an admitted
      // member then ingested or failed lives on the member source row.

          const report = (row.entry_report as EntryReportEntry[] | null) ?? [];
          const members = byBatch.get(base.id) ?? new Map<string, number>();
          return {
            ...base,
            entrySummary: {
              ingested: members.get("ingested") ?? 0,
              skipped: report.filter((entry) => entry.outcome === "skipped").length,
              failed: members.get("failed") ?? 0,
              total: report.length,
            },
          };
PLAIN ENGLISH

The why-comment states the split: the report is frozen at expand completion…

…it knows admitted-vs-skipped and nothing more…

…while member fate keeps evolving afterwards, on the member source rows.

(A gap in the file — we jump from the why-comment down into the per-row assembly.)

report is the photograph — the frozen per-entry outcome list off the batch row (query 1 already fetched it; no extra trip).

members is the live feed — this batch's status counts from query 5, the member-status aggregate.

Build the timeline item the page will render…

…starting from the fields every submission shares…

…and adding the one summary line a batch row carries.

ingested comes from the live feed: only member source rows know what happened after admission.

skipped comes from the photograph: skipped entries never became source rows at all — the report is the only place they exist.

failed is the live feed again — a member that was admitted, then died in its own ingest.

total is the report's length: every entry the ZIP contained, whatever became of it. Two data sources, one honest line per batch.

…close the summary…

…and the item is done: one row, both truths.
