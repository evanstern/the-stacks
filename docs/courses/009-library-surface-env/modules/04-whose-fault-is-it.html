Whose Fault Is It?

The honest 400 — how malformed input joined a closed error taxonomy without breaking its closure.

Triage at the Door

GET /api/uploads?limit=nope — before 009, that answered 500 "An internal error occurred." Whose fault does that claim it is? Ours. The server just confessed to a crime the caller committed.

The API's error design is triage at the door: every failed request gets blamed on exactly one party, and the status code IS the blame. You asked for a thing that doesn't exist (404), a thing we don't handle (415), you're not signed in (401), our dependency is down (503), we broke (500) — and, new in 009: your request's SHAPE is wrong (400). Every verdict travels in the same error envelope, and honest blame is the whole design.

The scrubbed catch-all. It blames the server, tells the caller nothing actionable, and pages an operator about a typo.

HTTP/1.1 400 Bad Request

{"error":{"code":"invalid_input","message":"querystring/limit must be integer"}}

Same envelope shape as every other error, correct blame, and a message that names the exact parameter and what it must be.

The Hazard: A 400 That Fell Into the 500 Pit

Here's the trap. 009's new GET /api/uploads declares a query schema, so Fastify validates limit and offset before the handler ever runs. When validation fails, Fastify throws an error carrying FST_ERR_VALIDATION and statusCode 400 — the framework already knew the right answer. But the API's error handler only recognized two things: a DomainError and Fastify's content-type errors. A validation error is neither, so it fell through to the scrubbed 500 catch-all — which threw away the 400 and confessed to an internal fault instead.

The fix is one branch in the one seam where errors become HTTP:

CODE — apps/api/src/app.ts:44-60 

  app.setErrorHandler((error: FastifyError, request, reply) => {
    if (error instanceof DomainError) {
      reply.code(statusForErrorClass(error.class)).send(errorEnvelope(error.class, error.message));
      return;
    }

    // Fastify schema-validation failures (malformed query/body per a route's
    // declared schema) are the caller's shape error — an honest 400, never the
    // scrubbed 500 the catch-all would turn them into (009, contracts/api.md).
    // error.message here is ajv's generated text (e.g. "querystring/limit must
    // be integer") — mechanical, parameter-scoped, safe for the wire.
    if (typeof error.code === "string" && error.code === "FST_ERR_VALIDATION") {
      reply
        .code(statusForErrorClass("invalid_input"))
        .send(errorEnvelope("invalid_input", error.message));
      return;
    }
    // … content-type branches and the scrubbed internal_fault catch-all elided …
  });

Aha!

The bug wasn't a missing feature — it was a fall-through. Fastify already produced a 400; the seam's catch-all overwrote it with a lie. The fix adds a branch to the mapping, not validation logic anywhere else. The next route that declares a schema gets the 400 path for free.

Boundary Codes vs Domain Classes

So why not add invalid_input to @stacks/core's ErrorClass union alongside unknown_thing and friends? Because the taxonomy stays closed by distinguishing two kinds of vocabulary: DOMAIN classes are shared and transport-agnostic — a worker handler can throw dependency_down as meaningfully as a route can. BOUNDARY codes only mean something at the HTTP edge. Only HTTP can have a malformed request; worker and db code can never throw one. The file's header comment argues it outright:

CODE — apps/api/src/errors.ts:13-32 

 * "invalid_input" (400) joined "unauthorized" as an API-only code in 009: a
 * request whose SHAPE is wrong (malformed query params caught by Fastify
 * schema validation) is neither an unknown thing nor an unsupported payload
 * type — and letting it fall through to the scrubbed 500 would lie about
 * whose fault it is. Like unauthorized, it stays out of the shared ErrorClass
 * union because only the HTTP edge can have malformed requests.
 */

/** ErrorClass plus the codes that only exist at the HTTP boundary. */
export type ApiErrorCode = ErrorClass | "unauthorized" | "invalid_input";

const STATUS_BY_CLASS: Record<ApiErrorCode, number> = {
  unknown_thing: 404,
  unsupported_type: 415,
  dependency_down: 503,
  internal_fault: 500,
  unauthorized: 401,
  invalid_input: 400,
};

The 009 paragraph of the file's header: it names the new code and its status up front, plus the precedent it follows.

A malformed SHAPE is the caller's doing — Fastify's schema validation caught it…

…and it fits none of the existing classes: not an unknown thing, not an unsupported payload type…

…so letting it fall through to the scrubbed 500 "would lie about whose fault it is."

The precedent is unauthorized — it took this exact path in 007, because the worker has no auth concept either.

The closing argument in one clause: only the HTTP edge can have malformed requests. So the code lives with the mapping, not in the shared union.

…end of the header argument.

Argument above, implementation below.

The doc-comment says in one line what the type is about to say structurally.

ApiErrorCode: the four shared classes, plus exactly two edge-only extensions.

Now the payoff of keeping the taxonomy small.

The whole translation table fits in one Record — and this file is the ONLY place in the system that knows HTTP numbers.

The four domain classes first: something asked for doesn't exist…

…the payload type isn't one we handle…

…something we depend on is unreachable…

…and our own fault, deliberately scrubbed to a generic 500.

Then the two boundary codes: no valid session…

…end of the mapping table. Six codes, six statuses, one file.

On the wire, live

From this cycle's quickstart A6 run against the 009 stack: {"error":{"code":"invalid_input","message":"querystring/limit must be integer"}} — the taxonomy's newest stamp, exactly as a real caller receives it.

One decision remains: not every weird limit deserves a 400. The route draws the line with its schema and one line of clamping:

CODE — apps/api/src/ingestion/list.ts:37-47, 71 

// ajv coerces query strings to integers and refuses what won't coerce
// ("nope") or violates minimum (-1) — the 400 path. Clamping beyond that is
// handler logic: 0 and 9999 are honest requests with honest nearest answers.
const LIST_QUERY_SCHEMA = {
  type: "object",
  properties: {
    limit: { type: "integer", minimum: 0 },
    offset: { type: "integer", minimum: 0 },
  },
  additionalProperties: true,
} as const;

// ...inside the GET /api/uploads handler:
const limit = Math.min(MAX_LIMIT, Math.max(1, request.query.limit ?? DEFAULT_LIMIT));
PLAIN ENGLISH

The comment IS the policy. ajv handles the refusals: "nope" won't coerce to an integer…

…-1 violates minimum: 0 — both hit the 400 path from Screen 2. Clamping is a different job…

…handler logic, because 0 and 9999 are "honest requests with honest nearest answers" — coercible, in-range integers that are merely extreme.

The schema itself is deliberately loose. It only rejects what cannot be answered honestly at all.

A plain object describing the query string…

…with exactly two declared params:

limit must be an integer, no lower than zero…

…and offset plays by the same rule.

…end of the declared params.

Unknown query params are tolerated, not refused — looseness is deliberate.

as const keeps the literal shape so the route's request.query types flow from it.

Schema above, the other half of the policy below.

Inside the handler now — validation already passed by the time this line runs.

And the clamp is one line: floor at 1, ceiling at MAX_LIMIT (200), default 50 when absent. No branch, no refusal, no fifth error code.

200
limit=9999 → 200, limit: 200  ·  limit=0 → 200, limit: 1

Coercible-but-extreme gets clamped to the nearest honest value — the response envelope reports the limit actually used, so the caller sees the clamp.

400
limit=nope → 400  ·  offset=-1 → 400

Uncoercible or negative gets refused with invalid_input naming the parameter. Honest requests get honest nearest answers; malformed requests get honest refusals.

💡
Why you care

Both halves are now precedent. The next route that declares a querystring schema inherits the 400 path with zero new code, and the clamp-vs-refuse line — coercible-but-extreme clamps, uncoercible/negative refuses — is written down where the next paging parameter will be designed.
