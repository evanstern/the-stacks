The Invisible Library

008 shipped a working ingestion pipeline — upload page, ticket page, an eight-stage worker, a pgvector index. Then the operator asked: “is there more to this than the home page?” There was. Nobody could see it.

A library with no card catalog

Picture a library where the books are real, shelved, and indexed — but there is no catalog, no signage, no front desk. The only way to find a book is the receipt you were handed at the door. Lose the receipt, and the book might as well not exist. That was The Stacks after 008: every submission got a claim ticket, and the ticket URL was the ONLY door back in.

📚

/library/upload — a working upload form (library.upload.tsx)

/library/uploads/:ticket — a live claim-ticket status page (library.uploads.$ticket.tsx)

POST /api/uploads — accept-then-async intake, sub-second tickets

An eight-stage worker pipeline, batch expansion, a pgvector index, an append-only event trail

The home page. Its nav had no library link — nothing pointed at any of it.

Both library pages were reachable only by typing the URL from memory.

By the time 009 converged, the live library held 57 submissions — the last page of the listing reads “Showing 51 – 57 of 57” — including this course’s two recurring actors: goblin-page.html (a standalone source, rendered as ddb-saved-html@1.0.0 · gen 1) and export-mixed.zip (a batch: 4 entries: 2 ingested · 2 skipped · 0 failed). 🔍
The gap, precisely

Nothing was broken. Every capability worked — the pipeline ingested, the ticket page polled, the events recorded. The gap was that working and visible turned out to be different properties, and 008’s spec had only gated on the first one.

The amendment: visibility becomes law

The operator’s 2026-07-09 request didn’t become a bug ticket — it became a constitutional amendment. Version 2.2.0 expands Principle V (Operator Control and Observability) with a mandate: every delivered capability must have a visibility avenue. That’s the right altitude for this fix: a nav link would have patched 008’s pages, but the failure mode — shipping a capability whose spec never asks “how does the operator see this?” — would recur in every future spec. So the rule moved into the document every spec is gated on. Here is the new text, verbatim:

CODE — .specify/memory/constitution.md (Principle V, v2.2.0) 

Every delivered capability MUST have a visibility avenue — being able to see inside is
a property of each feature, not only of the system:

- Operator-facing capabilities MUST be visible in the web UI: Records-style,
  URL-addressable, and reachable through the product's navigation. A page reachable
  only by typing its URL does not satisfy this.
- Where a web surface is not warranted or not yet feasible — developer-facing tooling,
  protocols, background machinery — the capability MUST instead be visible through
  other output: CLI output, log files, or a documented inspection path.
- Each feature's spec or plan MUST record which visibility avenue every delivered
  capability satisfies and why that avenue is the right one.
- A capability with no visibility avenue at all is incomplete, the same way a spec
  cycle without its Principle VIII learning artifact is incomplete.
PLAIN ENGLISH

The headline reframe: visibility is a property of each FEATURE, not of the system. “The system has an observability surface somewhere” no longer counts — each capability must name its own.

If it’s operator-facing, the avenue is the web UI — and the page must be reachable through navigation. A page you can only reach by typing its URL is explicitly called out as NOT satisfying this. That sentence is 008’s upload page, described exactly.

Not everything warrants a web page. Developer tooling, protocols, background machinery get a different avenue: CLI output, log files, or a documented inspection path. The mandate is an avenue, not a UI.

The avenue is recorded in the spec or plan, with a reason — and a capability with none at all is incomplete, by the same standard that makes a cycle without its learning artifact incomplete. Incomplete means the gate fails.

📜
The four-line rule

Web UI when operator-facing. CLI/logs/docs otherwise. The spec records which avenue and why. No avenue = incomplete. That’s the whole amendment — and 009 (FR-018) is the first spec gated on it.

009 itself had to file under this rule. Its converge evidence records four avenues: the library listing and upload/ticket pages (web UI, reachable via nav on every protected page), GET /api/uploads (consumed by the web UI, contract documented), the worktree env protocol (CLI output plus contracts/environment.md), and the Docker lifecycle rules (documentation). Every capability, an avenue, a reason.

🔁
Amendments come with debts

The amendment’s sync-impact record names its own retrofit: 008 predates v2.2.0, so its unreachable pages and avenue-less capabilities are explicitly tagged TODO(008-VISIBILITY-RETROFIT) — the debt 009 pays. Re-ingestion’s avenue is deliberately NOT part of that payment; it lands with the corpus-lifecycle spec. Keep that carve-out in mind — it’s the next screen’s scope line.

The slice: two parts and a hard scope line

009 is the retrofit spec that pays 008’s visibility debt — plus a second, unrelated-looking half that Module 5 will justify. The slice map:

🗂️

Part A — the library surface

A nav link on every protected page, a paged /library listing of all submissions with evidence columns (plugin, generation, section/passage counts, batch outcomes), each row linking to its ticket page. Strictly read-only.

🌲

Part B — the worktree protocol

A deterministic environment protocol so every worktree gets its own ports, compose project, and volumes by arithmetic — two stacks running side by side with zero collisions. Modules 5 and 6 live here.

🚫

Deliberately absent

No retry, no delete, no re-ingest, no corpus management — anywhere in the slice. Not deferred by omission: pinned by name to the corpus-lifecycle spec, by the operator, on 2026-07-07.

The scope line matters as much as the scope. A listing page is one temptation away from becoming a management console — a “retry” button here, a “delete” there. FR-009 slams that door:

CODE — specs/009-library-surface-env/spec.md — FR-009 

- **FR-009**: The listing capability MUST be read-only end to end: this slice adds no
  operation that creates, mutates, or deletes library content, and no re-ingestion or
  corpus-management operation (those stay pinned to the corpus-lifecycle spec per the
  2026-07-07 decision).
PLAIN ENGLISH

“End to end” is load-bearing: not just no delete button in the UI, but no new mutating route in the API, no new write path anywhere in the slice.

Re-ingestion and corpus management aren’t forgotten — they’re PINNED to a named future spec by a dated operator decision. A scope line with a forwarding address, not a shrug.

Note what this makes read-only mean here: an actual functional requirement with a testable surface (grep the API for mutating routes), not a styling choice about which buttons the listing renders.

And it was verified as a requirement, not a vibe: the converge evidence greps apps/api/src/ingestion/ and finds exactly one mutating route — 008’s POST /api/uploads. 009’s entire diff adds one GET (SC-007: zero new mutating operations).

📌
The 2026-07-07 pin

Two days BEFORE the visibility amendment, the operator had already pinned re-ingestion and corpus management to the corpus-lifecycle spec. So when 009 was specified, “make the library visible” could not quietly grow into “make the library editable” — the pin predated the temptation.
