The Navigation Shell

The fix for “no link to /library/upload” was NOT adding a link to /library/upload — it was fixing the navigation model, once, in the one place every page inherits from.

Fix the Lobby, Not the Rooms

A library can have perfect shelves, but if the lobby has no signage and no librarian, visitors wander back out. That was the-stacks before 009: the upload page worked, the ticket page worked — and both were reachable only by typing the URL.

The obvious patch is to add a link somewhere. The actual fix is structural: every authenticated route already nests under the protected layout. Put a nav header there, and every protected page — including pages that don't exist yet — renders it. The layout is the lobby. Fix the signage in the lobby and every room inherits it.

Watch the header slot in once and light up the whole tree.

The One-Place Rule

The layout was already the ONE place auth is checked — its loader turns unauthenticated traffic away before any child renders. 009 makes it the ONE place navigation renders, and the file's header comment says so out loud, with the doctrine attached.

A breath between doctrine and code.

The component itself is deliberately boring — no state, no hooks, just markup.

It returns one JSX tree…

…wrapped in a fragment, so no extra wrapper element hits the DOM.

First the nav — a real landmark element, not a styled div…

…labeled "Primary" so assistive tech can jump straight to it.

Tailwind utilities center it, cap its width, and space the links — the same rhythm as the pages below.

…and the opening tag closes.

The product name anchors the left edge — bold, but not a link.

Home is a Link — a client-side navigation, no full reload…

…with its visible label on its own line.

…and close the link.

Library gets the identical treatment, pointing at /library…

…its label, same pattern.

…and close the link.

…and close the nav.

Then <Outlet />: whatever child route matched renders below the header. Every protected page gets the shell without asking.

…close the fragment.

…and the return.

That's the whole nav model. Two links today; adding a third someday is one edit in one file, visible everywhere at once.

The Evidence Column

Reachable is half the job. The other half is that the listing page (/library) tells you something worth reaching: each row's Evidence cell renders a different story per kind of item — a discriminated union, one branch per story.

CODE — apps/web/app/routes/library.tsx:24-52 
/** The per-row evidence cell — one component per kind's story (US3). */
function Evidence({ item }: { item: LibraryListItem }) {
  if (item.kind === "batch") {
    const { ingested, skipped, failed, total } = item.entrySummary;
    return (
      <span>
        {total} entries: {ingested} ingested · {skipped} skipped ·{" "}
        <span className={failed > 0 ? "text-destructive" : ""}>{failed} failed</span>
      </span>
    );
  }
  if (item.lastError) {
    // The scrubbed why, one glance deep — full trail stays on the detail page.
    return (
      <span className="text-destructive">
        failed at {item.lastError.stage} ({item.lastError.class})
      </span>
    );
  }
  if (!item.plugin) {
    return <span className="text-muted-foreground">awaiting detect</span>;
  }
  return (
    <span>
      {item.plugin.name}@{item.plugin.version} · gen {item.generation} · {item.counts.sections}{" "}
      sections · {item.counts.chunks} passages
    </span>
  );
}
PLAIN ENGLISH

One component per kind's story — the header comment states the contract up front (US3).

Evidence takes one row item; the LibraryListItem union is what lets the branches below discriminate.

First story: batch rows.

Pull the four counts out of the entry summary…

…and render them as one line.

A plain span — this lives inside a table cell.

Batch rows summarize their entries — total, ingested, skipped…

…and the failed count goes destructive-red only when it's nonzero. Zero failures shouldn't shout.

…close the span.

…and the return.

End of the batch branch — everything after this is a single source.

Second story: a source with a lastError.

The why-comment says it: the scrubbed why, one glance deep — the full stage trail stays on the detail page where it belongs.

Render it…

…in destructive red, because a failure should read as one.

Which stage, which error class — the scrubbed why, nothing more.

…close the span.

…and the return.

End of the failure branch.

Third story: no plugin yet? The worker hasn't run detect.

Say exactly that — "awaiting detect", muted — instead of faking a blank.

…end the pending branch.

Fourth story, the fall-through: the happy path.

Again a plain span…

…the receipt: plugin@version, generation…

…and how much of the corpus this source contributes right now — sections and passages.

…close the span.

…and the return.

One component, four honest stories. Order matters: batch first (it has no lastError shape), then failure, then pending, then success.

Honesty at the Edges

Two edges betray a listing page: the empty case and the too-big case. /library handles both by telling the truth.

Empty: a blank page lies — it reads as broken. An empty library is a normal answer, so the empty state says so and points at the exit: "Nothing in the library yet. Upload your first document — a saved HTML page, Markdown, plain text, or a ZIP of them." The upload page links back to the listing too — the two surfaces cross-reference each other (US1 AC-5).

Too big: paging. And the offset doesn't live in component state — it rides the URL, because paging is URL-addressable state.

CODE — apps/web/app/routes/library.tsx:54-62 
export async function loader({ request }: Route.LoaderArgs) {
  // Offset rides the page URL so paging is addressable/bookmarkable
  // (Principle V: URL-addressable state). The API clamps; we only parse.
  const url = new URL(request.url);
  const rawOffset = Number.parseInt(url.searchParams.get("offset") ?? "0", 10);
  const offset = Number.isNaN(rawOffset) || rawOffset < 0 ? 0 : rawOffset;
  const page = await listUploads(request, { offset });
  return { page };
}
PLAIN ENGLISH

A server-side loader — the browser never calls the API itself (007 FR-019).

The why-comment carries the doctrine: paging state lives in the URL, so any page is bookmarkable…

…and the trust split is explicit — the API clamps limits; the loader only parses what the URL says.

Read the page's own URL — the offset arrives as a query parameter, not component state.

Parse it as a base-10 integer, defaulting to "0" when absent.

Garbage or negative input degrades to page one — never an error page for a mangled bookmark.

One call through the server-only API client, offset forwarded; the session cookie rides along.

Hand the page to the component — data flows loader → render, nothing else.

…and that's the whole loader: nine lines, URL in, page out.

The live run proves the arithmetic against the real ledger: at 57 submissions, the last page renders Showing 51 – 57 of 57 with a Newer link only — no Older link past the end, because the links render conditionally off the same total the API reported. The count is the real ledger size, not a guess.

🧭
One polling surface is enough

Deliberate non-feature: the listing does NOT live-poll. It reflects reality at load/refresh; the ticket page from 008 stays the live-updating surface. Two surfaces polling the same pipeline is how you get flicker, drift, and double the API chatter for zero new information.

Check the Model

You add /library/settings next month. What nav work is needed for it to render inside the navigation shell?

The shell makes every page reachable; the Evidence column makes every row honest; the URL makes every page of the ledger addressable. All of that assumed the request itself was well-formed. Module 4 asks the rude question: what does this surface do when it isn't — offset=banana, a ticket that never existed, a session that expired mid-click? That's Whose Fault Is It — typed refusals.
