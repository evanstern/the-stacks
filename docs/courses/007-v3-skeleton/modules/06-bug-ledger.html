Six bugs, all green at review time

Every one of these six bugs shipped in code that type-checked, passed unit tests, and looked correct in review. They were caught the only way these get caught: by starting the real stack and watching it fail.

Silence is the symptom

Bugs #1 and #4 share a shape: nothing screamed. Stopping Postgres didn’t produce an error page — it produced a dead process, because node-postgres raises an idle-client error that nobody was listening for.

CODE — v3/packages/db/src/client.ts 
  const pool = new Pool({ connectionString });
  // node-postgres crashes the process on an unhandled idle-client error
  // otherwise; a dead DB must surface as dependency_down, not a process exit.
  pool.on("error", (err) => {
    console.error("Postgres pool error:", err);
  });
PLAIN ENGLISH

Create the shared pool every query in the process flows through.

The comment is the scar: stopping Postgres once killed the entire API, because an idle pooled connection emitted 'error' with no listener — and Node treats that as fatal.

Registering the listener is the whole fix. There’s nothing to recover here — the pool discards the dead connection on its own.

Log it and stay alive. A dead DB now surfaces as 503 dependency_down from the health check, not a process exit — and a regression test stops Postgres to prove it.

Bug #4 was quieter still. The worker’s compose env was missing EMBEDDING_PROVIDER, so resolveModelRole threw a plain Error before the try/catch that writes failure outcomes — the job silently retried while the run sat at running forever.

CODE — v3/apps/worker/src/handlers/skeleton-check.ts 
  try {
    await runSkeletonCheck(db, runId);
  } catch (error) {
    // Anything NOT already handled below (e.g. a misconfigured env var
    // resolving the model role) must still fail the run — otherwise it's
    // stuck at "running" forever while only the underlying job retries.
    // This is a scar, not speculation: a missing env var once produced exactly
    // that silent hang. DomainErrors skip this — runSkeletonCheck already
    // called failRun for those; double-failing would clobber the real outcome.
    if (!(error instanceof DomainError)) {
      await failRun(
        db,
        runId,
        new DomainError({
          class: "internal_fault",
          seam: "inference",
          message: "Unexpected error before inference.",
          cause: error,
        }),
      );
    }
    // Re-throw so the poll loop also fails the JOB (retry/backoff lives there).
    throw error;
PLAIN ENGLISH

Wrap the entire check in one boundary try — including everything that runs before the handler’s own error handling exists.

The comments narrate the scar: the missing env var threw here, the job retried, and the run showed running forever. Absence of failure, the worst kind of bug.

DomainErrors skip this branch — runSkeletonCheck already called failRun with the real typed outcome, and double-failing would clobber it.

Anything else gets failed explicitly, wrapped as a typed internal_fault pinned to the seam it never reached — original error preserved as cause.

Then re-throw: the poll loop fails the JOB (retry/backoff lives there), this block fails the RUN. Two ledgers, both written.

💡
Aha: the worst failure mode isn’t a crash

It’s a system that keeps looking busy. A crashed process gets noticed; a run stuck at running gets a shrug and a refresh. Both fixes here do the same thing: convert silence into a typed, visible failure.

When the tool’s state lies

Bugs #3 and #6 share a different shape: your file is correct, but a tool between you and the runtime disagrees. Both fixes live in the code as self-narrating comments — scar tissue you can read in place.

Bug #3. bcrypt hashes start $2b$10$… — and Docker Compose interpolates $ in .env values, silently mangling the hash. Every login failed with a correct password, so the escaping rule now lives in the env template:

SCAR — v3/.env.example 
# IMPORTANT: Docker Compose interpolates .env values, so every literal `$` in
# the hash must be escaped as `$$` (e.g. `$2b$10$abc...` -> `$$2b$$10$$abc...`)

Bug #6. drizzle-kit generate computes the next migration prefix from the journal’s lastEntry.idx + 1. A manual rename (0000→0001_init) left idx=0, so the next generate minted 0001 — colliding with the existing file. Found by actually running generate, not by eyeballing:

SCAR — v3/packages/db/drizzle.config.ts 
 * Numbering gotcha: drizzle-kit derives the next migration prefix from
 * migrations/meta/_journal.json (lastEntry.idx + 1). The journal's idx was
 * deliberately set to 1 to match the 0001_init tag after a manual file
 * rename fixed a numbering collision — don't "correct" it back to 0, or the
 * next generate will mint a duplicate prefix.

Two more scars round out the ledger. Bug #2: every package’s tsconfig extends ../../tsconfig.base.json, but the Dockerfiles never COPY’d it — invisible locally, fatal in the image:

SCAR — v3/apps/web/Dockerfile 
# tsconfig.base.json is copied EXPLICITLY: every tsconfig extends it, and its
# absence is invisible locally but fatal in the image build — this exact line
# was missing once and broke the web build with a TSConfckParseError.
COPY pnpm-workspace.yaml package.json pnpm-lock.yaml tsconfig.base.json ./

Bug #5: web tests hung because RR7’s client runtime brand-checks Request/AbortSignal, and jsdom supplies cross-realm copies that always fail instanceof. The fix was swapping the test DOM itself:

SCAR — v3/apps/web/vitest.config.ts 
 * environment is happy-dom, NOT jsdom, and that choice is load-bearing:
 * RR7's client runtime constructs Request/AbortSignal objects, and jsdom's
 * realm-separated globals fail RR7's instanceof/brand checks (a real
 * cross-realm bug hit during this feature). happy-dom shares Node's
 * fetch primitives, so those checks pass.

Why the comments stay

Each of these comments costs a few lines and prevents a repeat — the next person who “cleans up” the journal idx or trims the COPY line reads the scar first. The ledger lives where the bug lived.
