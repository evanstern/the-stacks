Silence is the symptom

Bugs #1 and #4 share a shape: nothing screamed. Stopping Postgres didn’t produce an error page — it produced a dead process, because node-postgres raises an idle-client error that nobody was listening for.

CODE — v3/packages/db/src/client.ts 
  const pool = new Pool({ connectionString });
  // node-postgres crashes the process on an unhandled idle-client error
  // otherwise; a dead DB must surface as dependency_down, not a process exit.
  pool.on("error", (err) => {
    console.error("Postgres pool error:", err);
  });
PLAIN ENGLISH

Create the shared pool every query in the process flows through — which makes its failure mode a process-wide failure mode.

The comment is the scar: an idle pooled connection emitted 'error' with no listener, and Node treats an unhandled 'error' event as fatal. Stopping Postgres once killed the entire API.

The contract this fix restores: a dead DB surfaces as 503 dependency_down from the readiness check — never as a process exit.

Registering the listener is the whole fix. There’s nothing to recover here — the pool discards the dead connection on its own.

Log it and stay alive. A regression test in packages/db/test/client.test.ts now stops Postgres to prove the process survives.

That’s the entire listener — its presence, not its logic, is what keeps the process up.

Bug #4 was quieter still. The worker’s compose env was missing EMBEDDING_PROVIDER, so resolveModelRole threw a plain Error before the try/catch that writes failure outcomes — the job silently retried while the run sat at running forever.

CODE — v3/apps/worker/src/handlers/skeleton-check.ts 
  try {
    await runSkeletonCheck(db, runId);
  } catch (error) {
    // Anything NOT already handled below (e.g. a misconfigured env var
    // resolving the model role) must still fail the run — otherwise it's
    // stuck at "running" forever while only the underlying job retries.
    // This is a scar, not speculation: a missing env var once produced exactly
    // that silent hang. DomainErrors skip this — runSkeletonCheck already
    // called failRun for those; double-failing would clobber the real outcome.
    if (!(error instanceof DomainError)) {
      await failRun(
        db,
        runId,
        new DomainError({
          class: "internal_fault",
          seam: "inference",
          message: "Unexpected error before inference.",
          cause: error,
        }),
      );
    }
    // Re-throw so the poll loop also fails the JOB (retry/backoff lives there).
    throw error;
  }
PLAIN ENGLISH

One boundary try wraps the entire check — including everything that runs before the handler’s own error handling exists.

The whole unit of work. Inside, runSkeletonCheck fails runs itself with typed DomainErrors; the gap was everything outside that.

The catch that was missing. Bug #4 lived exactly here: resolveModelRole threw a plain Error before any failure-writing code ran.

The comments narrate the scar in place. First the rule: an unhandled error must still fail the run…

…because otherwise only the job layer reacts, retrying on its own schedule…

…while the run row shows running forever. Absence of failure — the worst kind of bug.

“A scar, not speculation” — this codebase’s marker for comments earned by a real incident.

The carve-out is deliberate: DomainErrors were already handled downstream…

…runSkeletonCheck called failRun with the real typed outcome; failing again here would clobber it.

So only untyped errors take this branch — exactly the class that used to vanish.

Fail the run explicitly, through the same path every other failure uses.

Same database handle the check ran on.

Same run the job was carrying.

Wrap the raw error as a typed outcome…

…classed internal_fault — one of the four error classes from module 5’s vocabulary…

…pinned to the seam it died before reaching…

…with an operator-readable message…

…and the original error preserved as cause for the logs.

Close the DomainError…

…and the failRun call. The run is now visibly, durably failed.

End of the untyped-error branch.

The second half of the fix, stated in code: run and job are separate ledgers.

Re-throw so the poll loop fails the JOB too — retry/backoff lives there, not here. Two ledgers, both written.

Close the catch. From here, no error class can leave a run stuck at running.

Aha: the worst failure mode isn’t a crash

It’s a system that keeps looking busy. A crashed process gets noticed; a run stuck at running gets a shrug and a refresh. Both fixes here do the same thing: convert silence into a typed, visible failure.

When the tool’s state lies

Bugs #3 and #6 share a different shape: your file is correct, but a tool between you and the runtime disagrees. Both fixes live in the code as self-narrating comments — scar tissue you can read in place.

Bug #3. bcrypt hashes start $2b$10$… — and Docker Compose interpolates $ in .env values, silently mangling the hash. Every login failed with a correct password, so the escaping rule now lives in the env template:

SCAR — v3/.env.example 
# IMPORTANT: Docker Compose interpolates .env values, so every literal `$` in
# the hash must be escaped as `$$` (e.g. `$2b$10$abc...` -> `$$2b$$10$$abc...`)
PLAIN ENGLISH

Compose interpolates .env values themselves — it has to, to support cross-references like ML_EMBEDDING_MODEL=${EMBEDDING_MODEL_ID} — so a bcrypt hash’s $2b$10$ reads as variable references and silently corrupts.

The rule lives in the template itself, right where the next operator will paste a hash: escape every literal $ as $$.

Bug #6. drizzle-kit generate computes the next migration prefix from the journal’s lastEntry.idx + 1. A manual rename (0000→0001_init) left idx=0, so the next generate minted 0001 — colliding with the existing file. Found by actually running generate, not by eyeballing:

SCAR — v3/packages/db/drizzle.config.ts 
/**
// …
 * Numbering gotcha: drizzle-kit derives the next migration prefix from
 * migrations/meta/_journal.json (lastEntry.idx + 1). The journal's idx was
 * deliberately set to 1 to match the 0001_init tag after a manual file
 * rename fixed a numbering collision — don't "correct" it back to 0, or the
 * next generate will mint a duplicate prefix.
 */
PLAIN ENGLISH

The scar is a stanza in the config file’s doc-comment header.

…the header’s first stanza omitted — it explains this file is dev tooling only, never imported by product code…

The trap: drizzle-kit generate computes the next migration filename prefix from tool-owned state…

…the journal’s lastEntry.idx + 1 — not from the filenames sitting in the directory.

The counter was deliberately corrected to 1 after a manual 0000→0001_init rename left it stale…

…so the comment guards the fix: reverting idx to 0 looks like a cleanup…

…and would make the next generate mint a duplicate prefix. Found by actually running generate during convergence, not by eyeballing.

End of the stanza — the ledger entry lives where the trap lives.

Two more scars round out the ledger. Bug #2: every package’s tsconfig extends ../../tsconfig.base.json, but the Dockerfiles never COPY’d it — invisible locally, fatal in the image:

SCAR — v3/apps/web/Dockerfile 
# tsconfig.base.json is copied EXPLICITLY: every tsconfig extends it, and its
# absence is invisible locally but fatal in the image build — this exact line
# was missing once and broke the web build with a TSConfckParseError.
COPY pnpm-workspace.yaml package.json pnpm-lock.yaml tsconfig.base.json ./
PLAIN ENGLISH

The scar comment rides directly on the line it protects — every package’s tsconfig extends this one file.

Invisible locally because the file exists on your disk; fatal in the image because Docker only sees what COPY declares.

The incident, named: this exact filename missing from the list broke the web image build with TSConfckParseError — all three Node Dockerfiles got the fix.

The lesson in one line: in a monorepo image build, the COPY list is a dependency declaration — audit it against every extends and workspace reference.

Bug #5: web tests hung because RR7’s client runtime brand-checks Request/AbortSignal, and jsdom supplies cross-realm copies that always fail instanceof. The fix was swapping the test DOM itself:

SCAR — v3/apps/web/vitest.config.ts 
/**
// …
 * environment is happy-dom, NOT jsdom, and that choice is load-bearing:
 * RR7's client runtime constructs Request/AbortSignal objects, and jsdom's
 * realm-separated globals fail RR7's instanceof/brand checks (a real
 * cross-realm bug hit during this feature). happy-dom shares Node's
 * fetch primitives, so those checks pass.
 */
PLAIN ENGLISH

Another config-header scar, this one from the web app’s test config.

…the header’s first stanza omitted — it explains why vitest’s config is deliberately separate from vite’s…

The load-bearing choice, stated up front: happy-dom, NOT the reflex default jsdom.

RR7’s client runtime constructs real Request/AbortSignal objects and brand-checks them…

…jsdom supplies its own realm’s copies, so every instanceof against them is false — the tests hung rather than failed.

A real incident hit during this feature — not a hypothetical…

…happy-dom shares Node’s own fetch primitives, so the brand checks pass.

Comment closes. The choice is defended in place so nobody “normalizes” it back to jsdom.
