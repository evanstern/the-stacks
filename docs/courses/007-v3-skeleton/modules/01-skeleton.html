The pencil sketch before the painting

A walking skeleton is a pencil sketch: every structural line present and load-bearing, zero rendering. No feature exists yet, but every line the features will hang on has already been drawn — and proven under weight.

The one command 
docker compose up -d --build --wait

--wait blocks until every container reports healthy — or fails loudly naming the one that never did. Five services come up:

🐘

Postgres + pgvector

One database, three roles: system of record, vector store, and job queue.

⚡

Fastify API

The HTTP edge on :4401 — validates env, migrates the schema, owns sessions and every route.

🧠

ML Sidecar

The only Python in the stack (D2) — embeds text behind a thin HTTP wall.

⚙️

Worker

No HTTP surface at all — polls the queue, a plain Postgres table (D12), and does the heavy lifting.

🖥️

RR7 SSR Web

Server-rendered React Router 7 front end on :4400 — talks only to the API.

Every arrow is a gate

Each edge in this chain is a compose service_healthy condition — a hard gate, not a race. So “the stack is up” provably means the schema is current and every dependency answers.

What the API does before it will talk to you

The whole doctrine lives in one function — the order of these lines is the contract. From v3/apps/api/src/main.ts:

CODE — v3/apps/api/src/main.ts 

async function main(): Promise<void> {
  assertRequiredEnv();
  // Fails fast (naming the variable) if the embedding role's env is missing/malformed.
  // The API never embeds — the worker does — but validating here surfaces a bad
  // embedding config at deploy time instead of on the first queued run.
  resolveModelRole("embedding");
  const { db, pool } = createDbClient(process.env.DATABASE_URL!);
  // Migrations apply before the port binds — /ready therefore implies
  // schema-current (research R10, FR-002).
  await runMigrations(db);
  const app = await buildApp({
    db,
    pool,
    operatorPasswordHash: process.env.OPERATOR_PASSWORD_HASH!,
    sessionSecret: process.env.SESSION_SECRET!,
    sessionCookieSecure: process.env.SESSION_COOKIE_SECURE === "true",
  });
  const port = Number.parseInt(process.env.V3_API_PORT ?? "4401", 10);
  await app.listen({ host: "0.0.0.0", port });
}
PLAIN ENGLISH

One async boot function owns the whole process. Any throw below kills the container — and compose's --wait reports exactly which link died.

Env gate first: collect and report ALL missing vars in one error — not one per restart.

A why-comment where the doctrine lives: fail fast, and name the offending variable in the error…

…while conceding the oddity up front — the API never embeds; the Worker does…

…so the call below is purely about timing: surface a bad config at deploy, not on the first queued run.

Validate a model role this service never uses. Model roles resolve env-first — no hardcoded model IDs (Principle VII / D14) — so a malformed embedding config stops boot here instead of stranding a Worker job hours later.

Open the Drizzle client and its raw pg pool — compose's service_healthy gate guarantees Postgres already answers.

Migrate-then-listen, stated as doctrine exactly where it is enforced…

…with the receipts cited: research R10 and FR-002 pin this ordering as a requirement, not a taste.

The await that makes /ready a schema certificate: no applied migrations, no port.

Assemble the Fastify app via the pure composition root (app.ts) — the process entry injects everything, so tests never come through main().

The Drizzle handle the routes query with…

…and the raw pool — /ready uses it for the cheapest possible “is Postgres there” probe, a bare select 1.

Auth material from env: the single operator's bcrypt hash (every $ escaped as $$ in .env, or compose interpolation mangles it)…

…and the secret that seals the sealed cookie sessions.

The cookie Secure flag comes from env, so dev over http and prod over https share one code path — only the prod overlay flips it true.

That is the app's entire dependency list; everything else is constructed inside buildApp.

Port from env with a 4401 default — the same value compose publishes and the healthcheck probes.

Only NOW does the socket open. Anything that answers on :4401 has, by construction, already migrated.

End of boot — from here, “is it healthy” belongs to the compose healthcheck probing the /ready this ordering just made meaningful.

💡
/ready is a proof, not a ping

Because migrations run before listen(), a green readiness check certifies schema-currency — not merely “a process exists.” Order of operations turned a healthcheck into evidence.

Two healthchecks that look wrong (and are not)

Open v3/docker-compose.yml and two checks stand out. Both are workarounds for a constraint — and both encode a definition of “healthy” that is stricter than “the process runs.”

CODE — v3/docker-compose.yml (api healthcheck) 

    # node -e fetch(): no curl/wget in node:22-slim, so probe with node itself.
    # /ready (not /health) so "healthy" means migrated + serving.
    healthcheck:
      test:
        [
          "CMD",
          "node",
          "-e",
          "fetch('http://localhost:4401/ready').then(r=>process.exit(r.ok?0:1)).catch(()=>process.exit(1))",
        ]
PLAIN ENGLISH

The constraint, documented in place: node:22-slim ships neither curl nor wget — so the probe is a one-liner of Node itself.

And the target is /ready, not /health — “healthy” here inherits the migrate-then-listen guarantee from the boot sequence.

The compose health block: this command's exit code is the signal every downstream service_healthy gate reads.

The probe command…

…in exec-array form: an argv handed straight to the container, no shell in between.

CMD, not CMD-SHELL — run the binary directly.

The app's own runtime doubles as the probe binary.

Evaluate a one-line script…

…that maps HTTP truth onto exit codes: a 2xx exits 0, anything else — including a refused connection, via the catch — exits 1. Docker turns that into healthy or unhealthy.

Five argv tokens, zero packages installed just to ask “are you up?”

CODE — v3/docker-compose.yml (worker healthcheck) 

    # Worker has no HTTP surface, so health = "heartbeat file touched within the
    # last minute". The poll loop refreshes /tmp/worker-heartbeat each tick; if
    # the loop wedges, the file goes stale and the check fails.
    healthcheck:
      test: ["CMD-SHELL", "find /tmp/worker-heartbeat -mmin -1 | grep -q ."]
PLAIN ENGLISH

The Worker exposes no port, so there is nothing to fetch — the comment redefines health for a poll-loop service…

…as freshness: the loop touches /tmp/worker-heartbeat on every tick…

…so a wedged loop stops touching it, the mtime goes stale, and the check fails — while the process is still technically “alive.”

Same compose mechanism as the API's check — a different definition of healthy plugs into the identical service_healthy gating.

find -mmin -1 matches only if the file changed within the last minute; grep -q . converts “matched anything” into exit 0. Health means “the loop is actually looping” — strictly stronger than “the process runs.”
