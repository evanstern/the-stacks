The accept side: 202 and out

The POST handler is pure accept-then-async: it creates the run and its job in one transaction, so there is no window where one exists without the other. Then it emits queued — but only after the commit.

CODE — v3/apps/api/src/skeleton-checks/routes.ts 
  app.post("/api/skeleton-checks", async (_request, reply) => {
    // Accept-then-async: run + job are created in one transaction (FR-009); the
    // worker does the actual seam-crossing work off this queue entry (D12).
    // One transaction means no window where a run exists without its job (or
    // vice versa) — a crash mid-accept leaves nothing behind. The job is
    // inserted first, then back-filled with the runId, because each row needs
    // the other's id and jobs.payload is the cheaper side to update.
    const run = await db.transaction(async (tx) => {
      const [job] = await tx.insert(jobs).values({ kind: "skeleton_check", payload: {} }).returning();
      const [createdRun] = await tx
        .insert(skeletonCheckRuns)
        .values({ jobId: job!.id, inputText: SKELETON_CHECK_INPUT_TEXT })
        .returning();
      await tx
        .update(jobs)
        .set({ payload: { runId: createdRun!.id } })
        .where(sql`${jobs.id} = ${job!.id}`);
      return createdRun!;
    });
    // 'queued' is emitted AFTER the commit: an event must never describe a run
    // that a rollback could erase. Events are observability, not state — losing
    // this one to a crash is acceptable; a phantom event is not.
    await recordEvent(db, { runId: run.id, seam: "queued" });
    reply.code(202);
    return { run: { id: run.id, status: run.status, createdAt: run.createdAt } };
  });

Register the POST handler. The _request underscore says it all: the body is ignored — the tracer sentence is fixed.

The comment carries the doctrine: accept-then-async — one transaction creates run + job here (FR-009)…

…the worker does the actual seam-crossing later, off this queue entry (D12, Principle IV).

…and one transaction means no window where a run exists without its job…

…or vice versa — a crash mid-accept leaves nothing behind, in either direction.

…the insertion order is declared up front too: job first, runId back-filled…

…because each row needs the other’s id, and jobs.payload is the cheaper side to update.

Open the single transaction — everything inside commits together or vanishes together.

Insert the job first, payload empty; its id must exist before the run row can point at it.

Second insert begins: the run itself…

…into skeleton_check_runs…

…wired to the job via jobId and carrying the fixed tracer sentence…

…and returning() hands the created row back so the route can answer with its id.

Third statement: back to the job row we created empty…

…an UPDATE on jobs…

…back-filling the payload with the run id — the circular reference resolved on the cheap side…

…scoped to exactly this job. The sql-tag interpolation is a bound parameter, not string splicing.

Return the run out of the callback — this value outlives the transaction.

The commit point. Run and job now exist atomically, or not at all.

Second doctrine comment: queued is emitted only AFTER the commit…

…because an event must never describe a run a rollback could erase. Events are observability, not state…

…so losing this event to a crash is acceptable; a phantom event is not. The callout below unpacks the asymmetry.

The emission itself: recordEvent — the sole writer to the append-only events table — records seam #1.

202 Accepted means “recorded, work will happen” — not “done”.

Hand back id, status, createdAt — everything the web page needs to start polling.

Handler ends ~5ms after it began. The other five seams belong to the worker.

💡
Aha: events are observability, not state

Losing an event to a crash is acceptable; a phantom event describing a rolled-back run is not. That asymmetry is why recordEvent lives outside the transaction — the queue row is the truth, the event is the breadcrumb.

The vector id in step 4 is not random. It is a deterministic id, which is what makes re-runs idempotent:

CODE — v3/packages/core/src/skeleton-check.ts 
export function deriveVectorId(input: DeriveVectorIdInput): string {
  const material = `${input.inputText}\n${input.provider}/${input.model}/${input.dimensions}`;
  return createHash("sha256").update(material).digest("hex");
}
PLAIN ENGLISH

Identity is a pure function of the input — no UUID, no clock.

The hash material is the sentence plus its embedding space: provider, model, dimensions. Same text under a different model gets a different id.

sha256 it. Run the check twice: same hash both times — that is where 8d6a97… comes from.

So the second write is a collision, not a duplicate.

The write side absorbs that collision with ON CONFLICT DO NOTHING — the second run’s vector_write event simply carries deduplicated: true:

CODE — v3/apps/worker/src/handlers/skeleton-check.ts 
  const insertedRows = await db

PLAIN ENGLISH

Capture the insert’s result — how many rows come back is the dedup signal, so it matters.

Target skeleton_vectors, the pgvector table.

Open the row…

…keyed by the deterministic id from deriveVectorId — this primary key is what turns a re-run into a collision instead of a duplicate.

…the raw text rides along, so the row is self-describing without a join.

…the 384 floats the sidecar just returned.

Then the row’s embedding-space coordinates: provider…

…model — resolved env-first from the model role, never hardcoded (Principle VII / D14)…

…and dimensions. The readback filter will match on all three.

Row closed.

If that id already exists, skip silently — no error, no duplicate; the second run’s write is a no-op.

returning() comes back empty on a dedup — that empty array is exactly how the vector_write event learns to say deduplicated: true.

The read-back: closing the loop

Writing the vector proves pgvector accepts our floats. Reading it back by similarity proves the whole space is queryable — and the search is deliberately scoped.

CODE — v3/apps/worker/src/handlers/skeleton-check.ts 
  const readbackStart = Date.now();
  const vectorLiteral = `[${embedding.join(",")}]`;
  const distanceExpr = sql<number>`${skeletonVectors.embedding} <=> ${vectorLiteral}::vector`;
  const [readback] = await db
    .select({ id: skeletonVectors.id, distance: distanceExpr })
    .from(skeletonVectors)
    .where(
      sql`${skeletonVectors.embeddingModel} = ${role.modelId}
        AND ${skeletonVectors.embeddingProvider} = ${role.provider}
        AND ${skeletonVectors.embeddingDimensions} = ${role.dimensions}`,
    )
    .orderBy(distanceExpr)
    .limit(1);
PLAIN ENGLISH

Start the clock — the readback duration lands in the vector_readback event detail.

Serialize the embedding we just wrote into pgvector’s [f1,f2,…] literal form.

<=> is cosine distance — build the nearest-neighbor expression once, so SELECT and ORDER BY share it.

Ask for exactly one row and destructure it straight off the result.

All we want back: the neighbor’s id and how far away it is.

Search the same table the write just landed in.

But not the whole table — the filter opens…

…only vectors from the same model,

…the same provider,

…and the same dimensions. Other embedding spaces are invisible to this query.

Filter closed — config drift is now a detectable mismatch, never a silent cross-space comparison (Principle VII).

Sort by that distance expression, nearest first.

Take one. If the loop is closed, it is our own vector — 8d6a97… at distance 0.0.

Distance 0.0 means we found the exact vector we just wrote. Accept, queue, claim, infer, write, read back — the loop is closed.

🎯
The filter is the point

Change the embedding model in env and old vectors fall out of scope as a detectable mismatch — never a silent cross-space comparison (Principle VII). A distance between two models’ vectors is numerically valid and semantically meaningless.
