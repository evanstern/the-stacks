The Monorepo Map

Seven workspaces, three forbidden moves, and the 90-line script that fails CI when you make one.

Load-Bearing Walls

Nothing stops apps/web from importing the DB schema — TypeScript would happily compile it. A 90-line script is the only thing standing between this architecture and entropy, on purpose.

In a monorepo every package can import every other; which walls are load-bearing is a decision. Here the walking skeleton enforces that decision the way a building inspector fails a permit: mechanically, at CI time.

The Building Inspector: check-boundaries.mjs

The three walls from the map, written down as an executable comment. Each rule is pinned to the doctrine that demanded it.

v3/scripts/check-boundaries.mjs 
#!/usr/bin/env node
// Fails the build on structural boundary violations the type system can't
// catch on its own (FR-019, FR-005, SC-006). Run via `pnpm verify`.
//
// Three rules, each pinned to constitutional doctrine:
//   1. apps/web may not import @stacks/db or reach into api/worker/ml source
//      (FR-019): the web app is a client of the HTTP contract, nothing more.
//      TypeScript can't stop a workspace import that would quietly couple
//      the UI to the schema, so this scan does.
//   2. No v3 file may import across the v3/ root into v2 code (decision D1,
//      FR-005): v3 is a clean rebuild, not a graft. A single relative import
//      escaping v3/ would silently re-entangle the old tree.
//   3. No hardcoded model identifiers in product code (Principle VII, SC-006):
//      models are configuration, resolved env-first via @stacks/core
//      model-roles. A literal model id in source would bypass FR-013/D14's
//      fail-fast env resolution and pin deploys to one model.
// These are lexical scans over import/source text — cheap, dependency-free,
// and intentionally blunt: they enforce architecture by failing CI, in the
// same spirit as the append-only-by-construction event table.
What it buys you

A build gate: it kills pnpm verify the moment a structural rule breaks — before any human reviews the code.

Rule 1: web can only ever break the API contract, never the schema. The UI physically cannot couple itself to the database.

TypeScript would compile that import happily — the scan exists precisely because the compiler won't object.

Rule 2: v2 can be deleted without grep-ing v3. No import ever crosses the v3/ root, so retirement stays a delete.

Rule 3: swapping the embedding model is an env change, not a code change. Model ids live in config, never in source.

Deliberately blunt: plain text scans, no AST, no dependencies. Cheap enough to run on every single verify.

The last line names the doctrine — make invariants structural, and you never have to remember them.

💡
Enforce architecture by failing CI

"Don't import that" is a convention; a script that fails the build is a fact. Same spirit as the append-only events table you'll meet in Module 4: invariants by construction, not by discipline.

The ban is documented on both sides of the wall — @stacks/db announces it in its own barrel header.

v3/packages/db/src/index.ts 
/**
 * @stacks/db — the persistence seam: drizzle schemas, the pooled client,
 * boot-time migrations, and the two data disciplines of the skeleton (the
 * Postgres-backed queue, D12, and the append-only event trail, Principle IV).
 * Consumed by apps/api and apps/worker only; apps/web is forbidden from
 * importing this package (FR-019, enforced by scripts/check-boundaries.mjs).
 */
What it buys you

Everything that touches Postgres lives behind this one barrel: schemas, the pooled client, boot migrations.

It also hosts the skeleton's two data disciplines — the job queue and the append-only event trail.

The header names its own enforcement: who may import it, and exactly which script fails CI if anyone else tries.
