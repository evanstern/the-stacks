What it buys you

A build gate: it kills pnpm verify the moment a structural rule breaks — before any human reviews the code.

Rule 1: web can only ever break the API contract, never the schema. The UI physically cannot couple itself to the database.

TypeScript would compile that import happily — the scan exists precisely because the compiler won't object.

Rule 2: v2 can be deleted without grep-ing v3. No import ever crosses the v3/ root, so retirement stays a delete.

The accept side: 202 and out

The POST handler is pure accept-then-async: it creates the run and its job in one transaction, so there is no window where one exists without the other. Then it emits queued — but only after the commit.

CODE — v3/apps/api/src/skeleton-checks/routes.ts 
  app.post("/api/skeleton-checks", async (_request, reply) => {
    // Accept-then-async: run + job are created in one transaction (FR-009); the
    // worker does the actual seam-crossing work off this queue entry (D12).
    // One transaction means no window where a run exists without its job (or
    // vice versa) — a crash mid-accept leaves nothing behind. The job is
    // inserted first, then back-filled with the runId, because each row needs
    // the other's id and jobs.payload is the cheaper side to update.
    const run = await db.transaction(async (tx) => {
      const [job] = await tx.insert(jobs).values({ kind: "skeleton_check", payload: {} }).returning();
      const [createdRun] = await tx
        .insert(skeletonCheckRuns)
        .values({ jobId: job!.id, inputText: SKELETON_CHECK_INPUT_TEXT })
        .returning();
      await tx
        .update(jobs)
        .set({ payload: { runId: createdRun!.id } })
        .where(sql`${jobs.id} = ${job!.id}`);
      return createdRun!;
    });

    // 'queued' is emitted AFTER the commit: an event must never describe a run
    // that a rollback could erase. Events are observability, not state — losing
    // this one to a crash is acceptable; a phantom event is not.
    await recordEvent(db, { runId: run.id, seam: "queued" });

    reply.code(202);
    return { run: { id: run.id, status: run.status, createdAt: run.createdAt } };
  });
PLAIN ENGLISH

Register the POST handler. The _request underscore says it all: the body is ignored — the input sentence is fixed.

The comment is the contract: one transaction creates run + job here; the worker does the real seam-crossing later, off this queue entry.

Open a single transaction — everything inside commits together or vanishes together. A crash mid-accept leaves nothing behind.

Insert the job first with an empty payload; we need its id before the run row can point at it.

Insert the run, wired to the job via jobId and carrying the fixed sentence.

Back-fill the job payload with the run id — each row needs the other, and jobs.payload is the cheaper side to update.

Return the run out of the callback. This is the commit point: run and job now exist atomically.

Only after the commit does queued get recorded — an event must never describe a run a rollback could erase.

202 Accepted means “recorded, work will happen” — not “done”. Hand back the run id the web page will poll.

💡
Aha: events are observability, not state

Losing an event to a crash is acceptable; a phantom event describing a rolled-back run is not. That asymmetry is why recordEvent lives outside the transaction — the queue row is the truth, the event is the breadcrumb.

The vector id in step 4 is not random. It is a deterministic id, which is what makes re-runs idempotent:

CODE — v3/packages/core/src/skeleton-check.ts 
export function deriveVectorId(input: DeriveVectorIdInput): string {
  const material = `${input.inputText}\n${input.provider}/${input.model}/${input.dimensions}`;
  return createHash("sha256").update(material).digest("hex");
}
PLAIN ENGLISH

Identity is a pure function of the input — no UUID, no clock.

The hash material is the sentence plus its embedding space: provider, model, dimensions. Same text under a different model gets a different id.

sha256 it. Run the check twice: same hash both times — that is where 8d6a97… comes from.

So the second write is a collision, not a duplicate.

The write side absorbs that collision with ON CONFLICT DO NOTHING — the second run’s vector_write event simply carries deduplicated: true:

04

The Postgres Queue

No broker, no second datastore — a jobs table, ~60 lines of SQL, and three mechanisms that survive crashes.

A queue with no queue software

There’s no Redis, no RabbitMQ, no SQS here. The queue is a table named jobs and about 60 lines of SQL — chosen deliberately (D12), because v2 proved the pattern and a second datastore is a second thing that pages you.

Picture a pharmacy prescription rack. Orders go on the rack, any pharmacist grabs the next slip — and the rack’s trick is that two pharmacists can reach in at once and never grab the same one.

🤲

Grab without colliding

SKIP LOCKED — the first transaction to lock a slip owns it; everyone else skips it instead of waiting.

⏱️

The rack takes it back

A pharmacist wanders off holding a slip? After the visibility timeout, the order goes back on the rack.

🔁

Retry with a note

Failed orders go back with a “don’t retry before” time — backoff, implemented as nothing more than a timestamp.

Here is the entire life of a jobs row. The two teal arrows going backward are the whole reason this is a queue and not a todo list:

Locked claims, attempt-counted backoff, timeout-based reclaim: three small mechanisms that survive concurrent workers, dependency outages, and workers dying mid-job. Knowing them tells you exactly what to check when a job looks stuck — and it’s a policy you can copy anywhere your job volume is one operator’s background tasks.

Claiming: two hands, one slip

The whole multi-worker story is one SQL clause inside one transaction. Everything else in claimNext is stamping the claim.

CODE — v3/packages/db/src/queue.ts 
export async function claimNext(db: Database, input: ClaimNextInput): Promise<Job | undefined> {
  return db.transaction(async (tx) => {
    // Raw SQL so the locking clause — the load-bearing part — stays literal
    // and greppable rather than hidden behind builder options.
    const candidates = await tx.execute<{ id: string }>(
      sql`SELECT id FROM jobs
          WHERE status = 'queued' AND run_at <= now()
          ORDER BY created_at
          FOR UPDATE SKIP LOCKED
          LIMIT 1`,
    );

    const candidate = candidates.rows[0];
    if (!candidate) {
      return undefined;
    }

    const [row] = await tx
      .update(jobs)
      .set({
        status: "claimed",
        claimedBy: input.workerId,
        claimedAt: new Date(),
        attempts: sql`${jobs.attempts} + 1`,
        updatedAt: new Date(),
      })
      .where(sql`${jobs.id} = ${candidate.id}`)
      .returning();

    return row;
  });
}
PLAIN ENGLISH

One call = one attempt to take the next job. undefined means the rack is empty.

One transaction wraps both statements — the row lock taken by the SELECT lives exactly as long as this block.

The comment is doctrine: the load-bearing clause stays raw SQL so grep "SKIP LOCKED" finds it.

Take the oldest queued job whose run_at has arrived — the same timestamp backoff will push into the future.

FOR UPDATE locks the row; SKIP LOCKED makes a concurrent claimer skip it — no waiting, no double-grab.

One slip at a time.

Nothing claimable? Return undefined; the worker sleeps a tick and polls again.

Still inside the transaction — the lock is still held — flip the row to claimed.

Stamp who took it and when. These two fields power both the reclaim clock and the forensic freeze later.

attempts is bumped at claim time, not failure time — so it always counts the run that is about to happen.

Commit releases the lock, and the claimed row goes back to the worker loop.

💡
Aha: keep the load-bearing part literal

Everything else in this file uses the query builder — only the locking clause is raw SQL. When one clause carries the whole concurrency story, you want it greppable and word-for-word identical to the docs that explain it, not reconstructed from builder options in your head.

Failing: back on the rack, with a note

A failed job faces one comparison: attempts ≥ maxAttempts. Below the limit it goes back on the rack with a “don’t retry before” time; at the limit it’s frozen where it fell.

CODE — v3/packages/db/src/queue.ts 
export async function fail(db: Database, jobId: string, error: JobFailure): Promise<void> {
  await db.transaction(async (tx) => {
    const [row] = await tx.select().from(jobs).where(sql`${jobs.id} = ${jobId}`);
    if (!row) {
      return;
    }

    // attempts was already bumped by claimNext, so after the Nth run
    // attempts === N: 1st failure waits base*2^0, 2nd base*2^1, ... The
    // Math.max guards against a hand-inserted row failing at attempts = 0.
    const exhausted = row.attempts >= row.maxAttempts;
    const backoffMs = BACKOFF_BASE_MS * 2 ** Math.max(0, row.attempts - 1);

    // On terminal failure we freeze claimed_by/claimed_at/run_at as forensic
    // evidence of the final attempt; on requeue we clear the claim so the row
    // is claimable again once run_at arrives.
    await tx
      .update(jobs)
      .set({
        status: exhausted ? "failed" : "queued",
        lastError: error,
        runAt: exhausted ? row.runAt : new Date(Date.now() + backoffMs),
        claimedBy: exhausted ? row.claimedBy : null,
        claimedAt: exhausted ? row.claimedAt : null,
        updatedAt: new Date(),
      })
      .where(sql`${jobs.id} = ${jobId}`);
  });
}
PLAIN ENGLISH

The worker calls this when a handler throws. The error arriving here is already typed — that vocabulary is module 5.

Same shape as the claim: one transaction, load the row.

Job gone? Nothing to record.

The comment does the arithmetic for you: attempts already counts the run that just failed, because claimNext bumped it up front.

One comparison picks the road: terminal, or try again.

Doubling backoff — 1st failure waits base×1, 2nd base×2, 3rd base×4. Math.max guards a hand-inserted row failing at attempts = 0.

Two exits, one UPDATE — every field picks its value off the same exhausted flag.

failed is terminal; queued is back on the rack.

The typed failure lands on the row either way — the “note” on the slip.

The schedule column is the backoff mechanism: requeue pushes run_at into the future, and claimNext’s WHERE clause won’t touch it before then. No timer service — just a timestamp.

Requeue clears the claim so the row is grabbable again; terminal failure freezes claimed_by/claimed_at as forensic evidence — who ran it last, and when.

    try {
      // Visibility timeout: any job claimed longer ago than this is presumed
      // orphaned by a crashed worker and returned to the queue. Reclaim runs
      // every tick so recovery latency is bounded by pollMs, not by luck.
      const reclaimed = await reclaimStale(db, { visibilityTimeoutMs });
      if (reclaimed > 0) {
        log("jobs_reclaimed", { count: reclaimed });
      }

Every worker tick starts by rescuing other workers’ orphans, then polls for its own work. Recovery latency is bounded by the timeout plus one poll interval — by design, not by luck.

The trail beside the rack

Next to jobs sits the event trail from module 3, and it is append-only BY CONSTRUCTION: no trigger, no grant — recordEvent is the sole writer, and no UPDATE or DELETE path exists anywhere in the codebase.

CODE — v3/packages/db/src/events.ts 
 * The append-only guarantee is BY CONSTRUCTION, not by trigger or grant:
 * keeping this function the sole writer is the invariant. If you find
 * yourself wanting to update or delete an event row, the design answer is
 * to append a correcting event instead.
 */
export async function recordEvent(db: Database, input: RecordEventInput): Promise<void> {
  await db.insert(skeletonCheckEvents).values({
    runId: input.runId,
    seam: input.seam,
    ok: input.ok ?? true,
    detail: input.detail ?? {},
    durationMs: input.durationMs,
  });
}

status can go backward — queued → claimed → queued again. Retrying is its whole point.

📜

The trail: a ledger, no eraser

Events only accumulate. A retry shows up as more events — never edited ones. Corrections are new events.

After a retry, the job row looks clean — queued, as if nothing happened. The trail can’t pretend: the failed attempt’s events are still there, timestamped. The job may loop; the record of what happened only grows.

Auth & Typed Failure

One sealed cookie, one global hook, four error classes — and nothing else in the codebase knows a status code.

One hash, one seal, zero tables

There is no users table, no sessions table, and no auth middleware per route. One bcrypt hash in env, one sealed cookie, one global hook — and an unknown URL returns 401, not 404, on purpose.

The session isn’t a locker-room ticket the server has to look up — it’s a letter sealed with the server’s own signet ring. A broken seal needs no investigation: it simply reads as no letter. Revocation is changing the ring — rotate SESSION_SECRET and every outstanding seal dies at once.

1

OPERATOR_PASSWORD_HASH + SESSION_SECRET live in env — that’s the whole identity store

→
2

Login: bcrypt-compare the one password (D13)

→
3

Match → seal a cookie with the signet ring

→
4

Every later request: unseal. Clean seal = session. No DB row anywhere

The seal itself carries its own rules:

maxAge: 30 days The letter expires on its own — an old seal reads as no session
httpOnly Page JavaScript can never touch the cookie — the seal is opaque to scripts
sameSite: lax Other sites can’t ride along on the cookie for cross-origin writes
no session table Nothing to look up, nothing to leak, nothing to migrate — the cookie is the whole session

One hook guards every route — including the ones that don’t exist

Auth is deny-by-default: a single global onRequest hook with an explicit 4-route exemption list. No route opts in to auth — four routes opt out, by name.

CODE — v3/apps/api/src/auth/session.ts 
const EXEMPT = new Set(["GET /health", "GET /ready", "POST /api/auth/login", "POST /api/auth/logout"]);

  // A global onRequest hook (not preHandler) so it also applies to routes that
  // don't exist yet — "every other route requires a valid session" (contracts/api.md).
  // Consequence worth knowing: an unauthenticated request to an unknown path
  // gets 401, not 404 — the API reveals nothing about its route map.
  app.addHook("onRequest", async (request, reply) => {
    // routeOptions.url is the route PATTERN (e.g. "/api/x/:id"), so exemption
    // can't be spoofed by crafting a matching raw URL; raw url is only the
    // fallback for unmatched (404-bound) requests, which are never exempt.
    const key = `${request.method} ${request.routeOptions?.url ?? request.url}`;
    if (EXEMPT.has(key)) {
      return;
    }

    if (!isSessionAuthenticated(request)) {
      // Same fixed "Sign-in failed." body as every credential failure — the
      // response never distinguishes missing vs. tampered vs. expired session.
      reply.code(401).send(errorEnvelope("unauthorized", "Sign-in failed."));
    }
  });
PLAIN ENGLISH

The full escape hatch: four literal METHOD pattern strings — health, ready, login, logout. Everything else lives behind auth.

onRequest, not preHandler: it fires before routing resolves a handler, so even paths with no route are guarded.

That’s the deliberate consequence — unknown path, no session, 401. The route map is invisible to the unauthenticated.

Registered once, app-wide. There is no per-route guard to forget.

The key uses the route pattern (/api/x/:id), never the raw URL — you can’t craft a URL that string-matches an exemption.

Raw URL only fills in for unmatched, 404-bound requests — and those are never in the set anyway.

Exempt key? Fall through to the route, no session needed.

Everyone else must present a cookie that unseals cleanly.

One fixed body — "Sign-in failed." — whether the seal is missing, tampered, or expired. The response teaches an attacker nothing.

💡
Aha: forgetting is impossible

Deny-by-default means forgetting auth on a new route is impossible — the failure mode of forgetfulness is 401, not exposure. The dangerous act isn’t adding a route; it’s editing EXEMPT, and that edit is loud, explicit, and one line.

Four triage tags, not incident novels

Every failure in this system wears exactly one of four tags. Only the ER front desk — the API edge — translates a tag into what the public hears: a status code.

404

unknown_thing

You asked for something that doesn’t exist — a run id, a resource.

Operator’s move: no such thing — check the id, not the system.

415

unsupported_type

The input is a kind this system doesn’t serve.

Operator’s move: change the input — retrying the same thing changes nothing.

503

dependency_down

Something we depend on is unreachable or not ready yet.

Operator’s move: retry later — it heals without you.

500

internal_fault

Our own bug or misconfiguration — the fault is on this side of the seam.

Operator’s move: go fix config or code — waiting is useless.

401

unauthorized

The auth-only bolt-on — not a domain class, just the edge’s own tag for a broken seal.

Operator’s move: sign in.

The entire tag-to-status translation is one tiny table — and that smallness is the point. Each mapping is pinned by FR-018, so it can’t drift silently.

CODE — v3/apps/api/src/errors.ts 
const STATUS_BY_CLASS: Record<ErrorClass | "unauthorized", number> = {
  unknown_thing: 404,
  unsupported_type: 415,
  dependency_down: 503,
  internal_fault: 500,
  unauthorized: 401,
};

export function statusForErrorClass(errorClass: ErrorClass | "unauthorized"): number {
  return STATUS_BY_CLASS[errorClass];
}

export function errorEnvelope(errorClass: ErrorClass | "unauthorized", message: string) {
  return { error: { code: errorClass, message } };
}
PLAIN ENGLISH

The only place in the codebase that knows an HTTP status. Four domain classes plus the auth bolt-on.

No such thing → 404.

Wrong kind of input → 415.

Dependency unreachable → 503 — a retryable answer.

Our bug → 500.

Broken seal → 401. Note the type: unauthorized isn’t an ErrorClass, it’s bolted on at the edge only.

End of doctrine.

Tag in, status out — a pure lookup, zero logic.

And every failure body has the same shape: { error: { code, message } }. Clients parse one envelope, forever.

The tag is chosen where the patient is found

Classification happens at the seam that failed — worker, ml client, routes all throw DomainError — and only the edge translates. Watch the ml client split two failures that look similar on the wire but demand opposite operator actions.

CODE — v3/apps/worker/src/ml-client.ts 
  // 503 is the sidecar's documented "model still loading / not ready" answer
  // (contracts/ml-sidecar.md) — reachable but not serving, still a down
  // dependency from the run's point of view, and retryable.
  if (response.status === 503) {
    throw new DomainError({
      class: "dependency_down",
      seam: "inference",
      message: "Inference sidecar is not ready.",
    });
  }

  // Any other non-2xx (400 unknown model, 404, ...) means we sent something
  // the contract doesn't allow — a misconfiguration on our side, so it must
  // NOT masquerade as a down dependency: the operator should go fix config,
  // not wait for the sidecar to "come back".
  if (!response.ok) {
    throw new DomainError({
      class: "internal_fault",
      seam: "inference",
      message: `Inference sidecar returned ${response.status}.`,
    });
  }
PLAIN ENGLISH

503 is the sidecar’s documented “still loading” answer — reachable, just not serving yet.

From the run’s point of view that’s still a down dependency, and crucially: a temporary one.

So the tag is dependency_down — retryable. The operator’s move is wait.

The error notes which seam failed and why, in one line.

Same for connection timeouts and refused connections, handled just above this block.

Any other non-2xx — 400 unknown model, 404 — means we sent something the contract doesn’t allow.

That must not masquerade as a down dependency: the sidecar is up and answering; our config is wrong.

So the tag is internal_fault. The operator’s move is go fix config — waiting for the sidecar to “come back” would wait forever.

The class is the operator’s next action, encoded.

Silence is the symptom

Bugs #1 and #4 share a shape: nothing screamed. Stopping Postgres didn’t produce an error page — it produced a dead process, because node-postgres raises an idle-client error that nobody was listening for.

CODE — v3/packages/db/src/client.ts 
  const pool = new Pool({ connectionString });
  // node-postgres crashes the process on an unhandled idle-client error
  // otherwise; a dead DB must surface as dependency_down, not a process exit.
  pool.on("error", (err) => {
    console.error("Postgres pool error:", err);
  });
PLAIN ENGLISH

Create the shared pool every query in the process flows through.

The comment is the scar: stopping Postgres once killed the entire API, because an idle pooled connection emitted 'error' with no listener — and Node treats that as fatal.

Registering the listener is the whole fix. There’s nothing to recover here — the pool discards the dead connection on its own.

Log it and stay alive. A dead DB now surfaces as 503 dependency_down from the health check, not a process exit — and a regression test stops Postgres to prove it.

Bug #4 was quieter still. The worker’s compose env was missing EMBEDDING_PROVIDER, so resolveModelRole threw a plain Error before the try/catch that writes failure outcomes — the job silently retried while the run sat at running forever.

CODE — v3/apps/worker/src/handlers/skeleton-check.ts 
  try {
    await runSkeletonCheck(db, runId);
  } catch (error) {
    // Anything NOT already handled below (e.g. a misconfigured env var
    // resolving the model role) must still fail the run — otherwise it's
    // stuck at "running" forever while only the underlying job retries.
    // This is a scar, not speculation: a missing env var once produced exactly
    // that silent hang. DomainErrors skip this — runSkeletonCheck already
    // called failRun for those; double-failing would clobber the real outcome.
    if (!(error instanceof DomainError)) {
      await failRun(
        db,
        runId,
        new DomainError({
          class: "internal_fault",
          seam: "inference",
          message: "Unexpected error before inference.",
          cause: error,
        }),
      );
    }
    // Re-throw so the poll loop also fails the JOB (retry/backoff lives there).
    throw error;
PLAIN ENGLISH

Wrap the entire check in one boundary try — including everything that runs before the handler’s own error handling exists.

The comments narrate the scar: the missing env var threw here, the job retried, and the run showed running forever. Absence of failure, the worst kind of bug.

DomainErrors skip this branch — runSkeletonCheck already called failRun with the real typed outcome, and double-failing would clobber it.

Anything else gets failed explicitly, wrapped as a typed internal_fault pinned to the seam it never reached — original error preserved as cause.

Then re-throw: the poll loop fails the JOB (retry/backoff lives there), this block fails the RUN. Two ledgers, both written.

Aha: the worst failure mode isn’t a crash

It’s a system that keeps looking busy. A crashed process gets noticed; a run stuck at running gets a shrug and a refresh. Both fixes here do the same thing: convert silence into a typed, visible failure.
