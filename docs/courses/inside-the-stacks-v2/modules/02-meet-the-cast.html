Meet the Cast

Five specialists, one call sheet — who actually runs The Stacks

The Call Sheet

In Module 1 you watched the relay race. Time for proper introductions — because everything you saw was performed by five separate programs, each sealed inside its own container.

One command wakes them all up, in the right order, ready to cooperate:

TERMINAL 
docker compose up
PLAIN ENGLISH

"Everyone on the call sheet — report to set, in dependency order, and shout when you're ready."

🎬
The metaphor for this module: a film set on shoot day

The file docker-compose.yml is the call sheet: it lists who's on the crew, when each person shows up, how you check they're ready ("ready on set!"), and which rooms they work in. Docker is the studio that runs the whole operation, and Docker Compose is the stage manager reading from it.

The Cast, In Order of Appearance

Five crew members, one job each. Nobody does two jobs — that's the whole trick.

🖥️

Web app

The storefront — the pages you actually see and click. Built with React.

🚪

API

The front desk — checks IDs, takes requests, orchestrates answers. Built with FastAPI in Python.

🛠️

Worker

The back room — processes your uploads while you do other things. A Python loop that never stops checking for jobs.

🗃️

Postgres

The ledger — a database that remembers everything: messages, uploads, jobs, citations.

🧭

Qdrant

The meaning-finder — a special vector database that searches by similarity, not by exact words.

🤖

OpenAI (off-set)

The outside contractor — turns text into meaning-numbers and drafts answers. Not on the crew, not in a container: it's an internet service the API calls.

💡
Why should you care who's who?

Knowing the cast lets you steer AI precisely ("put that logic in the Worker, not the API") and localize failures (uploads stall → suspect the Worker; login fails → suspect the API). Engineers call this one-job-per-specialist design separation of concerns.

Reading the Call Sheet for Real

Here's Postgres's actual entry in docker-compose.yml — written in YAML. Every line answers a stage-manager question: who, where, and are-you-ready.

docker-compose.yml 
  postgres:
    image: postgres:16-alpine
    environment:
      POSTGRES_DB: thestacks
      POSTGRES_USER: thestacks
      POSTGRES_PASSWORD: thestacks
    volumes:
      - postgres-data:/var/lib/postgresql/data
    ports:
      - "127.0.0.1:5433:5432"
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U thestacks -d thestacks"]
      interval: 5s
      timeout: 5s
      retries: 20
PLAIN ENGLISH

Cast member's name on the call sheet: "postgres". Everyone reaches this crew member by this exact name.

Which program to run — Postgres version 16, the lightweight edition.

Its personal settings: database name, username, and password (fine for your own machine, changed in production).

A volume: a keep-forever drawer, so your data survives even if the container is torn down.

A port opened as a window to your machine, so you can peek at the database directly.

The healthcheck: every 5 seconds Docker asks "ready on set?" — and nobody who depends on Postgres starts until it says yes, up to 20 tries.

And here's how the API learns its own settings, in apps/api/app/config.py. Every knob reads from an environment variable, with a safe default for your laptop.

Should login cookies require a secure connection? Off by default for your laptop — forced ON in production.

How long a login lasts: 7 days × 24 hours × 60 minutes × 60 seconds.

Where's the database? Look at the address inside…

…it dials @postgres:5432 — the call-sheet name, like an internal phone extension. No IP address needed.

But any of these can be overridden by the environment variable named here.

(End of the database setting.)

Which website addresses may talk to this API — a CORS guest list, defaulting to your local dev address.

📞
Two addresses for the same API — here's why

Inside Docker's private network, services reach each other by name: the API literally dials postgres:5432. But your browser lives OUTSIDE that network, so it reaches the API via localhost — same building, but you have to use the street entrance while the crew uses internal extensions.

The Bouncer at Every Door

One last cast trick before the quiz: how does the Web app guard its pages? Every protected page runs this tiny gatekeeper first, from apps/web/app/lib/auth.ts.

apps/web/app/lib/auth.ts 
export async function requireAuth() {
  try {
    await getAuthStatus();
  } catch (error) {
    if (isUnauthorized(error) || isApiNetworkError(error)) {
      throw redirect("/login");
    }
    throw error;
  }
}
PLAIN ENGLISH

The door guard, runnable by any protected page.

Attempt the check…

…ask the API: "am I logged in?"

If that goes wrong, look at WHY it went wrong:

If the API said 401 ("who are you?") — or the API can't be reached at all —

…bounce the visitor straight to the login page.

(Done with the bounce case.)

Any other error is a real problem — pass it along, don't hide it.

(End of the check.)

That's the whole bouncer.

🎟️
Aha: your browser never holds your password

After login, the browser holds only a signed cookie — a stamped wristband the bouncer checks at every door. It's HTTP-only, so even sneaky code running on the page can't read it.
