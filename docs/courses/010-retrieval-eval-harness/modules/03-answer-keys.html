Answer Keys That Survive

The gold set references passages by what they say, not where they live — labels that heal themselves, flag themselves, and never silently lie.

An Answer Key Written Against Content, Not Page Numbers

You labeled 40 questions last month. Today you re-ingested three sources — how many of those labels are now lies? Exactly the flagged ones, and zero of them will silently score as misses.

A gold item is a teacher's answer key written against the content of the textbook, not its page numbers. New edition, same paragraph — the key still works. Paragraph rewritten — the key flags itself for re-grading instead of silently marking students wrong.

Resolution Happens Server-Side

The UI never lets you type a passage reference by hand: you run a real search on /search, and each result offers "label as expected". The client sends bare chunk ids; resolveExpectedPassages in packages/retrieval/src/gold.ts turns them into durable references — and refuses anything a reader could not actually retrieve.

📏
The labeling standard is on the page (FR-012)

The /evals/gold screen shows it at authoring time, verbatim: "an expected passage is one whose text ALONE answers the question — not one that merely mentions its topic. Label from real search results (each result offers 'label as expected')." The standard travels with the form, not with tribal memory.

packages/retrieval/src/gold.ts:42–70 

/** Resolve bare chunk ids into durable references. Refuses chunks that are
 *  not current-generation: labeling against text a reader can't retrieve
 *  would be a lie at birth. */
export async function resolveExpectedPassages(
  db: Database,
  chunkIds: string[],
): Promise<ExpectedPassage[]> {
  if (chunkIds.length === 0) {
    throw new DomainError({ class: "invalid_input", message: "expected must name at least one chunk." });
  }
  const rows = await db.execute<{ id: string; source_id: string; hash: string }>(sql`
    SELECT c.id, c.source_id,
           encode(sha256(convert_to(c.content, 'UTF8')), 'hex') AS hash
    FROM chunks c JOIN sources s ON s.id = c.source_id
    WHERE c.id IN (${sql.join(chunkIds.map((id) => sql`${id}`), sql`, `)})
      AND c.generation = s.current_generation
  `);
  const byId = new Map(rows.rows.map((r) => [r.id, r]));
  return chunkIds.map((chunkId) => {
    const row = byId.get(chunkId);
    if (!row) {
      throw new DomainError({
        class: "invalid_input",
        message: `Chunk ${chunkId} is not a current-generation passage — expected passages must be retrievable as labeled.`,
      });
    }
    return { chunkId, sourceId: row.source_id, contentSha256: row.hash };
  });
}

The contract: bare chunk ids in, durable references out — and some ids get refused.

Refused because they are not current-generation: text no reader can retrieve today…

…would make the label a lie at birth. The refusal doctrine lives in the doc comment.

Both create and re-label funnel every reference through this one function.

It takes a database handle…

…and the bare chunk ids the operator picked from live search results.

Out come durable ExpectedPassage triples.

A label with zero expected passages is meaningless…

…so refuse it as the caller's mistake before touching the database.

End of the empty-input guard.

One query resolves every id at once, typed on the three columns coming back.

Fetch each chunk's id and which source it belongs to…

…and compute sha256 of its text IN Postgres — the same hash expression every other consumer uses.

Join to the source table to learn which generation is current.

Keep only the chunks the operator actually named…

…and only if they sit in the source's current generation. Stale chunks simply never come back.

End of the SQL.

Index the survivors by id, ready for the reconciliation pass.

Walk the REQUESTED ids — not the returned rows — so nothing is skipped quietly…

…looking each one up among the survivors.

Any id the query dropped was stale or bogus…

…and that is the refusal: a typed DomainError, not a warning…

…classed as the caller's mistake (HTTP mapping happens only at the API edge)…

…naming the exact chunk and the rule it broke: expected passages must be retrievable as labeled.

Close the error payload.

End of the missing-row branch.

A good id becomes the durable triple: chunk id, source id, content hash.

Close the reconciliation loop.

Done — every reference in the label is now anchored to content, not position.

The Derivation Twin

needsReconfirmation is never stored — there is no boolean column to go stale. Every time gold items are listed, one SQL fragment re-derives the flag from the corpus as it is right now.

packages/retrieval/src/gold.ts:81–92 

const RECONFIRM_SQL = sql`
  EXISTS (
    SELECT 1 FROM jsonb_array_elements(${goldItems.expected}) AS exp(entry)
    WHERE NOT EXISTS (
      SELECT 1 FROM chunks c
      JOIN sources s ON s.id = c.source_id
      WHERE c.source_id = (exp.entry->>'sourceId')::uuid
        AND c.generation = s.current_generation
        AND encode(sha256(convert_to(c.content, 'UTF8')), 'hex') = exp.entry->>'contentSha256'
    )
  )
`;
PLAIN ENGLISH

A SQL fragment, not a column — the flag is computed fresh on every read.

The item flags itself if ANY of its expected passages is orphaned…

…unpacking the stored expected JSON array into one row per reference…

…and asking, per reference: is there NO living chunk that still carries this text?

Scan the chunks…

…joined to their sources, to know which generation is current…

…restricted to the source the label pointed at…

…counting only current-generation chunks — the same rule resolution enforced at label time…

…matched by the same sha256-of-content expression the reference was minted with.

Close the inner probe: no match means the text is gone from the live corpus.

Close the outer check: one orphaned reference flags the whole item.

End of the fragment — listGoldItems selects it as the derived needsReconfirmation column.

🪞
Same hash trick, opposite consumer

Module 2's receipts derive superseded with a NOT EXISTS over current-generation hashes — the record admits the past has been overtaken. RECONFIRM_SQL runs the identical probe for gold items — the label admits the future needs re-checking. One hash expression, two directions of honesty, zero stored booleans to drift.

The Split Is Not Negotiable

Every gold item lands in one of two piles at creation, and the held-out split is the pile you are not allowed to study from. Question, passages, notes — all freely editable forever. The split is not.

🔧

Tuning (~75%)

Measure freely. Compare configurations, iterate, chase these numbers — they exist to guide choices.

🔒

Held-out (~25%)

Excluded from every choose-between-configs metric. Spent only to validate the final choice — the exam you never studied from.

🎲
Every 4th item — zero randomness to argue about

defaultSplit counts the corpus's existing items and assigns (n + 1) % 4 === 0 → "heldout": a deterministic ~25% holdout with no seed, no shuffle, no debate. The operator may override the default at creation — and never again after.
