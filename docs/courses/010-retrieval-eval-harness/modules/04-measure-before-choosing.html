Measure Before Choosing

RRF or weighted fusion? α 0.5 or 0.7? Rerank on? The constitution’s answer: you don’t get an opinion — you get a measurement.

A Dyno for the Search Engine

Module 1 left the engine’s knobs hanging: which fusion strategy, which parameters, whether the reranker earns its latency. You don’t ship an engine mod because it feels faster — you strap the engine to the dyno, change one thing at a time, and the printout goes in the file. That dyno pull is an eval run.

And the CI floor is the dyno bolted into the build: every PR re-proves the baseline before it merges. Here is one pull, end to end — the fixture set’s 12 questions, from POST to completed:

Pin History Before You Measure

A dyno printout is worthless if someone can re-tune the reference engine afterward. So createEvalRun does two things in one breath: copy the answer key onto the run row — the gold snapshot — and mark the run running:

CODE — packages/retrieval/src/eval/run-eval.ts:42-70 
export async function createEvalRun(db: Database, input: CreateEvalRunInput): Promise<string> {
  const items = await db
    .select({
      id: goldItems.id,
      question: goldItems.question,
      expected: goldItems.expected,
      split: goldItems.split,
    })
    .from(goldItems)
    .where(sql`${goldItems.corpusId} = ${input.corpusId}`)
    .orderBy(sql`${goldItems.createdAt} ASC`);
  if (items.length === 0) {
    throw new DomainError({
      class: "invalid_input",
      message: "The gold set is empty — label at least one item before running an eval.",
    });
  }
  const [row] = await db
    .insert(evalRuns)
    .values({
      corpusId: input.corpusId,
      config: input.config,
      configName: input.config.configName,
      goldSnapshot: items,
      status: "running",
    })
    .returning({ id: evalRuns.id });
  return row!.id;
}

Metrics With a Contract

Two dyno pulls are only comparable if the gauges never get recalibrated between them. The harness reports recall@5 / MRR / nDCG@10 per slice, and their meanings are fixed:

🎯

recall@5 / @10

Did any expected passage surface in the top k? The fraction of items answering yes.

🥇

MRR

How high was the first hit? Mean of 1/rank — rank 1 scores 1.0, rank 3 scores 0.333.

📐

nDCG@10

Are all expected passages ordered well? High placements earn more, normalized against the ideal ordering.

The implementation is pure arithmetic over hash lists — and it opens with its own rules of engagement:

CODE — packages/retrieval/src/eval/metrics.ts:1-12, 51-73 (interfaces elided) 
/**
 * Pinned metric definitions (spec 010 US4) — the ONLY metric semantics the
 * harness may implement, transcribed from contracts/metrics.md. Pure
 * arithmetic over hash lists: no DB, no engine — the deterministic CI floor
 * (T031) and every eval report stand on these functions computing exactly
 * what the contract says, forever.
 *
 * Matching is by content hash (a re-ingested identical passage still
 * counts); `unresolvable` items are EXCLUDED from every denominator and
 * surfaced as their own count — a stale answer key is a labeling problem to
 * fix, never a silent miss to absorb.
 */
// … input/output interfaces and NDCG_DEPTH (10) elided …
function outcomeOf(item: EvalItemInput): EvalItemOutcome {
  if (item.unresolvable) {
    return {
      goldItemId: item.goldItemId,
      split: item.split,
      status: "unresolvable",
      firstHitRank: null,
      hitRanks: [],
    };
  }
  const expected = new Set(item.expectedHashes);
  const hitRanks: number[] = [];
  item.resultHashes.forEach((hash, index) => {
    if (expected.has(hash)) hitRanks.push(index + 1);
  });
  return {
    goldItemId: item.goldItemId,
    split: item.split,
    status: hitRanks.length > 0 ? "hit" : "miss",
    firstHitRank: hitRanks[0] ?? null,
    hitRanks,
  };
}

The Report: A Decision You Can Re-Run

Completed runs compose into docs/eval-reports/010-retrieval-baseline.md. This is its first edition — 2026-07-11, the fixture baseline, the report that justified the shipped defaults. Three configs, one variable at a time, every row a live receipt:

Config Eval run id tuning r@5 tuning MRR tuning nDCG@10 heldout

weighted-a07 (α=0.7) b0079b04… 1.000 0.833 −0.111 0.834 1.000

Read it like the report does: RRF and weighted-α0.5 tie on every metric. Push α to 0.7 and tuning MRR drops 0.111 — the weighted family is sensitive to a calibration knob with no principled per-corpus value. A tie under measurement resolves to the option with no knob: RRF ships.

🚫
Rerank ships OFF — because nothing justifies ON

RETRIEVAL_RERANK=off cites the absence of measurement: SC-006’s on/off comparison needs a live cross-encoder role, and none was configured. Turning it on before measuring would invert D11. Not-yet-measured is a first-class, documented state — the report says so in writing.

First edition, deliberately. These numbers prove the machinery on constructed vector geometry — ranking, fusion, metric math — not semantic quality. The same dyno re-ran later on a real corpus with real embeddings and a 41-item gold set: it re-confirmed RRF, and overturned something else entirely. That story is Module 6’s.

Prove You Can Read the Dyno

A teammate reviewing the report table asks: “tuning MRR 0.944 — how would I audit what the harness actually searched to get that number?” What makes that possible?

The morning after run 6e05e95f… completed, you rewrite a badly-phrased gold question. What happens to that run’s recorded metrics?

A PR refactors fusion “for clarity” and quietly degrades ranking — held-out MRR would now measure 0.667. What happens when CI runs pnpm verify?

One stage of the pipeline has been conspicuously quiet: the reranker — a cross-encoder specialist that re-orders the top candidates, shipped off, its adoption blocked on a measurement that doesn’t exist yet. Module 5 is its audition: the stage that must earn its place on this dyno.
