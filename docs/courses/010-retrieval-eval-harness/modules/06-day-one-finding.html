Two Mechanisms, One Miss

Both signals from Module 1 failed the same query at once — for two completely different reasons. FTS died on AND semantics; the vector died on the floor.

🔤

Mechanism 1 — FTS AND

websearch_to_tsquery turns "how does a riposte work" into riposte & work. The passage has "riposte" but not "work" — the FTS signal returns nothing. The bare keyword still hits; the natural question cannot.

📉

Mechanism 2 — the floor

The riposte rule is one sentence inside a multi-topic page, so the question-vs-whole-page cosine sits low — under the 0.3 similarity floor. The vector signal, the one that should rescue natural questions, dropped it too.

Here is the uncomfortable part: mechanism 1 was never a surprise. The fixture corpus from Module 4 recorded it, at fixture level, before any live query existed:

What the System Did With Its Own Failure

Nobody patched anything that afternoon. Instead, the failure was filed — through every layer this course has built, in order:

1
The receipt (Module 2)

The missed search left a retrieval run like any other — query, config, stage timings, zero results. The failure is a first-class record, not a shrug.

2
The eval run (Module 4)

A one-item eval run against the live gold item "how does a riposte work" scored recall 0 — honestly. No fudging, no re-run until green.

3
The eval report

Both mechanisms went into the report's known characteristics: FTS AND semantics dropped "work"; the question-vs-page cosine fell under the 0.3 floor.

4
The board task

TASK-10 opened with the finding as its first question: is 0.3 too high for natural-question phrasing against real embeddings? With instructions to gather real-corpus data before answering.

🔔
"The harness earning its keep" — the evidence file's own words

The spec's evidence ledger records the zero under that verbatim heading, and closes with the doctrine: changing the default without measurement "would invert D11 — the harness exists precisely to catch this." A retrieval stack that can prove it failed is one you can improve deliberately. One that cannot is one you tune by vibes.

Why the Fix Did Not Ship That Afternoon

The tempting patch was obvious within minutes: the cosine fell under 0.3, so lower RETRIEVAL_MIN_SIMILARITY. One env var. The whole apparatus exists to say not yet — measure before choosing.

🩹

The vibes path

Lower the floor to whatever makes today's query pass. No idea what it does to every other query, no receipt, no way to argue about the next tweak either.

📐

The D11 path

Build a gold set worth trusting over a real corpus, sweep the candidate values, read the metrics, and move the default only where the measurement points — with run ids attached.

⚖️
What the day-one evidence could actually justify: nothing

The zero came from a 1-item gold set on a 1-chunk corpus. It proves the miss is real; it cannot tell you whether 0.2 fixes it, breaks something else, or whether the floor should exist at all. TASK-10's job was to gather the data that can — and only then touch the default.

TASK-10 Closes the Loop

TASK-10 built the measurement the afternoon patch lacked: a 41-item gold set (31 tuning / 10 heldout, split immutable per Module 3) over the operator's real corpus — a saved DDB "Monsters (G)" page (36 chunks, ingested by ddb-saved-html) plus the Emberfall homebrew page (1 multi-topic chunk) — real MiniLM embeddings, real engine, every item leaving an eval-origin receipt. Then it swept the floor: 0.3 / 0.2 / 0.15 / 0.0.

0.15 ties 0.2 on every metric, so 0.2 wins as the knee with the most headroom against honest-empty. And the report measured the day-one mechanisms directly, cosine by cosine:

The riposte rule is one sentence inside a chunk dominated by grappling and torchlight. Query-vs-whole-chunk cosine for riposte phrasings: riposte 0.161, what is a riposte 0.167, how does a riposte work 0.209 — all below the 0.3 floor. (By contrast how does grappling work scores 0.542: grappling is the chunk.)

FTS can't rescue the natural question either: websearch_to_tsquery's AND-semantics turn "how does a riposte work" into riposte & work; the chunk has "riposte" but not "work", so the FTS signal returns nothing. The bare keyword riposte still hits via FTS — which is why the keyword gold item passes and only the natural-question one failed.

The why-comment IS the receipt pointer: which task, which report, which date.

What 0.3 broke: natural questions whose answer is one sentence in a multi-topic chunk…

…with the measured cosine right in the margin — about 0.21, under the old floor.

What 0.2 fixed — recall back to 1.0 — and why not zero: no floor REGRESSED the rank metric…

…because weak matches outrank true answers. The knee, not the extremes.

The default itself: per-run overrides first, env next (D14), and the fallback is 0.2 — the measured value.

Real data paid a second dividend nobody ordered — two engine bugs the synthetic fixtures structurally never could have caught:

🕵️

Detect vs. real saved pages

A browser's "Save Page As" inlines every stylesheet into <head>, pushing <body> past ddb-saved-html's 64 KiB detect prefix — the real page scored 0 and fell through as unsupported. Fixed to detect on prefix-stable identity signals; pinned by regression tests.

♻️

The job that never finished

The worker handled jobs but never called complete() — every job sat claimed and was re-run each visibility timeout, forever. Idempotent handlers hid it until real data showed a job at 38 attempts (max 3). Measured validation pays twice.

🧊
The CI floor did NOT move

The deterministic per-PR slice keeps its own floor pinned at 0.3: its hash-constructed embeddings place unrelated text at ~0.2–0.3, where real MiniLM puts it near 0, so its honest-empty guarantee is calibrated to 0.3. The operational default and the fixture floor are decoupled on purpose — a different number for a different embedding space.
