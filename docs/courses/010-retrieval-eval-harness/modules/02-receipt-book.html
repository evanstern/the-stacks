The Receipt Book

Every search writes a carbon-copy receipt nobody can tear out or edit — and "this item was discontinued" is stamped on when you look, never written into the book.

A Page Nobody Can Tear Out

Open a months-old search — say run 8571faef… from this cycle's live validation. The source was re-ingested twice since. The receipt still renders every passage it returned — and truthfully marks which ones no longer exist.

The mechanism is a carbon-copy receipt book. Every sale writes through to a page that can't be edited afterward; when the shop renovates, old receipts still read perfectly. Module 1 closed with "every search you just saw was recorded." Here's that recording happening:

The Only Writer, and It Only INSERTs

Immutability here isn't a code-review policy — it's append-only by construction. One module owns all writes to the receipt tables, and the only verb it contains is INSERT. Read the header first: the doctrine is written where the code lives.

CODE — packages/db/src/retrieval-runs.ts:1-94 (interfaces & score fields elided) 
/**
 * retrieval_runs / retrieval_results — append-only BY CONSTRUCTION
 * (spec 010, Principle III: citations are receipts).
 *
 * This module is the ONLY writer, and it only ever INSERTs — one transaction
 * per receipt, header and lines together, so a torn receipt is
 * unrepresentable. There is deliberately no update or delete helper here and
 * none may be added: "superseded" is derived at view time, corrections are
 * new runs, and history never changes shape. Same construction — and same
 * warning — as recordEvent in ./events.ts.
 */
// … imports and the typed input/output interfaces …
export async function recordRetrievalRun(
  db: Database,
  input: RecordRetrievalRunInput,
): Promise<RecordedRetrievalRun> {
  return db.transaction(async (tx) => {
    const [run] = await tx
      .insert(retrievalRuns)
      .values({
        query: input.query,
        config: input.config,
        // … corpusId, origin, the embedding stamp, stageTimings …
        // Derived HERE from the lines actually inserted below — a caller
        // cannot make the denormalization disagree with the receipt.
        resultCount: input.results.length,
      })
      .returning({ id: retrievalRuns.id, createdAt: retrievalRuns.createdAt });
    if (input.results.length > 0) {
      await tx.insert(retrievalResults).values(
        input.results.map((r) => ({
          runId: run!.id,
          rank: r.rank,
          chunkId: r.chunkId,
          sourceId: r.sourceId,
          generation: r.generation,
          contentSnapshot: r.contentSnapshot,
          anchorSnapshot: r.anchorSnapshot,
          sectionIds: r.sectionIds,
          contentSha256: r.contentSha256,
          // … ftsScore, vectorScore, fusedScore, rerankScore, prerankPosition …
        })),
      );
    }
    return { id: run!.id, resultCount: input.results.length, createdAt: run!.createdAt };
  });
}

Snapshots: The Receipt Outlives the Shelf

When the shop renovates — a generation sweep deletes every chunk of the old generation — what keeps an old receipt readable? Each result line carries a snapshot: the passage itself, copied onto the row at retrieval time.

🔗

The pointer

chunk_id — durable identity, deliberately NOT a foreign key. It's allowed to dangle after a sweep; the receipt keeps rendering anyway.

📄

The copy

content_snapshot + anchor_snapshot + section_ids — what the operator actually saw, self-sufficient forever.

🔏

The fingerprint

content_sha256 — lets "is this text still live?" be answered without ever comparing full passages.

"Superseded" Is a Question, Not a Column

If old passages can vanish, receipts must say so — but a stored flag would need an UPDATE, and this design makes UPDATEs unrepresentable. So superseded is derived fresh, every time someone views a run. Three things can have happened to a passage since its receipt was written:

1
Source untouched

The hash still exists at the current generation → no mark. The receipt reads clean.

2
Re-ingested, text identical

New chunk id, same bytes, same hash → still found at the current generation → no mark. That's auto-heal.

3
Re-ingested, text rewritten

The hash is gone from the current generation → the viewer stamps superseded on that line. Nothing was written to say so.

All three outcomes fall out of one NOT EXISTS subquery in the run-view route:

CODE — apps/api/src/retrieval/routes.ts:185-216 (row-shape interface elided) 
      // The receipt renders from its own snapshots; "superseded" is DERIVED
      // here and only here (data-model.md): no chunk with this content hash
      // exists at the source's CURRENT generation. An identical re-ingest
      // keeps the hash alive under a new chunk id — not superseded.
      // … the ReceiptRow row-shape interface …
      const resultRows = await db.execute<ReceiptRow>(sql`
        SELECT r.*,
               NOT EXISTS (
                 SELECT 1 FROM chunks c
                 JOIN sources s ON s.id = c.source_id
                 WHERE c.source_id = r.source_id
                   AND c.generation = s.current_generation
                   AND encode(sha256(convert_to(c.content, 'UTF8')), 'hex') = r.content_sha256
               ) AS superseded
        FROM retrieval_results r
        WHERE r.run_id = ${request.params.id}
        ORDER BY r.rank
      `);

🩹
Auto-heal: repair without a write

Re-ingest a source with identical text and every old receipt quietly reads clean again — the hash survives under a new chunk id, so the derivation finds it. A stored flag would have needed a second UPDATE to un-mark those rows. Deriving the answer means there was never a mark to remove.

Audit the Receipt Book

Why is superseded derived by the NOT EXISTS query instead of stored as a column on retrieval_results?

The API process is killed right after the retrieval_runs header INSERT but before the retrieval_results lines INSERT. What does the database hold afterward?

A source is re-ingested and every passage comes back byte-identical (new chunk ids, same text). You open a receipt from before the sweep. What does it show?

Hold on to that last answer. A content hash that survives re-ingestion is a reference more durable than any row id — and the receipt book isn't the only place this repo needs one. Module 3 uses the exact same hash trick to build answer keys that survive: gold labels that point at passages by content_sha256, so re-ingesting your library never invalidates your test set.
