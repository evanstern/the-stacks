Two Signals, One Answer

Search “grapple” and the exact-match librarian wins. Search “holding an enemy in place” — no shared words at all — and the passage still surfaces. Same engine, one ranking. How?

Two librarians, one desk

The Stacks answers every search with hybrid retrieval: two librarians read the same corpus and never talk to each other. The literalist matches exact words — Postgres full-text search. The associative matches meaning — pgvector embeddings. Neither is trusted alone; a rank-based vote at the fusion desk merges their ballots into one answer.

📖

The literalist (FTS)

Finds chunks whose words match the query. Wins “grapple” instantly. Handed “holding an enemy in place”, it shrugs — no shared terms, no matches.

🧭

The associative (vectors)

Finds chunks whose meaning sits near the query in embedding space. The paraphrase surfaces the grappling passage with zero words in common.

What each librarian actually runs

Both librarians run one SQL query each, against the same chunks table, and both carry the same guard: the reader predicate. The literalist’s side in two strokes:

websearch_to_tsquery Accepts raw human input safely (research R3) — and every term must match (AND semantics). A query about nothing in the corpus matches nothing.
ts_rank_cd Scores matches by how close together the terms sit — proximity, which is what relevance looks like in passage-sized chunks.

The associative’s query is the more interesting one — it carries the reader predicate and the similarity floor, and its comment records a deliberate decision:

CODE — packages/retrieval/src/search.ts:147–163 

    const vectorLiteral = `[${queryStamp.vector.join(",")}]`;
    // Exact scan by design (research R2): no ANN index below ~100k chunks,
    // so recall measurements reflect ranking math, not index approximation.
    vectorCandidateRows = await timed("vector", async () => {
      const res = await db.execute<CandidateRow>(sql`
        SELECT c.id, c.source_id, c.generation, c.content, c.anchor,
               c.section_ids, 1 - (c.embedding <=> ${vectorLiteral}::vector) AS score
        FROM chunks c JOIN sources s ON s.id = c.source_id
        WHERE c.corpus_id = ${corpusId}
          AND c.embedding IS NOT NULL
          AND c.generation = s.current_generation
          AND 1 - (c.embedding <=> ${vectorLiteral}::vector) >= ${config.minSimilarity}
        ORDER BY c.embedding <=> ${vectorLiteral}::vector
        LIMIT ${config.candidateDepth}
      `);
      return res.rows;
    });

…because below ~100k chunks an approximate index would only add noise to the recall measurements this cycle exists to take.

Run the vector stage under a timer — its milliseconds land on the receipt.

One SQL query does the entire candidate fetch:

select everything a result line will need — id, source, generation, content, anchor…

…plus the score: cosine similarity, computed as 1 minus the <=> cosine-distance operator.

Join each chunk to its source row — the reader predicate needs both sides…

…scope to the corpus being searched…

…only chunks that actually have an embedding…

…the reader predicate itself: the chunk’s generation must be the source’s CURRENT one. A half-swapped re-ingestion is structurally invisible.

…and the similarity floor: below RETRIEVAL_MIN_SIMILARITY, a nearby-but-meaningless chunk is not a candidate at all.

Nearest first — best-first order is exactly what the fusion desk’s rank math consumes.

Stop at candidateDepth (50 by default) — each librarian proposes at most that many.

End of the SQL.

Hand back the candidate rows…

…and close the timed stage.

📏
Exact scan, deferred index (R2)

There is no ANN index because the corpus is small enough not to need one — and adding one now would make every recall number partly measure the index’s approximation instead of the ranking math. The index is deferred until ~100k chunks, when it becomes worth paying for.

The fusion desk throws the scores away

Here is the problem the fusion desk faces: ts_rank_cd is unbounded and corpus-dependent; cosine similarity lives in [−1, 1]. The two scales are incomparable — adding them is adding meters to fahrenheit. So the default strategy, RRF, ignores the scores entirely and votes with rank positions — needing zero per-corpus calibration:

CODE — packages/retrieval/src/fusion.ts:62–82 

  const fusedScoreOf = (() => {
    if (knobs.fusion === "rrf") {
      // rank position is 1-based within each signal's best-first list;
      // absence contributes nothing (the standard RRF missing-list rule).
      const ftsRank = new Map(ftsCandidates.map((c, i) => [c.chunkId, i + 1]));
      const vectorRank = new Map(vectorCandidates.map((c, i) => [c.chunkId, i + 1]));
      return (id: string) => {
        let score = 0;
        const fr = ftsRank.get(id);
        if (fr !== undefined) score += 1 / (knobs.rrfK + fr);
        const vr = vectorRank.get(id);
        if (vr !== undefined) score += 1 / (knobs.rrfK + vr);
        return score;
      };
    }
    const ftsNorm = normalized(ftsCandidates);
    const vectorNorm = normalized(vectorCandidates);
    return (id: string) =>
      knobs.weightAlpha * (vectorNorm.get(id) ?? 0) +
      (1 - knobs.weightAlpha) * (ftsNorm.get(id) ?? 0);
  })();
PLAIN ENGLISH

Build the scoring function once, up front — which strategy you get is decided here, not per candidate.

The default branch: reciprocal-rank fusion.

The comment states rule one: ranks are 1-based within each signal’s best-first ballot…

…and rule two: a chunk missing from a ballot simply adds nothing. No penalty, no invented score — the standard RRF missing-list rule.

Map each chunk the literalist proposed to its rank: first place is 1, second is 2…

…same for the associative’s ballot. Notice what is NOT kept: the raw scores.

The scorer for one chunk id:

start at zero…

…look up its rank on the FTS ballot…

…if present, add 1/(k + rank) — first place with the default k=60 is worth 1/61.

…same lookup on the vector ballot…

…if present, add its 1/(k + rank) too. Appearing on BOTH ballots is how a chunk wins.

The fused score is just those two votes, summed.

End of the per-chunk scorer.

That closes the RRF branch.

The rival strategy (weighted sum): min-max normalize the FTS scores onto [0,1]…

…and the vector scores too. This normalization is precisely the calibration step RRF never needs.

Its scorer:

alpha times the normalized vector score…

…plus (1 − alpha) times the normalized FTS score — two tuning surfaces (alpha, normalization) where RRF has none.

Invoke immediately — fusedScoreOf is now a plain function from chunk id to fused score.

⚖️
Ties break lexicographically — on purpose

When two chunks fuse to the exact same score, the winner is decided by chunkId.localeCompare — never by Map iteration order. Determinism is a receipt property: the same query against the same corpus must produce the same recorded ranking, or the record is worthless as evidence.

The floor: teaching the associative to say “nothing”

🚨
A true story from this cycle

The first honest test asked the engine about something the corpus doesn’t contain — and the vector signal answered anyway. It always will: pure nearest-neighbor returns the nearest chunk, and “nearest” exists even for nonsense. RETRIEVAL_MIN_SIMILARITY was born mid-cycle to fix exactly this.

The knob ships with its reasoning attached — and with real numbers. The paraphrase from the opening hook scored cosine 0.316, comfortably above the floor; unrelated text sits near 0:

CODE — .env.example:80–86 

# Vector candidates below this cosine similarity are NOT candidates: pure
# nearest-neighbor always answers with SOMETHING, so honest empty results
# (US1) need a floor. Unrelated text sits near 0; related prose well above.
# 0.2 is the real-corpus-measured knee (TASK-10 eval report): high enough to
# still return honest-empty on unrelated text, low enough to keep a
# natural-question answer buried in a multi-topic chunk (cosine ~0.21).
RETRIEVAL_MIN_SIMILARITY=0.2
PLAIN ENGLISH

The rule: below this cosine similarity, a vector match is NOT a candidate…

…because pure nearest-neighbor ALWAYS answers with something — there is a “nearest” chunk even for a question about nothing in the corpus…

…so honest empty results need a floor. The scales help: unrelated text scores near 0, related prose well above it.

0.2 was not guessed — it is the measured knee from the real-corpus eval report…

…high enough that unrelated text still comes back honestly empty…

…low enough that a real answer buried in a multi-topic chunk (measured at cosine ~0.21) stays above the line.

The knob itself — env-first, like every retrieval knob (Principle VII).

Put the two signals’ exits side by side and “zero results” becomes a feature: the literalist returns nothing when the words don’t match (AND semantics), and the associative returns nothing when no meaning clears the floor. Both empties are honest — and both still get recorded.
