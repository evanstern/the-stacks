Answers with Receipts

Why this app almost never makes things up — and how "no evidence" beats a confident guess

The Rules of Evidence

AI chatbots are famous for confidently making things up — engineers call it hallucination. The Stacks almost never does, and not because its AI is smarter. The code around the AI is suspicious of it.

Module 3 stocked the warehouse of chunks. When a question arrives, the app runs it like a courtroom with strict rules of evidence: found passages are the exhibits, the LLM is an eloquent witness who may only testify about admitted exhibits, and a judge reviews every citation. Four independent exits all lead to the same honest verdict:

📂

No exhibits found

The search turns up nothing convincing. Case dismissed before the witness is even called — OpenAI never gets contacted, and never gets paid.

🙊

Testimony cites nothing

The witness answers but doesn't point at a single valid exhibit. The whole answer is thrown out.

🔧

Markers beyond repair

The [1]-style markers in the answer are too garbled to fix. Thrown out.

👻

Citing a phantom exhibit

A marker points at evidence that was never admitted into the room. Thrown out.

All four exits deliver the exact same reply: an honest "I couldn't find evidence for that in your library." Never a confident story invented on the stand. This whole pattern has a name — RAG — and this module shows you the real machinery behind it.

The Librarian's Shortlist

The Librarian deliberately over-fetches — asking Qdrant for roughly 10× more results than needed — then filters hard. Three ways a candidate gets bounced:

📉

Too weak

Its similarity score is under the threshold. Out.

👯

Duplicate

An identical passage was already admitted — deduped. Out.

👻

Ghost

It points at a chunk that's no longer part of the live collection. Out.

CODE — apps/api/app/retrieval_service.py 
        for hit in hits:
            candidate = retrieval_candidate_from_hit(self.db, hit, scope)
            if candidate is None:
                filtered_missing_chunk_count += 1
                continue
            if candidate.score < self.settings.retrieval_min_score:
                filtered_low_score_count += 1
                continue
            context_key = retrieval_candidate_identity_key(candidate)
            if context_key in seen_context_keys:
                deduplicated_count += 1
                continue
            seen_context_keys.add(context_key)
            candidates.append(candidate)
            if len(candidates) >= self.settings.retrieval_top_k:
                break
PLAIN ENGLISH

Walk through every result Qdrant sent back, one at a time...

Look up the actual passage this result points at.

If that passage no longer exists in the live library — it's a ghost...

...add one to the "missing" tally...

...and move on to the next result.

If the match score is under the confidence bar...

...tally it as "too weak"...

...and skip it too.

Build a fingerprint identifying this exact passage.

If we've already admitted an identical one...

...count it as a duplicate...

...and skip it.

Otherwise: remember the fingerprint...

...and admit the passage as an exhibit.

Once the shortlist holds the best 8...

...stop looking. Court is in session.

Notice every rejection is tallied — those counters get saved as an audit trail, so an empty search is never a mystery.

🎛️
Knobs, not surgery

The confidence bar (retrieval_min_score) and shortlist size (retrieval_top_k — top-k in the trade) are plain settings. Tuning how strict or generous retrieval is takes one changed value, not a code rewrite.

Notice how repetitive and specific it is. Real prompts read like instructions to a well-meaning but extremely literal-minded intern.

💡
A prompt isn't magic

It's a paragraph of English sitting in a code file. You can open chat_session_rag.py and edit your app's personality the way you'd edit an email — or tell your AI assistant, "make the system prompt stricter about X."

And how do the exhibits reach the witness? They're literally pasted under the question — this five-line function is the entire trick people call RAG:

CODE — apps/api/app/chat_session_rag.py 
def _user_prompt(question: str, contexts: Sequence[ContextChunk]) -> str:
    context_text = "\n\n".join(
        f"chunk_id={context.chunk_id}\n{context.content}" for context in contexts
    )
    return f"Question: {question}\n\nContext:\n{context_text}"
PLAIN ENGLISH

A tiny function: take the question and the admitted exhibits.

Stack the exhibits into one block of text, blank lines between them...

...each labeled with its chunk ID, so the witness can point at exhibits by name.

(Close up the stack.)

Final message: the question on top, the exhibits underneath. That's the whole context the model ever sees.
