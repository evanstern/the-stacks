When Things Break

Debugging is reading, not guessing — because well-built software writes down everything it does

The Flight Recorder

Here's the secret nobody tells beginners: the difference between a pro and a beginner isn't fewer bugs. It's that when something breaks, the pro's system already wrote down what happened.

Every airliner carries a flight data recorder — the "black box" that narrates every instrument, every second, into permanent storage. Investigators never guess; they pull the recorder and read. The Stacks is built the same way: it assumes things WILL break, so it keeps logs of everything, everywhere.

You've actually already met its three recorders — one per module, hiding in plain sight:

🎫

Job Event Logs

Module 3's assembly line stamped the ticket at every station. Those stamps are an append-only event log: parse started, chunking done, failed here, because of this.

🧾

Retrieval Run Receipts

Module 4's Librarian counted everything: every search writes an audit run with candidate scores and a tally of how many passages were rejected, and why.

🏗️

Version Lifecycle Events

Module 5's demolition log: every time an index version was built, activated, or torn down, a lifecycle event went on the record. No silent switches.

Engineers call this whole discipline observability. This module teaches you to be the investigator who reads the recorders.

Honest Status Codes

When something fails at the front door, the API 🚪 doesn't shrug. Every HTTP reply carries a three-digit status code, and three of them are your debugging vocabulary:

🔍

404 — "No such thing"

404 means the thing you asked for doesn't exist. Wrong ID, deleted session — not a crash, an absence.

🔌

503 — "My helper is down"

503 means a dependency I need is down — not my code's fault. Check Qdrant or OpenAI, not the app.

💥

500 — "My code broke"

500 means the server's own code fell over. This is the one that actually means "there's a bug in here."

Here's the aha: the API sorts failures by cause before picking a code. Each kind of failure is a differently-named exception, and only at the front door do they get translated into honest numbers:

CODE — apps/api/app/routes_sessions.py 
    try:
        return chat_session_service_dependency.answer_session_message_envelope(
            db,
            session_id,
            payload.content,
            chat_client=chat_client,
            graph_invoker=graph_invoker,
            retrieval_service=retrieval_service,
            settings=settings,
        )
    except SessionMessageNotFoundError as exc:
        raise HTTPException(
            status_code=status.HTTP_404_NOT_FOUND, detail="Session not found"
        ) from exc
    except (EmbeddingError, QdrantIndexError, RuntimeError) as exc:
        db.rollback()
        raise HTTPException(
            status_code=status.HTTP_503_SERVICE_UNAVAILABLE,
            detail=_chat_failure_detail(exc),
        ) from exc
PLAIN ENGLISH

Give it a shot: ask the chat service to answer the message…

…handing over everything it needs: the database connection, which session, what was typed…

…plus all the helpers — the chat brain, the graph runner, the Librarian, the settings.

If it works, that answer goes straight back to the user. Done.

But if the session simply doesn't exist — answer 404, "no such thing." Not a crash, an absence.

And if OpenAI or Qdrant let us down (note: the failures are named by cause)…

…first rollback: undo the half-finished database changes so nothing is left messy…

…then answer 503, "a helper I depend on is down — not my code's fault"…

…with a detail message that diagnoses which helper, not a generic shrug.

That's why error messages in this app read like diagnoses instead of apologies. The sorting happened before the number was chosen.

Self-Healing and Patience

Some failures aren't bugs at all — they're weather. When OpenAI 🤖 gets too many requests at once, it applies a rate limit and answers 429: "slow down." The right response is patience, programmed:

CODE — apps/api/app/embeddings.py (lightly abridged) 
for attempt in range(OPENAI_EMBEDDING_REQUEST_MAX_RETRIES + 1):
    response = httpx.post(
        "https://api.openai.com/v1/embeddings",
        headers={"Authorization": f"Bearer {api_key}"},
        json={"model": model, "input": batch, "dimensions": dimensions, "encoding_format": "float"},
        timeout=60,
    )
    try:
        response.raise_for_status()
        return response
    except httpx.HTTPStatusError as exc:
        if response.status_code != 429 or attempt == OPENAI_EMBEDDING_REQUEST_MAX_RETRIES:
            raise EmbeddingError(...) from exc
        time.sleep(_retry_delay_seconds(response.headers.get("retry-after"), attempt))
PLAIN ENGLISH

Allow ourselves the maximum number of attempts — first try plus a few retries.

Each attempt: ask OpenAI to turn this batch of text into meaning coordinates…

…proving who we are with our key…

…specifying the model, the text, and the format we want back…

…and refusing to wait longer than 60 seconds for a reply.

If OpenAI said "OK" — done, hand the answer back. The loop only continues on trouble.

On an error, check WHICH error: anything other than 429 — or 429 on the last attempt — give up and raise a real, named error. (That (...) is shortened for the page; the real file writes a full message. Retrying a genuine bug just hides it.)

But if it was 429: wait exactly as long as OpenAI asked — via its Retry-After header — then loop around and try again. That waiting-longer-between-tries pattern is called backoff.

Meanwhile, on the other side of the glass: how does the upload page show "live" progress? No magic. The page just politely re-asks — a pattern called polling:

CODE — apps/web/app/routes/upload.tsx 
  useEffect(() => {
    if (!job || terminalStatuses.has(job.status)) {
      return;
    }

    let cancelled = false;
    const refresh = async () => {
      const [nextJob, nextEvents] = await Promise.all([getIngestionJob(job.id), getJobEvents(job.id)]);
      if (!cancelled) {
        setJob(nextJob);
        setEvents(nextEvents);
      }
    };
    const interval = window.setInterval(() => {
      void refresh().catch((refreshError) => {
        if (!cancelled) {
          setError(refreshError instanceof Error ? refreshError.message : "Could not refresh job state.");
        }
      });
    }, 1500);

    return () => {
      cancelled = true;
      window.clearInterval(interval);
    };
  }, [job]);
PLAIN ENGLISH

Set up a little watcher for the current job.

If there's no job — or it already reached a terminal status like completed or failed — don't watch at all. No pointless asking.

Keep a sticky note ("cancelled?") so late-arriving replies can be told "never mind."

One refresh = ask the API for the job's latest status AND its event log, both at once.

If the page still cares by the time the answers land, repaint with the fresh info.

Now the heartbeat: run that refresh every 1,500 milliseconds — every 1.5 seconds…

…and if a refresh itself fails, show a readable error message instead of exploding.

That's the entire "live" progress bar. A timer and a question.

Finally, cleanup: when you leave the page, flip the sticky note and shut off the timer. No zombie timers haunting other pages.

⏱️
Transient failures get patience, not crashes

Notice the split: 429 gets retried with backoff, real bugs get raised immediately, and the progress page stops asking the moment the story ends. Each failure gets the response its cause deserves.

Truth Without Secrets

One last recorder rule: not every truth belongs on screen. A raw error can leak server file paths or a full traceback — a map of your server's insides, handed to whoever is looking.

So the frontend 🖥️ checks each failure message against a pattern of unsafe giveaways, and swaps anything suspicious for a calm "Check server logs for details":

unsafeDiagnosticPattern The detector: does this error text look like a leaked path or traceback?
safeBatchErrorCopy The replacement: a safe, human message when the raw one can't be shown — in apps/web/app/routes/upload.tsx
job metadata & events Where the FULL diagnosis still lives — operator-only, one click away in Records

Vague-on-purpose is a security feature, not laziness. The truth isn't hidden — it's filed, in a room only the operator can enter.

Aha: the debugging playbook

Debugging is reading, not guessing — and the whole skill is knowing which recorder to pull for which symptom. Here's the entire playbook for The Stacks:

📥
Stuck or failed upload

Pull the job's event log in Records — the stamps say exactly which station failed, and why.

🕳️
Empty or refused answers

Pull the retrieval run — the rejection tallies show whether nothing matched or everything got filtered out.

🔑
Login weirdness

Read the API's status codes — 401 vs 404 vs 500 each tell a completely different story.

🔌
Total outage

Run docker compose ps and check the healthchecks — find out which cast member is missing before blaming any code.

The Final Investigation

Badge on, investigator. Four real incidents — for each one, don't ask "what do I try?" Ask "which recorder do I pull?"

Incident 1

Chat suddenly returns: "Retrieval index is unavailable" — with a 503.

1. Using the status-code vocabulary: what KIND of problem is this, and which two things do you check first?

Incident 2

An upload shows failed, and the only message on screen is: "Check server logs for details."

2. Why won't the app just show the real error — and where does the real detail live?

Incident 3

A user swears the app "hallucinated" an answer last Tuesday. They're upset. You weren't there.

3. What evidence exists to settle it?

Incident 4 — the boss fight

Your AI assistant has been stuck for 30 minutes guessing why uploads fail: "try this… hmm, now try this…"

4. Based on this course, which ONE instruction do you give it?
