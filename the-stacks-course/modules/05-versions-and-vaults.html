Versions & Vaults

How the app rebuilds its entire library next to the live one — and why demolishing the gallery people are standing in is impossible

The Sealed Wing

Module 4 showed one question getting one answer, with receipts. Now the scary part: how do you replace the app's entire knowledge base while people are using it — and be certain you can't destroy the one that's live? This codebase's answer is its most professional habit.

Think of a museum that never renovates the open gallery. A new exhibition is built in a sealed wing with its own storage rooms and its own catalog, and each sealed wing is what the code calls a runtime version.

One World, Derived From One Name

How do two wings stay perfectly separate? Not by discipline — by arithmetic. When the app provisions a new version, every name in its world is calculated from the version's ID:

        database_name=database_name,

One token → its own database, its own collection, its own three folders. That calculated territory is a namespace, and because nobody invents names by hand, two versions can never collide by accident.

The Shipping Manifest

Filling the sealed wing isn't a casual move-in — it's a five-step ritual, and every arriving crate is checked against a manifest:

1
Preflight — rehearse on a stage set

All the safety guards run first against a throwaway in-memory database. If any guard misfires, you find out where mistakes are free.

2
Lock — write the manifest

Record every book's SHA-256 fingerprint plus the exact expected counts of chunks and rows. The wax seals are now on file.

3
Seed — move the crates in

Now ingest for real: every book runs down Module 3's assembly line, into this wing's private database and shelves.

4
Verify — recount everything

Compare what actually landed against the manifest: every fingerprint, every count. Off by anything? The wing stays sealed.

5
Activate — move the sign

Only now does the “Now Showing →” pointer swing to the new wing. Visitors mid-question never notice a thing.

Here's the wax-seal check itself — the moment a crate's actual bytes are held up against the manifest:

CODE — apps/api/app/corpus_manifest.py 
def validate_manifest_archives(manifest: CorpusManifest, archive_root: Path) -> None:
    for source in manifest.sources:
        archive_path = archive_root / source.filename
        if not archive_path.is_file():
            raise CorpusManifestError(f"Archive for source {source.source_id} is missing: {source.filename}")
        if source.sha256 is not None:
            actual_sha256 = hashlib.sha256(archive_path.read_bytes()).hexdigest()
            if actual_sha256 != source.sha256:
                raise CorpusManifestError(f"Archive hash mismatch for source {source.source_id}")
PLAIN ENGLISH

The inspection routine: hand it the manifest and the room where the crates sit.

Walk crate by crate through everything the manifest lists.

Find where this crate should be sitting.

Is the crate missing entirely?

Stop the whole operation — and say exactly which crate.

If the manifest recorded a wax-seal fingerprint for this crate...

...recompute the fingerprint from the file's actual bytes, right now.

Does it not match the seal on file?

Stop everything — this is not the crate that was sealed.

🔏
One byte = full stop

One changed byte in an archive means a completely different fingerprint, which means the whole operation halts before anything mutates. A corrupted or swapped book can't quietly poison the library — it can't even get through the door.

The Refusals

Guardrails aren't warnings — they're refusals, written into the version lifecycle code. First, the velvet rope at activation — three bouncer checks before the sign moves:

CODE — apps/api/app/version_lifecycle.py 
    version = db.get(RuntimeVersion, runtime_version_id)
    if version is None:
        raise ValueError("Runtime version does not exist")
    if version.status != VERSION_STATUS_READY:
        raise ValueError("Only ready runtime versions can be activated")
    if _is_teardown_locked(db, runtime_version_id):
        raise ValueError("Runtime version is teardown-locked and cannot be activated")
PLAIN ENGLISH

Look the wing up in the records.

Bouncer check #1: does this wing even exist?

No? Nobody moves any sign.

Check #2: did it pass final inspection — is it stamped ready?

Half-built wings don't open. Ever.

Check #3: is it already scheduled for demolition?

A condemned wing can't be the one you open to the public.

And the most important refusal in the codebase — four lines that make a whole class of catastrophe impossible. You cannot demolish the open gallery:

CODE — apps/api/app/version_lifecycle.py 
def _refuse_active_version(db: Session, runtime_version_id: str) -> None:
    pointer = db.get(ActiveVersionPointer, DEFAULT_ACTIVE_POINTER_NAME)
    if pointer is not None and pointer.runtime_version_id == runtime_version_id:
        raise ValueError("Active runtime version cannot be torn down")
PLAIN ENGLISH

The demolition crew's one unbreakable rule, as a tiny function.

Fetch the “Now Showing →” sign and see where it points.

Is the wing you want to demolish the one visitors are standing in right now?

Refuse. No override, no exceptions.

It goes further: wiping a corpus from the CLI requires a flag spelling out --confirm-version <name> — you must literally type the name of the thing you're destroying. And mid-operation it re-checks the vault copy's fingerprint, aborting if the immutable archive changed.
