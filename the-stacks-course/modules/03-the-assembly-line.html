The Assembly Line

How an uploaded rulebook becomes searchable — one factory station at a time

The Receiving Dock

Module 2 ended with a promise: follow the Worker 🛠️ into the back room. Here we go — starting at the loading dock.

You drop a 300-page rulebook on the Upload page, and the response comes back in under a second. The book is not processed yet — you've been handed a claim ticket, and the real work happens behind the wall as a background job.

Before accepting any crate, the API 🚪 plays receiving-dock inspector. Three things happen in that first second:

1
Inspect the crate

Is this a supported file type? The dock accepts Markdown (.md), plain text (.txt), HTML, EPUB, and ZIP archives. Nothing else gets in.

2
Fingerprint the bytes

The API computes a SHA-256 fingerprint of the file — so the exact same book uploaded twice can be recognized instantly.

3
Write the job ticket

A ticket saying "this file needs processing, status: queued" gets dropped on the spike, and you get back a claim-ticket number. Total elapsed time: well under a second.

🚫
The dock says no — instantly

Try uploading a .docx and the crate is rejected on the spot with HTTP 415. A fast rejection at the door beats a mysterious failure three stations down the line.

The Job Ticket and the Patient Worker

The job ticket is just a row in a database table, and the Worker is the most patient employee ever hired. Its entire life is one polite infinite loop:

CODE — apps/worker/worker.py 
while running:
    with SessionLocal() as db:
        job = process_next_job(db)
    if job is not None:
        print(f"Processed ingestion job {job.id}; status={job.status}", flush=True)
    if run_once:
        break
    time.sleep(poll_seconds)
PLAIN ENGLISH

Keep walking the floor forever (until someone tells the Worker to clock out)...

Open a fresh connection to the filing cabinet...

...and try to grab and process exactly one job ticket.

If there actually was a job waiting...

...jot a note in the log: which ticket got finished, and how it ended.

(A test-mode switch: do one lap of the floor...

...then stop — handy for testing.)

Nap for five seconds, then check the pile again.

That check-nap-check rhythm is called polling. And grabbing a ticket sounds trivial — until you hire a second Worker. This SQL query makes fights over the same ticket impossible:

CODE — apps/api/app/ingestion.py 
statement = (
    select(IngestionJob)
    .where(IngestionJob.status == "queued")
    .order_by(IngestionJob.created_at, IngestionJob.id)
    .with_for_update(skip_locked=True)
    .limit(1)
)
job = db.scalars(statement).first()
PLAIN ENGLISH

Build a question for the filing cabinet:

look through the job tickets...

...only the ones still stamped "queued"...

...oldest ticket first — first come, first served...

...put your hand on it so nobody else can take it, and skip any ticket that already has a hand on it...

...and take just one.

(That's the whole question.)

Ask the database, keep the first ticket it hands back.

💡
Aha: SKIP LOCKED = no fights, by construction

"Hand on the ticket" is a database lock, and skip_locked means "if someone's hand is already on one, don't wait — grab the next." Two Workers can never process the same ticket, not because they're careful, but because the rules of the game make it impossible.

📦
Boring is a feature

There's no fancy queue system here — the "queue" is literally a database table with a status column. One less machine to install, break, and debug.

The Cutting Station

After the Worker parses the crate into plain text, it cuts the book into chunks. Why cut at all? The AI can only read a few pages at a time, and search works far better on small passages than whole books.

But you don't cut fabric mid-pattern — you cut along the seams:

CODE — apps/api/app/ingestion.py 
hard_end = min(start + MAX_CHUNK_CHARS, len(text))
end = hard_end
if hard_end < len(text):
    paragraph_break = text.rfind("\n\n", start, hard_end)
    sentence_break = text.rfind(". ", start, hard_end)
    whitespace_break = text.rfind(" ", start, hard_end)
    end = max(paragraph_break, sentence_break + 1 if sentence_break != -1 else -1, whitespace_break)
    if end <= start:
        end = hard_end
PLAIN ENGLISH

The farthest this cut may reach: 1,200 characters — or the end of the book, whichever comes first.

Assume, for now, we cut at that hard limit.

But if there's more text beyond the limit...

...look backwards for the last paragraph break (a blank line)...

...and the last sentence ending (period + space)...

...and the last plain old space.

Cut at the best seam found: paragraph beats sentence beats space.

If no seam exists anywhere in range...

...fall back to the hard cut. An ugly cut beats no cut.

The cutting order, as a house rule:

1
Prefer a paragraph break

A blank line is the cleanest seam — whole thoughts stay together.

2
Else, a sentence end

A period followed by a space — at least no sentence gets severed.

3
Else, any space

Last resort before cutting mid-word.

4
And always: a 160-character overlap

The next chunk starts 160 characters before this one ended — so a thought straddling the cut line appears whole in at least one chunk.

Measuring & Warehousing

Now each chunk gets measured. An embedding is a list of 1,536 numbers that works like GPS coordinates for meaning — a chunk about "fire damage" lands near a chunk about "burning", even though they share no words.

The Worker sends chunks to OpenAI in batches, as a plain HTTP request:

CODE — apps/api/app/embeddings.py 
response = httpx.post(
    "https://api.openai.com/v1/embeddings",
    headers={"Authorization": f"Bearer {api_key}"},
    json={"model": model, "input": batch, "dimensions": dimensions, "encoding_format": "float"},
    timeout=60,
)
PLAIN ENGLISH

Send a package of data...

...to OpenAI's embeddings counter.

Flash the membership card — the secret API key — so OpenAI knows whose account to bill.

The package: which measuring model to use, the batch of passages, and how many coordinates (1,536) to send back per passage.

If no answer within 60 seconds, hang up instead of waiting forever.

(That's the whole request.)

Back come the coordinates — vectors — and the Worker shelves them in the Qdrant warehouse, 25 at a time, so they're indexed for instant lookup:

CODE — apps/api/app/qdrant_index.py 
response = httpx.put(
    f"{self.url}/collections/{self.collection}/points?wait=true",
    json={"points": [
        {"id": point.id, "vector": point.vector, "payload": point.payload}
        for point in batch
    ]},
    timeout=self._UPSERT_TIMEOUT,
)
PLAIN ENGLISH

Send a shipment to the warehouse (Qdrant)...

...to this collection's shelves — and wait=true means "don't hang up until it's actually shelved."

The shipment is a list of points; each point is...

...a slot number (id), the coordinates (vector), and a luggage tag (payload) saying which file and section it came from.

...one point per chunk in this batch of 25.

(Close up the list.)

Give up politely if the warehouse takes too long.

(That's the whole shipment.)

One last detail — where does that slot number come from? It's calculated, never random:

CODE — apps/api/app/ingestion.py 
def deterministic_point_id(chunk: DocumentChunk) -> str:
    return str(uuid5(NAMESPACE_URL, f"thestacks:{chunk.upload_id}:{chunk.ingestion_job_id}:{chunk.id}"))
PLAIN ENGLISH

A tiny recipe for a warehouse slot number (a UUID)...

...computed from the chunk's identity — which upload, which job, which chunk. Same chunk in, same slot number out. Every single time.

💡
Aha: same slot every time = safe retries

Because the slot number is calculated from the chunk's identity, re-running the same job overwrites the same warehouse slots instead of creating duplicates. Engineers call this idempotent: retrying costs nothing, so crashes are annoying instead of catastrophic.

Walk the Factory Floor

You've seen the whole line: dock → ticket → cut → measure → shelve. Remember — "make it a background job" is one of the highest-leverage sentences you can say to an AI assistant. Now put on the floor-manager hat:
