01

The Question Journey

What actually happens in the two seconds between hitting Enter and getting an answer with receipts

Two seconds, with receipts

You type “Can a wizard wear armor?” and hit Enter. Two seconds later: an answer, with little [1] [2] footnotes that open the exact paragraph of the rulebook you uploaded.

This course is about everything that happens inside those two seconds — and inside the machine you built. Welcome to Module 1 of 6 of Inside The Stacks.

The Stacks is a self-hosted web app for tabletop RPG game masters: you upload your own rulebooks and campaign notes (Markdown, text, HTML, EPUB, or saved-webpage ZIPs), then ask questions in a chat. Three promises define it:

📎

Ask with receipts

Every factual sentence carries a numbered citation like [1] that opens the exact source passage.

📚

Bring your own books

It answers ONLY from the material you uploaded — your rules, your notes, your table.

🙅

Refuses to bluff

No evidence in your books? It says so, plainly, instead of making something up.

A closed-stacks library, digitized

In an ordinary library you wander the shelves yourself. In a closed-stacks library — which is literally what “the stacks” means — you hand a librarian a question slip at the desk, a runner disappears into the shelves, and your material comes back with call slips pointing at exact pages.

The Stacks is that library, digitized. Six characters staff it — here’s a teaser of the cast (full introductions in Module 2):

🖥️

Web app

The reading room — the page you see and type into. Engineers call the visible part the frontend.

🚪

The librarian at the front desk. An API is the backend program every request goes through.

🛠️

The intake clerk in the back room — a worker processes your uploaded books off-stage.

🗃️

Honesty is cheaper

Perceived vs. actual progress

Why does naming the hops matter? Because now you can point an AI assistant at the right one — “that’s a backend retrieval problem, not a UI problem” — and know where to look when something breaks. Prove it to yourself:

2. The chat replies “no evidence found” to every single question — even easy ones. Which hop of the journey would you suspect FIRST?

3. A friend wants the app to ALSO answer from general knowledge when the books don’t cover something. Which step of the journey would have to change?

The Call Sheet

In Module 1 you watched the relay race. Time for proper introductions — because everything you saw was performed by five separate programs, each sealed inside its own container.

The file docker-compose.yml is the call sheet: it lists who's on the crew, when each person shows up, how you check they're ready ("ready on set!"), and which rooms they work in. Docker is the studio that runs the whole operation, and Docker Compose is the stage manager reading from it.

The Cast, In Order of Appearance

🛠️

🗃️

🧭

💡
Why should you care who's who?

Knowing the cast lets you steer AI precisely ("put that logic in the Worker, not the API") and localize failures (uploads stall → suspect the Worker; login fails → suspect the API). Engineers call this one-job-per-specialist design separation of concerns.

03

The Assembly Line

How an uploaded rulebook becomes searchable — one factory station at a time

The Receiving Dock

Module 2 ended with a promise: follow the Worker 🛠️ into the back room. Here we go — starting at the loading dock.

You drop a 300-page rulebook on the Upload page, and the response comes back in under a second. The book is not processed yet — you've been handed a claim ticket, and the real work happens behind the wall as a background job.

Before accepting any crate, the API 🚪 plays receiving-dock inspector. Three things happen in that first second:

1
Inspect the crate

Is this a supported file type? The dock accepts Markdown (.md), plain text (.txt), HTML, EPUB, and ZIP archives. Nothing else gets in.

2
Fingerprint the bytes

The API computes a SHA-256 fingerprint of the file — so the exact same book uploaded twice can be recognized instantly.

3
Write the job ticket

A ticket saying "this file needs processing, status: queued" gets dropped on the spike, and you get back a claim-ticket number. Total elapsed time: well under a second.

🚫
The dock says no — instantly

Try uploading a .docx and the crate is rejected on the spot with HTTP 415. A fast rejection at the door beats a mysterious failure three stations down the line.

The Cutting Station

After the Worker parses the crate into plain text, it cuts the book into chunks. Why cut at all? The AI can only read a few pages at a time, and search works far better on small passages than whole books.

But you don't cut fabric mid-pattern — you cut along the seams:

CODE — apps/api/app/ingestion.py 
hard_end = min(start + MAX_CHUNK_CHARS, len(text))
end = hard_end
if hard_end < len(text):
    paragraph_break = text.rfind("\n\n", start, hard_end)
    sentence_break = text.rfind(". ", start, hard_end)
    whitespace_break = text.rfind(" ", start, hard_end)
    end = max(paragraph_break, sentence_break + 1 if sentence_break != -1 else -1, whitespace_break)
    if end <= start:
        end = hard_end
PLAIN ENGLISH

The farthest this cut may reach: 1,200 characters — or the end of the book, whichever comes first.

Assume, for now, we cut at that hard limit.

But if there's more text beyond the limit...

...look backwards for the last paragraph break (a blank line)...

...and the last sentence ending (period + space)...

...and the last plain old space.

Cut at the best seam found: paragraph beats sentence beats space.

If no seam exists anywhere in range...

...fall back to the hard cut. An ugly cut beats no cut.

The cutting order, as a house rule:

1
Prefer a paragraph break

A blank line is the cleanest seam — whole thoughts stay together.

2
Else, a sentence end

A period followed by a space — at least no sentence gets severed.

3
Else, any space

Last resort before cutting mid-word.

4
And always: a 160-character overlap

The next chunk starts 160 characters before this one ended — so a thought straddling the cut line appears whole in at least one chunk.

Because the slot number is calculated from the chunk's identity, re-running the same job overwrites the same warehouse slots instead of creating duplicates. Engineers call this idempotent: retrying costs nothing, so crashes are annoying instead of catastrophic.

04

Answers with Receipts

Why this app almost never makes things up — and how "no evidence" beats a confident guess

The Rules of Evidence

AI chatbots are famous for confidently making things up — engineers call it hallucination. The Stacks almost never does, and not because its AI is smarter. The code around the AI is suspicious of it.

Module 3 stocked the warehouse of chunks. When a question arrives, the app runs it like a courtroom with strict rules of evidence: found passages are the exhibits, the LLM is an eloquent witness who may only testify about admitted exhibits, and a judge reviews every citation. Four independent exits all lead to the same honest verdict:

📂

No exhibits found

The search turns up nothing convincing. Case dismissed before the witness is even called — OpenAI never gets contacted, and never gets paid.

🙊

Testimony cites nothing

The witness answers but doesn't point at a single valid exhibit. The whole answer is thrown out.

🔧

Markers beyond repair

The [1]-style markers in the answer are too garbled to fix. Thrown out.

👻

Citing a phantom exhibit

A marker points at evidence that was never admitted into the room. Thrown out.

All four exits deliver the exact same reply: an honest "I couldn't find evidence for that in your library." Never a confident story invented on the stand. This whole pattern has a name — RAG — and this module shows you the real machinery behind it.

The Librarian's Shortlist

The Librarian deliberately over-fetches — asking Qdrant for roughly 10× more results than needed — then filters hard. Three ways a candidate gets bounced:

📉

Too weak

Its similarity score is under the threshold. Out.

👯

Duplicate

An identical passage was already admitted — deduped. Out.

👻

Ghost

It points at a chunk that's no longer part of the live collection. Out.

CODE — apps/api/app/retrieval_service.py

The Shipping Manifest

Filling the sealed wing isn't a casual move-in — it's a five-step ritual, and every arriving crate is checked against a manifest:

1
Preflight — rehearse on a stage set

All the safety guards run first against a throwaway in-memory database. If any guard misfires, you find out where mistakes are free.

2
Lock — write the manifest

Record every book's SHA-256 fingerprint plus the exact expected counts of chunks and rows. The wax seals are now on file.

3
Seed — move the crates in

Now ingest for real: every book runs down Module 3's assembly line, into this wing's private database and shelves.

4
Verify — recount everything

Compare what actually landed against the manifest: every fingerprint, every count. Off by anything? The wing stays sealed.

5
Activate — move the sign

Only now does the “Now Showing →” pointer swing to the new wing. Visitors mid-question never notice a thing.

Here's the wax-seal check itself — the moment a crate's actual bytes are held up against the manifest:

CODE — apps/api/app/corpus_manifest.py 
def validate_manifest_archives(manifest: CorpusManifest, archive_root: Path) -> None:
    for source in manifest.sources:
        archive_path = archive_root / source.filename
        if not archive_path.is_file():
            raise CorpusManifestError(f"Archive for source {source.source_id} is missing: {source.filename}")
        if source.sha256 is not None:
            actual_sha256 = hashlib.sha256(archive_path.read_bytes()).hexdigest()
            if actual_sha256 != source.sha256:
                raise CorpusManifestError(f"Archive hash mismatch for source {source.source_id}")
PLAIN ENGLISH

The inspection routine: hand it the manifest and the room where the crates sit.

Walk crate by crate through everything the manifest lists.

Find where this crate should be sitting.

Is the crate missing entirely?

Stop the whole operation — and say exactly which crate.

If the manifest recorded a wax-seal fingerprint for this crate...

...recompute the fingerprint from the file's actual bytes, right now.

Does it not match the seal on file?

Stop everything — this is not the crate that was sealed.

One byte = full stop

One changed byte in an archive means a completely different fingerprint, which means the whole operation halts before anything mutates. A corrupted or swapped book can't quietly poison the library — it can't even get through the door.

Honest Status Codes

When something fails at the front door, the API 🚪 doesn't shrug. Every HTTP reply carries a three-digit status code, and three of them are your debugging vocabulary:

🔍

404 — "No such thing"

404 means the thing you asked for doesn't exist. Wrong ID, deleted session — not a crash, an absence.

🔌

503 — "My helper is down"

503 means a dependency I need is down — not my code's fault. Check Qdrant or OpenAI, not the app.

💥

500 — "My code broke"

500 means the server's own code fell over. This is the one that actually means "there's a bug in here."

Here's the aha: the API sorts failures by cause before picking a code. Each kind of failure is a differently-named exception, and only at the front door do they get translated into honest numbers:

Self-Healing and Patience

Some failures aren't bugs at all — they're weather. When OpenAI 🤖 gets too many requests at once, it applies a rate limit and answers 429: "slow down." The right response is patience, programmed:

CODE — apps/api/app/embeddings.py (lightly abridged) 
for attempt in range(OPENAI_EMBEDDING_REQUEST_MAX_RETRIES + 1):
    response = httpx.post(
        "https://api.openai.com/v1/embeddings",
        headers={"Authorization": f"Bearer {api_key}"},
        json={"model": model, "input": batch, "dimensions": dimensions, "encoding_format": "float"},
        timeout=60,
    )
    try:
        response.raise_for_status()
        return response
    except httpx.HTTPStatusError as exc:
        if response.status_code != 429 or attempt == OPENAI_EMBEDDING_REQUEST_MAX_RETRIES:
            raise EmbeddingError(...) from exc
        time.sleep(_retry_delay_seconds(response.headers.get("retry-after"), attempt))
PLAIN ENGLISH

Allow ourselves the maximum number of attempts — first try plus a few retries.

Each attempt: ask OpenAI to turn this batch of text into meaning coordinates…

…proving who we are with our key…

…specifying the model, the text, and the format we want back…

…and refusing to wait longer than 60 seconds for a reply.

If OpenAI said "OK" — done, hand the answer back. The loop only continues on trouble.

On an error, check WHICH error: anything other than 429 — or 429 on the last attempt — give up and raise a real, named error. (That (...) is shortened for the page; the real file writes a full message. Retrying a genuine bug just hides it.)

But if it was 429: wait exactly as long as OpenAI asked — via its Retry-After header — then loop around and try again. That waiting-longer-between-tries pattern is called backoff.

Meanwhile, on the other side of the glass: how does the upload page show "live" progress? No magic. The page just politely re-asks — a pattern called polling:

CODE — apps/web/app/routes/upload.tsx 
  useEffect(() => {
    if (!job || terminalStatuses.has(job.status)) {
      return;
    }

    let cancelled = false;
    const refresh = async () => {
      const [nextJob, nextEvents] = await Promise.all([getIngestionJob(job.id), getJobEvents(job.id)]);
      if (!cancelled) {
        setJob(nextJob);
        setEvents(nextEvents);
      }
    };
    const interval = window.setInterval(() => {
      void refresh().catch((refreshError) => {
        if (!cancelled) {
          setError(refreshError instanceof Error ? refreshError.message : "Could not refresh job state.");
        }
      });
    }, 1500);

    return () => {
      cancelled = true;
      window.clearInterval(interval);
    };
  }, [job]);
PLAIN ENGLISH

Set up a little watcher for the current job.

If there's no job — or it already reached a terminal status like completed or failed — don't watch at all. No pointless asking.

Keep a sticky note ("cancelled?") so late-arriving replies can be told "never mind."

One refresh = ask the API for the job's latest status AND its event log, both at once.

If the page still cares by the time the answers land, repaint with the fresh info.

Now the heartbeat: run that refresh every 1,500 milliseconds — every 1.5 seconds…

…and if a refresh itself fails, show a readable error message instead of exploding.

That's the entire "live" progress bar. A timer and a question.

Finally, cleanup: when you leave the page, flip the sticky note and shut off the timer. No zombie timers haunting other pages.

Transient failures get patience, not crashes

Notice the split: 429 gets retried with backoff, real bugs get raised immediately, and the progress page stops asking the moment the story ends. Each failure gets the response its cause deserves.

One last recorder rule: not every truth belongs on screen. A raw error can leak server file paths or a full traceback — a map of your server's insides, handed to whoever is looking.

job metadata & events Where the FULL diagnosis still lives — operator-only, one click away in Records

Vague-on-purpose is a security feature, not laziness. The truth isn't hidden — it's filed, in a room only the operator can enter.
